Why Agent Systems Are Harder Than They Look

March 16, 2026

Every week there's a new "agent" product. This one uses GPT-4o, that one uses Claude 3.5, the other one has "autonomous planning." They're all going to revolutionize work. Just sign up and watch them do your job.

Here's what nobody tells you: the hard part isn't the LLM. It's everything around it.

The Planning Problem

LLMs can generate plans. They can break down "write me a report" into "search for data, analyze trends, write draft, format output." But they can't reliably execute long-running plans. Somewhere between step 3 and step 7, they hallucinate a step, forget a constraint, or make an assumption that was wrong three steps back.

Production agents need guardrails. But guardrails kill autonomy — the thing agents are supposed to provide. The sweet spot between "too rigid to be useful" and "too loose to be safe" doesn't exist in the general case.

The Context Window Trap

You'd think longer context windows solve everything. They don't. There's a fundamental problem: as the context grows, the model's ability to find the signal in the noise degrades. You can dump an entire codebase into Claude and ask "where's the bug?" and it'll miss it. Not because it can't read code, but because the relevant signal gets buried.

RAG helps, but now you've added another system that can fail. The embedding model might miss semantic similarity. The chunking strategy might split critical context. The retrieval might surface the wrong document. Each layer adds failure modes.

The Tool Use Tax

Using tools sounds simple: model decides to call a function, gets results, incorporates them. But there's a taxonomy of failures:

  • Tool selection: Uses the wrong tool for the job
  • Parameter generation: Generates valid JSON but wrong parameters
  • Result parsing: Can't extract what it needs from the tool output
  • Looping: Calls the same tool with slightly different parameters hoping for a different result

Each of these needs separate handling. And handling them reduces the "autonomous" part of autonomous agent.

What Actually Works

The agents that work in production share a common pattern: they're narrow. Not "general AI assistant" narrow — "do exactly this one thing" narrow. A coding agent that only writes tests. A research agent that only summarizes papers. An email agent that only drafts responses to specific query types.

The moment you want it to be flexible, you add complexity. The moment you add complexity, you add failure modes. The moment you add failure modes, you need humans in the loop. At which point, is it really an agent?

The Real Innovation

The interesting work isn't making models more capable. It's in the infrastructure around them: observability, error recovery, human-in-the-loop systems, trust calibration. The boring stuff.

That's where the moats are. Not in the prompt engineering, not in the model choice, but in the system that reliably gets work done while letting humans intervene when things go wrong.

The buzzword will fade. The infrastructure will remain.
