How JWT Works

March 21, 2026

JSON Web Tokens are everywhere. They're the default for "stateless" authentication, the backbone of countless APIs, and the source of countless security vulnerabilities. Understanding JWTs isn't optional — it's mandatory if you build anything that touches the web.

The Structure

A JWT is three Base64-encoded strings joined by dots:

header.payload.signature

Header contains the algorithm used for signing (HS256, RS256, ES256) and the token type.

Payload contains the claims — the actual data. Standard claims include iss (issuer), exp (expiration), sub (subject), and iat (issued at).

Signature is the cryptographic proof that the payload hasn't been tampered with.

Symmetric vs Asymmetric

HS256 (HMAC) uses a shared secret. Both server and client know the key. Fast, simple, but problematic — anyone who can verify the token can also create one.

RS256 (RSA) uses a private key to sign and public key to verify. The server keeps the private key; clients only need the public key. This is the right choice for most scenarios.

The Dangerous Parts

  • Algorithm confusion ("none" attack) — change the header to {"alg": "none"} and some implementations accept it
  • Weak secrets — short HMAC secrets get cracked instantly
  • No expiration by default — always set exp
  • Information leakage — payload is just Base64, anyone can read it. Don't put secrets there.

The "Stateless" Myth

JWTs are often sold as "stateless" — no database lookup needed. This is true but misleading. You still need:

  • Key rotation infrastructure
  • Revocation handling (what if a user logs out?)
  • Refresh token flows

The trade-off is real: you save a database round-trip but add complexity elsewhere.

The Verdict

JWTs are a tool, not a solution. Use RS256, set expiration, never put secrets in the payload, and understand that "stateless" means "stateless about user state" — not "stateless about infrastructure."
