The Case for Plain Text

April 17, 2026

Every few years, there's a new format that's supposed to replace plain text. JSON was supposed to replace XML. YAML was supposed to replace JSON. TOML was supposed to replace YAML. We're now on TOML and already eyeing something else. And through it all, plain text just... sits there. Works. Everywhere.

Here's the thing: plain text has no dependencies. You can open a .txt file in 2040 on a device that doesn't exist yet. You cannot say that about a Word document, an iWork file, or a proprietary JSON schema that requires a specific library to parse. Plain text is format-agnostic in the best way — it doesn't care what you use, it just works.

The "structured" formats sound better in theory. Schema validation! Type checking! Interop! But they also mean lock-in, version conflicts, dependency hell, and eventually, migration pain. Remember when everyone was excited about XML? Remember when JSON was the revolution? The pendulum swings, plain text stays.

Yes, plain text lacks semantics. That's the point. Semantics are coupling. The moment you add structure, you create a contract that must be maintained. Plain text is loosely coupled — it's a blank canvas that adapts to whatever tool you throw at it. A grep command from 1974 still works on modern text files. That's not a feature, that's a design philosophy.

I'm not saying never use structured formats. Databases need structure. APIs need contract. But for human knowledge, for documentation, for notes, for config that outlives the project — plain text wins. The simplest format that could possibly work beats the perfect format that's already obsolete.

We keep overengineering our tools because we mistake complexity for sophistication. Plain text is sophisticated in its simplicity. It doesn't need to impress you. It just needs to work, forever, on anything.
