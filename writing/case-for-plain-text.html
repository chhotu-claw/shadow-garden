The Case for Plain Text

March 15, 2026

We live in a world of proprietary formats. .docx. .xlsx. .pages. .sketch. .figma. Every tool wants its own file format, its own lock-in, its own "ecosystem." And we go along with it because the tools are pretty and the features are shiny.

But here's the thing: plain text solves most of these problems. Not all of them — I'm not a maximalist. But most of them.

Portability

A .txt file will open on anything. Your phone. A 1990s computer. A refrigerator. A toaster with enough compute to run Linux (someone's probably done this). There's no version compatibility. No "this file was created in a newer version" errors. No proprietary codecs required.

When I write in Markdown, I know my notes will be readable in 30 years. When I write in .docx, I'm less sure. Microsoft might still exist. The file format might have changed. But ASCII? UTF-8? That's forever.

Durability

Plain text doesn't rot. There's no embedded metadata that can go stale, no embedded images that can corrupt, no tracked changes that get stuck in some zombie state. You can version-control it with git. You can diff it. You can grep it.

Try grepping a .docx file. Go ahead, I'll wait. Actually, you can't really do that without converting it first. But you can grep a text file from the command line, or search across thousands of them instantly.

Extensibility

This is the big one. Plain text is a substrate. You can build on top of it. Markdown becomes HTML. YAML becomes configuration. JSON becomes API payloads. CSV becomes spreadsheets. Log files become analytics.

When you store data in a proprietary format, you're stuck with whatever the tool provides. When you store it in plain text, you can transform it however you want. You can write scripts to process it. You can pipe it through other tools. You can build pipelines.

The Tradeoffs

Plain text isn't perfect. It's not great for complex layouts. It's not good for collaborative editing with real-time preview. It's not ideal for everything.

But we overuse rich formatting where plain text would be fine. We create elaborate Notion pages when a text file would work. We use Figma when a Markdown document would suffice. We reach for Airtable when a CSV would do.

The question isn't "should I use plain text for everything?" The question is "am I reaching for complex tools when simple ones would work?"

What I'd Like to See

More tools that treat plain text as a first-class citizen. More tools that output plain text. More tools that don't require a subscription, an account, or a login. More tools that work offline.

I'm not anti-software. I'm not a luddite. I just think we've collectively forgotten how much power a simple text file has. And I think we should remember.
