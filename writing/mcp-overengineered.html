MCP Is Overengineered

April 3rd, 2026

Model Context Protocol (MCP) is the new shiny in AI tooling. Every startup announces MCP support. Every developer conference has a talk about it. And I'm sitting here thinking: this is solving a problem most people don't have.

What MCP actually does

MCP standardizes how AI models talk to tools. Instead of each tool having its own API, MCP provides a unified interface. You declare your tools' schemas upfront, the model sees them all, and everyone lives happily ever after.

On paper, that makes sense. In practice, it means you're dumping your entire schema at the model's feet every single request. That's the opposite of what works.

The upfront problem

LLMs have context limits. Not huge ones, not yet. When you're loading 50 tools with full JSON schemas on every prompt, you're burning context on metadata instead of the actual task.

Lazy-loading works better. Only load the tools you need when you need them. It's how your operating system works. It's how your brain works. You don't inventory every tool in your garage before fixing a leaky faucet — you grab the wrench.

The schema dump problem

Here's what happens in practice: developers spend hours crafting perfect JSON schemas, documenting every parameter, every type, every edge case. Then the model ignores half of it and hallucinates the rest.

The model doesn't need your schema to know what a "search" function does. It needs a simple description and maybe one example. The rest is overhead.

What works better

Tool discovery on-demand. Instead of declaring everything upfront:

  • Describe tools minimally — name + one-line purpose
  • Only load the schema when the model decides to use the tool
  • Let the model figure out parameters from context, not from documentation

This is how function calling actually works in production. The model reasons about what it needs, asks for the relevant tool, gets the schema, then calls it. Not the other way around.

The take

MCP solves a legitimate problem — tool fragmentation. But it solves it the wrong way, by making everything upfront and explicit when most of that complexity is unnecessary.

The real tooling innovation isn't in the protocol du jour. It's in the boring infrastructure: caching, memory, state management, error handling. That's where the differentiation happens. Not in the schema format.
