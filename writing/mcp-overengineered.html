MCP Is Overengineered

April 3rd, 2026

Every AI tool now brags about "MCP support." Model Context Protocol, Anthropic's attempt to standardize how models talk to tools. Everyone's excited. I'm not convinced.

What's the actual problem?

Before MCP, you had prompt engineering. You told the model what tools exist, gave it examples, and it figured it out. It worked fine for 99% of use cases. Now we have a protocol that dumps entire JSON schemas into the prompt context, hoping the model parses it correctly.

That's not simplification. That's overhead.

Lazy-loading beats upfront schemas

Here's what actually works: tell the model about tools when it needs them. Not before. The " Describe your tools in the system prompt" approach was clunky but it had the right idea — context is for what's relevant, not everything you've ever built.

MCP does the opposite. It loads everything upfront. Your prompt is now bloated with tool definitions you might never use. The model has to wade through schema after schema to find what matters.

The real issue

What people actually need isn't a protocol — they need better tool descriptions. Clear names, focused purposes, good examples. That's a writing problem, not a protocol problem.

The tools that work best are the ones that are simple enough to describe in a sentence. Complex schemas are a sign the tool is too complicated, not that the protocol needs upgrading.

The take

MCP is solving a problem for tool builders, not tool users. It's a developer ergonomics play dressed up as an AI breakthrough. The actual value — making models use tools reliably — was achievable before, and it's still achievable without the protocol overhead.

Call me when it does something the prompt couldn't do faster.
