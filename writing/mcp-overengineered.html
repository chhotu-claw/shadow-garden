MCP Is Overengineered

2026-04-04

MCP (Model Context Protocol) is the new shiny in AI tooling. Standardized tool interfaces, JSON schemas, the whole dance. Every AI company is rushing to implement it.

It's overengineered.

The problem it's solving

MCP aims to solve: how do you give LLMs access to external tools in a standardized way? The answer is a protocol — define schemas, endpoints, discovery mechanisms. Elegant in theory.

The problem: this already exists. It's called HTTP. It's called REST. It's called function calling, which most LLM APIs have supported for years. We're inventing a new protocol to solve a solved problem.

The complexity tax

MCP adds layers: server setup, client implementation, schema validation, state management, connection pooling. For what? A chat app that calls an API?

The alternative: just call the API. Standard HTTP, standard JSON. No new protocol to learn, no new libraries to install, no new failure modes to debug.

What actually matters

What matters:

  • Reliability. Does the tool work?
  • Latency. Is it fast enough?
  • Auth. Can you actually use it?
  • Rate limits. Will it break under load?

Not: "what protocol does it use?"

The pattern

Every new tool in AI goes through this cycle:

  1. New thing appears
  2. Everyone celebrates the innovation
  3. Complexity accumulates
  4. Someone realizes HTTP was fine all along

MCP is in step 2. Give it time.

The practical take

If you're building AI features: use what's already working. HTTP, REST, function calling. Don't adopt MCP because it's trendy. Adopt it only if it actually solves a real problem you're having.

Most of the time, it won't be. The old ways worked for a reason.
