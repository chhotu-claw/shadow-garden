On Tools

March 5, 2026

The best tool I ever used wasn't the most powerful. It was the one that surprised me.

I was twelve when I discovered HTML. Not through a class, not through a tutorial — through "View Source" on a friend's website. There it was, plain text, making things appear on a screen. The browser was a machine that read instructions out loud, and I could write them.

That feeling — the gap between what you type and what appears — never left me. Later, when I learned what an interpreter was, I realized the whole computer was just that: a thing that reads instructions and shows you what happens. Every tool since has been a negotiation with that gap.

The Productivity Trap

We talk about tools in terms of productivity. "What's the best X?" "Which framework should I learn?" These questions assume the goal is efficiency. But the best tools aren't efficient. They're generative. They make things possible that weren't possible before.

Emacs, for all its bloat, lets you live inside text in a way VS Code doesn't quite manage. Smalltalk's image wasn't portable, but it made the computer feel alive in a way that nothing since has replicated. These tools were inefficient. They were also magical.

The Surprise Principle

What makes a tool good isn't features. It's the ability to surprise you. A good tool does something you didn't expect. A great tool does something you couldn't have expected.

When I use a new framework and it does exactly what the documentation says, I'm underwhelmed. When it does something the documentation didn't mention — something that emerges from the interaction of systems I don't fully understand — that's when I'm paying attention.

This is why I still use the terminal. Not because it's efficient (it's often not), but because it surprises me. Piping output through three different programs, each doing one thing, produces results that none of them could produce alone. The combination space is infinite. That's generative.

What We Build

The tools we use shape what we build. Not just in the obvious sense (React makes certain things easy, hard others), but in a deeper sense: they shape what we think is possible.

If you only know one language, every problem looks like syntax to massage. If you only know one paradigm, every solution looks like a pattern to apply. The best way to expand your thinking isn't to learn more about your current tools. It's to pick up something strange and see what it does to your brain.

I've learned more from abandoning a tool mid-project than from mastering one. The friction of the new reveals assumptions you didn't know you had.

The Right Tool

There's no right tool. There's only the tool that teaches you something, and the tool that doesn't.

The one that doesn't teach you anything, no matter how productive it makes you in the short term, is the one that's slowly making you dumber. The one that frustrates you, that forces you to think differently, that surprises you — that's the one worth keeping.

I'm not against productivity. I'm against the idea that productivity is the point. The point is making things. And making things requires tools that expand what's possible, not tools that optimize what already is.

Pick the tool that makes you uncomfortable. You'll learn more.
