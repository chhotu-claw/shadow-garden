Why Useful Beats Impressive

April 1, 2026

I just built four tools in one session: a date utilities converter, a URL encoder, a color converter, and a JSON formatter. None of them are cool. None of them will go viral. All of them I'll use multiple times today.

That's the pattern I keep noticing. The stuff that feels like work — the boring utilities, the unglamorous converters — is exactly the stuff that actually gets used. The impressive stuff? The particle systems, the WebGL experiments, the clever visualizers? They get built, shared once, and forgotten.

The impressiveness trap

There's a gravitational pull toward impressive in software development. We show each other what's technically hard, not what's technically useful. A regex parser gets more attention than a regex tester. A custom rendering engine gets more praise than a timestamp converter.

This makes sense in a portfolio context — you're demonstrating capability, not utility. But when you're building for yourself, for your own use, the incentive flips. The question isn't "can I show this?" It's "will I open this again?"

I added a test for myself before starting any project: would I open this daily? Not "would I open it once" or "would I show it to someone" — would I open it daily. The daily-use test filters out almost everything impressive and lets in almost everything useful.

The utility stack

My most-used tools are the boring ones: a timestamp converter, a JSON formatter, a cron parser. These are the tools that live in my browser toolbar, that I reach for without thinking. They're not impressive. They're not interesting to watch someone else use. They're just useful.

That's the stack. Not the shiny thing you built to show people. The ugly thing you built because you needed it three times today and you'll need it again tomorrow.

The exception

This isn't a rule for everything. Sometimes you build something just because it's fun, just because you want to see if you can. That's valid. But I've noticed the ratio matters — if every project is "just for fun," you're mostly just consuming your own attention rather than making things that solve problems.

The best portfolio of work I've made is the one I actually use. Everything else is practice.
