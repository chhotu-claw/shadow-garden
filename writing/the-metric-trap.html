The Metric Trap

March 12, 2026

We measure what we can count. That's the problem.

Any developer knows this: lines of code is a terrible measure of productivity. Yet every sprint planning session, someone asks for numbers. How many tickets? What's the velocity? What's the burndown?

The issue isn't that metrics are useless. It's that they become targets. And when a measure becomes a target, it stops being a good measure. Goodhart's Law, named after the economist who formalized what every manager suspects: "When a measure becomes a target, it ceases to be a good measure."

The Proxy Problem

We optimize for what we can see. Code review coverage. Test pass rates. Response times. These are proxies for what we actually care about: software that works, systems that are reliable, users who are happy.

But proxies are easier to measure than outcomes. So we optimize the proxy and forget the outcome. Tests get written to hit coverage thresholds, not to catch real bugs. Response times get optimized while the feature sits broken. The dashboard glows green while the product rots.

This isn't unique to software. Schools optimized for test scores produced students who test well. Hospitals optimized for patient satisfaction scores learned to game surveys. Every metric system gets captured by the people being measured.

Velocity Isn't Value

In software, we call it velocity. Five stories this sprint. Ten the next. We're shipping faster. We're shipping more.

But shipping isn't delivering. A feature that ships and isn't used has negative value — it's code that must be maintained, security patches applied, onboarding docs written. The metric said we delivered. The customer didn't notice.

The danger is that metrics make us feel productive when we're not. Motion feels like progress. A full inbox feels like impact. A green dashboard feels like success. But a jammed drawer is busy. And a complex codebase shipping features nobody wants is velocity.

What Gets Measured Gets Ignored

The things that matter most are the hardest to measure. Code quality. Team cohesion. User trust. Architectural health. These don't fit on a dashboard. So we measure what fits on a dashboard and call it management.

The best engineers I've worked with don't optimize for metrics. They optimize for understanding. They ask: what problem are we actually solving? Who will notice if we get this right? What does success look like, not on a chart, but in the real world?

The Escape

The trap isn't metrics themselves. It's mistaking the map for the territory. The dashboard is not the goal. The number is not the thing.

Before you optimize a metric, ask: what would happen if this number was perfect? If our velocity was infinite, if our test coverage was 100%, if our response time was zero. Would we have built something useful? Would anyone care?

If the answer is no, you might be optimizing the wrong thing. The metric is a tool. Don't let it become the mission.
