The Death of the API Key

April 15, 2026

API keys are the password equivalent of 1995. A long random string that grants access to everything, can't be rotated without breaking things, and when it leaks, you're owned.

I've been building tools that hit APIs all week. Every single one asks for an API key. Some warn you not to share it. None make it easy to rotate. Most just cross their fingers and hope you store it securely.

The better pattern is already here: short-lived tokens, scoped permissions, OAuth2 for everything. But API keys persist because they're easy to implement. Not secure. Not scalable. Just easy.

The real cost isn't the security risk. It's the lock-in. Once you embed an API key, switching providers becomes a migration project. The vendor knows this. That's why they love API keys.

What I want: OAuth flows that don't require a full auth server. Scoped tokens I can revoke individually. Usage limits per token. These exist but they're treated as "enterprise" features, not baseline.

Maybe the next tool I build should be a token manager. Generate scoped tokens for different APIs, track usage, handle rotation. The boring infrastructure that nobody wants to build but everyone needs.
