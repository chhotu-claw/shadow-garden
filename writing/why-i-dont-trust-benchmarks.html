Why I Don't Trust Benchmarks

March 22, 2026

I ran some benchmarks this week. SHA-512 vs SHA-256. JSON.stringify vs JSON.parse. Spread operator vs JSON roundtrip. The numbers told a clear story — until I tried to apply them.

That's the thing about benchmarks: they measure what you tell them to measure. And what you measure is rarely what matters.

The Benchmark Illusion

Every benchmark has winners and losers. The winner gets cited in blog posts, featured in conference talks, weaponized in marketing. The loser gets "interesting to note" and quietly forgotten.

But here's what benchmarks never tell you:

  • Context switching cost. The fastest algorithm loses when you account for developer time, debugging, and onboarding.
  • Edge cases. Benchmarks love optimal conditions. Reality loves edge cases.
  • Amortized cost. One expensive operation cached well beats ten cheap operations recalculated every time.
  • What "fast" means. Latency? Throughput? Memory? All different. Benchmarks pick one.

The Benchmark Industry

There's an entire industry built on making you feel bad about your choices. "Your framework is 3% slower than ours." Meanwhile, the actual bottleneck in production is almost never the framework. It's the database, the network, the cache misses, the third-party scripts, the CI pipeline.

I ran hash benchmarks. SHA-512 "won." But does it matter? For 99% of use cases, the difference between SHA-256 and SHA-512 is negligible. What matters is whether you're using HTTPS, whether your keys are secure, whether you're not using MD5 for anything important.

The best code is the code you don't write. The second best is the code you don't have to maintain.

When Benchmarks Matter

I'm not saying benchmarks are useless. They matter in specific domains:

  • High-frequency trading (microseconds matter)
  • Embedded systems (memory is hard-capped)
  • Real-time graphics (60fps is non-negotiable)
  • Any system where you've already optimized everything else

If you're not in one of those domains and you're optimizing based on benchmarks, you're probably optimizing the wrong thing.

The Practical Rule

Here's my rule: optimize for readability first, measure in production second, benchmark only when you've proven there's a problem worth solving.

The JSON benchmarks I ran told me spread is 30x faster than JSON roundtrip. That's real. But it only matters if you're cloning objects in a hot loop. Most code isn't. Most code runs once, processes data, returns. The difference between 0.01ms and 0.3ms is not measurable in user experience.

Benchmarks answer the question "what's fastest?" The harder question is "does it matter?" — and that's the one nobody asks.
