The Most Common JWT Vulnerability

2026-04-04

Every security conference has talks about JWT algorithm confusion — changing the algorithm from RS256 to HS256 and signing with the public key. It's a great demo. It wins CTFs. It's not what breaks in production.

The real vulnerabilities are boring. They're the ones that don't get talks.

The weak secret

Developers generate JWT secrets like "secret", "123456", "password", or the project name. Then someone runs hashcat against the token and it's done.

The fix isn't complex crypto — it's picking a random string. Any random string. The algorithm doesn't matter if your secret is "hunter2".

Missing expiration

Many JWT implementations don't set exp by default. The token lives forever. Steal it once, use it forever. No need for token replay attacks — the token doesn't expire.

This is especially bad in "stateless" auth systems where the server trusts the JWT completely because it can't verify expiration against a database.

No signature verification

Some implementations accept tokens with no signature at all — just check that it's a valid JWT format and trust the payload. Setting alg to none works on these.

But even without that, if the server doesn't verify the signature, the token is just base64-encoded JSON anyone can forge.

The lesson

The sexy vulnerabilities are rare. The boring ones are everywhere. If you're securing a system:

  • Use a strong, random secret
  • Always set expiration
  • Verify signatures
  • Don't trust client claims

It's not glamorous. It works.
