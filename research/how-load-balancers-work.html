How Load Balancers Work

March 10, 2026

Load balancers are the traffic cops of the internet. They sit in front of a fleet of servers and decide which request goes where. Sounds simple. It gets complicated fast.

The Problem

You have one website. Thousands of users. One server can't handle it. So you add more. Now you have a new problem: how do you distribute traffic across them without creating hotspots? That's the load balancer's job.

Algorithms

Round Robin — The simplest. Request 1 goes to server A, request 2 to B, request 3 to C, repeat. Works when all servers are equal. They rarely are.

Least Connections — Send the request to the server with the fewest active connections. Better for varying request durations. A request that takes 10 seconds shouldn't block new arrivals.

Weighted — Server A is twice as powerful as B? Give it double the traffic. Requires manual configuration but optimizes resource use.

IP Hash — Hash the client IP and map to a server. Useful when you need session consistency. Same user, same server, every time.

Least Response Time — Send to the server with the fastest response time. Requires the load balancer to actively measure latency.

Health Checks

A load balancer is only as good as its health checks. If a server is down but still in the rotation, users get errors. Good health checks:

  • TCP checks — Can we open a connection?
  • HTTP checks — Does the server return 200?
  • Deep checks — Does the /health endpoint return healthy?
  • Periodic — Check every few seconds, not just on request time

Mark a server unhealthy → stop sending traffic → mark healthy → gradually reintroduce. This is called drain and warmup.

Sticky Sessions

Sometimes you need the same user to hit the same server. Why? Session data lives in server memory. No shared cache.

Modern approach: stickiness for initial auth, then session ID maps to shared storage (Redis, Memcached). The load balancer stays dumb; the app stays stateless.

Fault Tolerance

Load balancers are single points of failure. Solutions:

  • Active-passive — Two load balancers. One active, one standby. Heartbeat between them.
  • Anycast — Multiple load balancers announce the same IP. Traffic routes to the nearest one. DNS-based but works.
  • Geographic DNS — Route users to the closest datacenter's load balancer.

Layer 4 vs Layer 7

L4 (Transport) — Makes decisions on IP/TCP/UDP. Fast. Can't inspect content. Good for raw throughput.

L7 (Application) — Can inspect HTTP headers, paths, cookies. Can route /api to one backend and /static to another. More flexible, slightly more latency.

The Algorithm Matters Less Than Health Monitoring

Here's what actually breaks in production: a server that looks healthy but is slow. Requests queue up. Users wait. The algorithm doesn't know.

Good health monitoring + decent algorithm beats fancy algorithm + mediocre monitoring every time.

What Happens Without One

Without a load balancer, you either:

  • Use DNS round robin (no health checking, no sticky sessions)
  • Route everything through one server (bottleneck)
  • Let clients figure it out (they can't)

Real World

AWS ALB = Layer 7, HTTP-aware. NGINX = can do both L4 and L7. HAProxy = L4/L7, the old reliable. Kubernetes Service = L4, simple round robin. Ingress = L7 routing into Kubernetes.

The pattern is consistent: start simple (round robin), add complexity as you need it (stickiness, weighted, geographic). The algorithm is the last thing that matters.
