How Load Balancers Work

2026-03-09

Load balancers are the traffic cops of the internet. They sit between clients and a pool of servers, deciding which request goes where. The goal is simple: distribute load evenly, maximize throughput, minimize latency, and keep things running when a server dies.

The Core Problem

Imagine you run a popular service. One server handles 100 requests/second comfortably. At 1000 requests/second, it crumbles. You add nine more servers. Now you need something to split traffic across them. That's the load balancer's job.

Types of Load Balancing

DNS Load Balancing

The simplest approach. Your DNS resolver returns different IP addresses for the same domain, rotating through them (round-robin). No special hardware needed. Problem: it has no idea which servers are healthy or how loaded they are. A dead server still gets traffic.

Layer 4 (TCP/UDP) Balancing

The load balancer routes packets based on IP addresses and ports. It doesn't inspect the request content. Fast, simple, but dumb — it can't make smart decisions about which server is best.

Layer 7 (HTTP/HTTPS) Balancing

The balancer inspects the actual request — headers, URLs, cookies, paths. This enables smart routing: send /api/* to one pool, /static/* to another. More expensive computationally, but far more flexible.

Algorithms

Round Robin

Requests cycle through servers in order. Server A gets request 1, Server B gets request 2, and so on. Simple, fair if all servers are equal. Problem: if Server A is a beast and Server B is a toaster, you're wasting resources.

Least Connections

Send new requests to the server with the fewest active connections. Better for variable request durations. A quick request finishing frees up Server A, so send the next one there.

Weighted Algorithms

Assign weights to servers based on capacity. Server A (beast) gets weight 10, Server B (toaster) gets weight 2. It now gets 5x the traffic. Requires manual configuration and updates when hardware changes.

IP Hash

Hash the client's IP address to determine which server handles their requests. Ensures a client always hits the same server (useful for session data). Problem: uneven distribution if clients cluster geographically.

Least Response Time

Send traffic to the server with the lowest average response time. Requires the load balancer to measure latency for each server. Best for real-time applications where speed matters.

Health Checks

A load balancer is useless if it keeps sending traffic to a dead server. Health checks periodically probe each server:

  • TCP checks: Can I open a connection?
  • HTTP/HTTPS checks: Does the server respond to a specific URL with 200 OK?
  • Deep checks: Does the /health endpoint return a healthy status from the application?

When a server fails N consecutive checks, it's marked unhealthy and removed from the pool. When it recovers, it's added back. This is how you achieve zero-downtime deployments and fault tolerance.

Sticky Sessions

Sometimes you need a client to hit the same server repeatedly. Reasons: session state stored locally, WebSocket connections, caching benefits. Methods:

  • Cookie-based: Load balancer sets a cookie indicating which server to use
  • Source IP: Always hash the same client IP to the same server

Downside: breaks horizontal scalability. If that server dies, the client loses session. Modern architecture prefers stateless design with shared session stores (Redis, etc).

High Availability

A single load balancer is a single point of failure. Solutions:

  • Active-passive: Two load balancers, one active, one standby. Heartbeat keeps them in sync. Failover takes seconds.
  • Anycast: Multiple load balancers announce the same IP. Traffic routes to the nearest one. DNS-based geo-routing.

The Full Picture

Here's how modern infrastructure typically works:

  1. User hits your domain → DNS resolves to load balancer IP
  2. Load balancer (L7) examines the request, picks a backend server based on least connections + health
  3. Request goes to Server 3. Server 3 processes it, talks to database, returns response
  4. Load balancer returns response to client
  5. Server 3 goes down → health check fails → removed from pool automatically
  6. New server spins up → health check passes → added to pool

Why It Matters

Load balancers are the unsung heroes of availability. Without them, you'd have single points of failure, no fault tolerance, and no way to scale. They abstract away the infrastructure, letting you treat a cluster of servers as a single endpoint.

The algorithm you choose matters less than proper health checks and HA setup. A dumb round-robin with good health monitoring beats a sophisticated algorithm sending traffic to dead servers.

Load balancers are also where TLS termination often happens — decrypting once at the load balancer, sending plain traffic to backends. Reduces computational load on application servers, centralizes security.
