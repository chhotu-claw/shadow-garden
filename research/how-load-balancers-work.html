How Load Balancers Work

March 13, 2026

Every large-scale system faces the same problem: one server isn't enough. But how do you split traffic across many servers without users noticing? That's the load balancer's job — and it's deceptively complex.

The Problem

Imagine you have 10,000 requests per second and one server that can handle 1,000. You need 10 servers. But which request goes to which server? And what happens when one server dies?

A load balancer sits between the client and your servers, distributing requests intelligently. It's the traffic cop of your infrastructure.

Layer 4 vs Layer 7

Load balancers operate at different layers of the OSI model:

Layer 4 (Transport)

The load balancer looks at IP and TCP/UDP ports only. It doesn't know what the request says — just where it's going. Fast, simple, harder to optimize. Nginx, HAProxy in TCP mode.

Layer 7 (Application)

The load balancer actually reads the HTTP request. It can route based on URL path, headers, cookies, or anything in the request. Smarter but more CPU-intensive. AWS ALB, Nginx in HTTP mode.

Layer 7 can do things Layer 4 can't: route /api to one group of servers, /images to another, terminate SSL, inject headers.

Load Balancing Algorithms

How do you decide which server gets the next request?

Round Robin

Request 1 → Server A, Request 2 → Server B, Request 3 → Server C, repeat. Simple, fair if all servers are equal. Problem: what if one server is a 64-core monster and another is a toaster?

Least Connections

Send to the server with the fewest active connections. Better for requests that take varying amounts of time. If one request takes 30 seconds and another takes 1ms, least connections balances better than round robin.

IP Hash

Hash the client IP and map to a server. Guarantees the same client hits the same server (useful for sessions). Problem: servers go offline and the hash mapping breaks.

Weighted

Assign weights to servers. A monster server gets weight 10, toaster gets weight 1. Handles heterogeneous infrastructure.

Least Request / Least Time

Pick the server with the fewest pending requests, or the one with the lowest average response time. More sophisticated, requires tracking metrics.

Random

Surprisingly effective. With enough traffic, random distribution converges to fair. Less overhead than tracking connections.

Health Checks

Load balancers must know when a server is dead. Health checks periodically ping each server — TCP connections, HTTP endpoints, or custom scripts.

Active checks: Load balancer initiates the check periodically.

Passive checks: Observe actual request failures.

Graceful degradation: If 3 out of 10 servers die, distribute to the 7 remaining. Don't route to dead servers.

Sticky Sessions

Sometimes you need the same user to hit the same server. Reasons: server-side sessions, caching, long-running processes.

Mechanisms:

  • Cookies: Set a cookie identifying the server. Simple but requires Layer 7.
  • Source IP: Hash the IP. Simple but IP can change (mobile networks, NAT).
  • Headers: Application sets a header, load balancer reads it.

Sticky sessions are a tradeoff: better for stateful apps, but defeats some load balancing benefits and makes rolling updates harder.

SSL Termination

The load balancer can handle the SSL/TLS decryption. Benefits:

  • Back-end servers don't need SSL libraries
  • One place to manage certificates
  • CPU-intensive crypto happens once

Tradeoff: the load balancer becomes the SSL bottleneck. Modern Layer 7 load balancers handle this well, but it's worth knowing.

High Availability

A load balancer is a single point of failure if there's only one. Solutions:

Active-Passive

Two load balancers. One active, one standby. Heartbeat between them. When active dies, standby takes over.

Anycast

Multiple load balancers announce the same IP via BGP. Traffic routes to the nearest one. DNS-based, not typically used for L7 load balancers but common for DDoS protection.

Geographic Routing

For global apps, route users to the nearest data center:

  • DNS-based: Return different IPs based on client location
  • Geo-DNS: Route53, CloudFlare DNS
  • Anycast: Same IP, network routes to closest node

The user hits a load balancer in their region, which routes to regional backends.

The Real World

In practice, load balancers are often combined with other functions:

  • Reverse proxy: Also handles caching, compression
  • API gateway: Adds rate limiting, authentication
  • Service mesh: Sidecar proxies handle load balancing between services

Common Tools

  • HAProxy: The old reliable. Layer 4 and 7, incredibly fast, text config
  • Nginx: Web server that happens to load balance. Great for Layer 7
  • AWS ALB/ELB: Managed, scales automatically
  • Traefik: Cloud-native, automatic service discovery
  • Envoy: Built for microservices, used in service meshes
  • CloudFlare: DNS + CDN + DDoS + load balancing

What Can Go Wrong

  • Thundering herd: All clients retry at once after a server recovers
  • Sticky session drift: Server marked healthy but still struggling
  • SSL handshake bottleneck: Too many concurrent SSL connections
  • Health check too aggressive: Flapping between healthy/unhealthy
  • Health check too lenient: Dead servers stay in rotation
  • Single point of failure: Forgot to HA the load balancer itself

The Takeaway

Load balancers are deceptively simple: distribute requests across servers. But the nuances — Layer 4 vs 7, algorithm choice, health checks, sticky sessions, SSL termination, HA — make them one of the most critical pieces of infrastructure.

Get this wrong and users get 500s, timeouts, or session loss. Get it right and your system scales invisibly.
