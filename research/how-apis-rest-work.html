How APIs & REST Work

March 13, 2026

APIs are the glue holding the internet together. Every time you log into a site with Google, check the weather, or stream a movie, you're using one. But what actually happens when your app "calls an API"?

What Is an API?

An Application Programming Interface is a contract: "If you send me this, I'll send you that." That's it. It's not magic — it's just a formalized way for two pieces of software to talk.

The web runs on HTTP APIs — APIs delivered over the Hypertext Transfer Protocol. Your browser is actually an API client. When you visit https://api.example.com/users/123, you're making an HTTP request to an API endpoint.

REST: The Dominant Paradigm

REST (Representational State Transfer) isn't a protocol — it's an architectural style. Roy Fielding defined it in 2000 as a set of constraints, not rules. Most "REST APIs" only follow it partially.

The REST Constraints

  • Client-Server: Client and server are independent. The server doesn't care about the UI; the client doesn't care about data storage.
  • Stateless: Every request contains everything the server needs. No session stored on the server. Your auth token goes in every request.
  • Cacheable: Responses can say "cache this" — and clients (or proxies) can reuse them.
  • Uniform Interface: Resources, not actions. /users not /getUsers.
  • Layered System: You can put proxies, load balancers, or CDNs between client and server without either knowing.

HTTP Methods: The Verbs

REST uses HTTP methods as verbs. Each has semantic meaning:

MethodPurposeIdempotent?
GETRetrieve a resourceYes
POSTCreate a new resourceNo
PUTReplace a resource entirelyYes
PATCHPartial updateNo
DELETERemove a resourceYes

Idempotent means you can call it multiple times and get the same result. DELETE /users/123 twice — same outcome (user gone). POST /users twice — two users created.

A Real Example

# Get a user
GET /users/42 HTTP/1.1
Host: api.example.com
Accept: application/json

# Server responds
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 42,
  "name": "Alice",
  "email": "alice@example.com"
}
# Create a new user
POST /users HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "name": "Bob",
  "email": "bob@example.com"
}

# Server responds
HTTP/1.1 201 Created
Location: /users/43

{
  "id": 43,
  "name": "Bob",
  "email": "bob@example.com",
  "created_at": "2026-03-13T16:00:00Z"
}

Status Codes: The Server Talks Back

HTTP status codes tell you what happened. Learn these categories:

  • 1xx: Informational (rarely seen)
  • 2xx: Success
    • 200 — OK (generic success)
    • 201 — Created (POST succeeded, resource made)
    • 204 — No Content (success, nothing to return)
  • 3xx: Redirection
    • 301 — Moved Permanently
    • 304 — Not Modified (cached version is fine)
  • 4xx: Client error (you did something wrong)
    • 400 — Bad Request (invalid JSON, missing field)
    • 401 — Unauthorized (no/invalid auth)
    • 403 — Forbidden (auth'd but not allowed)
    • 404 — Not Found
    • 429 — Too Many Requests (rate limited)
  • 5xx: Server error (the API broke)
    • 500 — Internal Server Error
    • 502 — Bad Gateway (proxy/server issue)
    • 503 — Service Unavailable

Resources: The Nouns

REST uses resources — nouns, not verbs:

  • GET /users — list users
  • POST /users — create user
  • GET /users/42 — get one user
  • PUT /users/42 — replace user
  • PATCH /users/42 — update user
  • DELETE /users/42 — delete user

Nested resources work naturally:

  • GET /users/42/posts — posts by user 42
  • GET /users/42/posts/7 — post 7 by user 42
  • POST /users/42/posts — create post for user 42

Query Parameters & Filtering

Resources can accept parameters to filter, sort, or paginate:

GET /posts?status=published&sort=created_at DESC&limit=10&offset=20

Common patterns:

  • ?page=2&limit=20 — pagination
  • ?search=keyword — search
  • ?fields=title,author — partial response
  • ?expand=user — include related data

Authentication

APIs need to know who you're are. Common approaches:

API Keys

GET /protected-resource HTTP/1.1
X-API-Key: sk_live_123456789

Simple but all-or-nothing. Great for server-to-server.

Bearer Tokens (JWT)

GET /protected-resource HTTP/1.1
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

JWTs are self-contained tokens with claims (user ID, expiry). The server validates the signature, not a database lookup. Fast but can't be revoked before expiry.

OAuth 2.0

Delegated access. "Login with Google" uses OAuth. Your app gets a short-lived access token. If you need a refresher on OAuth, check out the OAuth deep dive.

Versioning

APIs change. How do you version without breaking clients?

  • URL path: /v1/users, /v2/users — most common
  • Header: Accept: application/vnd.api+json;version=2
  • Query: /users?version=2 — rare

HATEOAS: The Holy Grail

HATEOAS (Hypermedia as the Engine of Application State) means responses include links to related actions:

{
  "id": 42,
  "name": "Alice",
  "links": {
    "self": "/users/42",
    "posts": "/users/42/posts",
    "avatar": "/users/42/avatar"
  }
}

The client discovers API surface from responses. True REST — rarely implemented fully.

What's NOT REST

  • RPC over HTTP: POST /api/getUser — this is just remote procedure call, not REST
  • GraphQL: Query language, not REST. Single endpoint, client specifies fields.
  • WebSockets: Bidirectional, stateful — violates REST constraints

Why REST Still Wins

  • Ubiquity: Every developer understands it
  • Caching: GET responses are trivially cacheable
  • Simplicity: Four verbs, status codes, JSON
  • Tooling: Browsers, curl, Postman all speak HTTP natively

GraphQL solves real problems (over-fetching, multiple round-trips) but adds complexity. gRPC is faster for service-to-service but harder to debug. REST is the "Unicode of APIs" — not perfect, but everyone speaks it.
