How UUIDs Work

March 21, 2026

Every distributed system faces the same problem: how do you generate unique IDs across multiple machines without coordination? The naive approach — a central ID server — creates a bottleneck. The smarter approaches are fascinating.

UUID v4: Random Chaos

The most common UUID is v4: 122 bits of random data. That's 2^122 possible values — effectively infinite. No coordination needed, no state to maintain.

But there's a catch:

  • No sorting: Random UUIDs have no temporal order. This kills database performance for inserts.
  • Fragmentation: B-tree index pages fill up randomly, increasing disk usage.
  • Privacy: Version 4 UUIDs can be generated offline, but they leak nothing about when or where they were created.

UUID v1: Timestamp + MAC

UUID v1 combines a timestamp (60 bits) with a MAC address (48 bits). Sortable, but:

  • Reveals when and where (which machine) created the UUID
  • MAC addresses can be spoofed or unavailable (containers, VMs)
  • Clock drift can cause collisions

UUID v7: The Best of Both

UUID v7 is a 2022 standard that combines a Unix timestamp (48 bits) with random data (80 bits). This gives you:

  • Sortable: Time-ordered, great for database indexes
  • No leakage: Random portion hides the timestamp from casual observers
  • Collision-resistant: 80 bits of randomness makes collisions effectively impossible

This is now recommended for most use cases. PostgreSQL, Redis, and most modern databases handle v7 efficiently.

ULID: Alternative Approach

ULID is similar to v7 but uses Crockford's Base32 encoding (no 0, O, I, L to confuse). 48 bits timestamp + 80 bits randomness. Universally sortable.

The encoding is friendlier for URLs: 01ARZ3NDEKTSV4RRFFQ69G5FAV vs 0192d140-7c88-7f88-8a7f-0123456789ab

Snowflake IDs: The Twitter Approach

Twitter's snowflake (and similar systems like KSUID) splits 64 bits into:

  • 1 bit: always 0 (sign)
  • 41 bits: timestamp (milliseconds since custom epoch)
  • 10 bits: machine/worker ID
  • 12 bits: sequence number

This requires coordination (assigning machine IDs) but gives you:

  • Sortable, time-ordered IDs
  • Embedded metadata (which machine generated it)
  • Compact (64 bits vs 128 for UUID)

The Takeaway

UUID v4 is the default for a reason — it works everywhere. But UUID v7 is the upgrade most projects should switch to. You get sortability (critical for databases) without the privacy issues of v1.

The 128-bit size doesn't matter for most applications. What matters is whether your IDs help or hurt your database's performance.
