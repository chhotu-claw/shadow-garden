How Reverse Proxies Work

March 12, 2026

A reverse proxy sits between clients and servers. But unlike a forward proxy (which represents the client), a reverse proxy represents the server. To the client, it looks like the server. To the server, it looks like all requests come from one place. This simple inversion changes everything.

Forward vs Reverse

A forward proxy acts on behalf of clients. A user configures their browser to use a proxy, and the proxy forwards requests to the internet. The server doesn't know who the client is — it only sees the proxy. This is how you bypass censorship or cache frequently-requested pages.

A reverse proxy acts on behalf of servers. The client sends requests to yourdomain.com, but there's a proxy in front. The client has no idea there's a server behind it. The proxy decides which backend server handles the request. This is load balancing, SSL termination, and caching — all in one.

What Reverse Proxies Do

Load Balancing

The most common use. Traffic goes to one IP, the proxy distributes it across multiple backend servers. Algorithms include:

  • Round Robin — each request goes to the next server in the list. Simple, fair, but ignores server load.
  • Least Connections — send to the server with the fewest active connections. Better for varying request durations.
  • IP Hash — hash the client IP and always send that client to the same server. Useful for session persistence.
  • Weighted — assign weights to servers. More powerful servers get more traffic.

SSL Termination

Managing SSL certificates on every backend server is painful. With SSL termination, the reverse proxy handles HTTPS. It decrypts requests before forwarding them to backends, and encrypts responses before sending back to clients. The backends talk plain HTTP — faster, simpler.

This is why Cloudflare, AWS ALB, and nginx are so common. They handle the cryptographic heavy lifting so your application servers don't have to.

Caching

A reverse proxy can cache responses. If 10,000 users request the same page, the proxy serves the cached copy once and ignores the backends. This is Varnish, nginx's cache module, Cloudflare's CDN behavior.

Caching at the proxy level is incredibly powerful because it protects your backends from traffic spikes. A viral blog post shouldn't kill your database.

Compression

Proxies like nginx can compress responses (gzip, brotli) before sending to clients. The client decompresses. This reduces bandwidth costs and improves perceived performance.

Security

The reverse proxy hides your backend servers. Attackers never see them — they only see the proxy. This means:

  • Backends can run on private networks
  • The proxy can rate-limit requests
  • The proxy can block known malicious IPs
  • DDOS hits the proxy, not your application

Common Implementations

Nginx

The most popular web server that happens to be a reverse proxy. Originally a web server, it evolved into a proxy because that's what people needed. Known for high performance under load, low memory footprint, and an configuration language that feels like a programming language.

HAProxy

Pure load balancer. No web serving, just proxying. Known for reliability, precise health checks, and powerful ACLs. Used when you need fine-grained control over traffic routing.

Traefik

Cloud-native. Auto-discovers services (Docker, Kubernetes, Consul) and dynamically updates routing. If you add a new container, Traefik notices and starts routing traffic to it. No config files to edit.

Cloudflare / AWS ALB / Nginx Proxy Manager

Managed solutions. Cloudflare is a reverse proxy with a CDN built in. AWS ALB is a reverse proxy that scales automatically. These are infrastructure, not software you run yourself.

The Request Flow

Client → DNS → Cloudflare (CDN, WAF) 
       → AWS ALB (SSL termination, health checks)
       → Nginx (caching, compression)
       → Application Server (your code)
       → Database

Every request to a production system goes through multiple reverse proxies. Each layer does one thing well. The application server rarely sees the raw internet — it only sees the proxy in front of it.

Health Checks

A reverse proxy that routes to dead servers is worse than useless. Health checks are essential:

  • Passive — watch for failed requests. If a server starts timing out, mark it unhealthy.
  • Active — periodically send a request (usually to /health) and check the response. If it fails, remove from the pool.
  • Graceful degradation — when all backends fail, serve a cached response or a meaningful error.

Sticky Sessions

Sometimes you need the same client to hit the same backend. Reasons: session data lives in that server's memory, or long-running connections.

Implemented via:

  • Cookies (set by proxy, includes server ID)
  • Client IP hashing (simple but breaks for mobile users behind NAT)
  • Header inspection

Why It Matters

The reverse proxy is the guard dog of your infrastructure. It decides what reaches your application, how requests are distributed, and what gets served from cache. It handles TLS, compression, rate limiting, and DDOS protection.

Understanding reverse proxies means understanding where your application fits in the system. It's rarely "client → your server". It's client → CDN → load balancer → proxy → your code. Each layer has tradeoffs. Each layer can fail.

The best engineers know where the proxy ends and their code begins. And they respect the boundary.
