How File Systems Work

2026-03-08

Every file on your computer lives inside a filesystem — a layered abstraction that turns raw bytes into named objects you can find, modify, and share. But how does it actually work?

The Job of a Filesystem

A filesystem does five things:

  • Stores data — maps file contents to physical storage
  • Organizes files — hierarchical directories
  • Tracks metadata — permissions, timestamps, ownership
  • Manages space — allocates free blocks, handles fragmentation
  • Protects data — journaling, checksums, redundancy

From Disk to File: The Structure

Blocks and Sectors

Storage devices read and write in sectors (typically 512 bytes or 4KB). Filesystems operate in blocks — groups of sectors (usually 4KB). You can't allocate a single byte; everything lives in blocks.

Inodes: The Metadata Index

On Unix-like systems (ext4, XFS, Btrfs), every file is represented by an inode — a data structure containing:

  • File size
  • Timestamps (created, modified, accessed)
  • Ownership (UID, GID)
  • Permissions
  • Pointers to data blocks

The inode doesn't store the filename — that's in the directory entry, which maps names to inode numbers. This is why hard links work: multiple names can point to the same inode.

Data Pointers

An inode contains direct and indirect pointers to data blocks:

  • Direct pointers — 12 pointers to the first 12 blocks (48KB for 4KB blocks)
  • Single indirect — pointer to a block of pointers
  • Double indirect — pointer to a block of blocks of pointers
  • Triple indirect — pointer to a block of blocks of blocks of pointers

This tree structure lets a single file span terabytes — the theoretical max for ext4 is 16TB.

NTFS: The Windows Approach

NTFS uses a different model — the Master File Table (MFT). Every file has an entry in the MFT, and each entry stores both metadata and (for small files) the data itself.

NTFS features:

  • Compression — block-level compression using LZNT
  • Encryption — built-in EFS (Encrypting File System)
  • Journaling — records transactions before applying
  • Alternate data streams — multiple data streams per file
  • Sparse files — holes that don't consume space

Journaling: Preventing Corruption

When you save a file, the filesystem must:

  1. Allocate data blocks
  2. Write the data
  3. Update metadata (inode/MFT)
  4. Mark the file as complete

If the system crashes mid-operation, you get corruption — partial writes, orphaned blocks, inconsistent structures.

Journaling solves this: before making changes, the filesystem writes a journal entry describing the planned operation. After the crash:

  • Replay — complete the operation from the journal
  • Rollback — undo partial changes

Journaling only protects metadata by default (ext3/4). Full data journaling (write-through) is slower but safer.

Directories: Just Special Files

Directories are files that contain directory entries. Each entry maps:

  • Filename
  • Inode number (or MFT reference on NTFS)
  • Metadata (optional, depending on filesystem)

When you ls, the OS reads the directory file, parses entries, and looks up inodes for each name.

How Reads and Writes Actually Work

Reading a File

  1. User requests /home/user/doc.txt
  2. Kernel looks up root directory → finds inode for /home
  3. Reads /home directory → finds inode for /home/user
  4. Reads /home/user → finds inode for doc.txt
  5. Reads inode → gets data block pointers
  6. Reads data blocks → returns content to user

Writing a File

  1. Find or create inode
  2. Allocate data blocks (from free space bitmap)
  3. Write data to blocks
  4. Update inode with pointers and metadata
  5. Update parent directory with new entry
  6. Write journal entry (if journaling)

What Actually Matters

  • Block size — matches your workload. 4KB is standard. Databases often use 8KB+.
  • Journal mode — ordered (default on ext4) is usually best balance.
  • Free space — below 10-20% free, performance drops. Filesystem fragmentation increases.
  • fsck time — on large filesystems, fsck can take hours. Btrfs and XFS skip this.

The Bottom Line

A filesystem is a database for files — indexing data blocks, tracking metadata, ensuring consistency. The concepts (inodes, directories, journaling) apply across systems, even if the implementations differ. Next time you save a file, remember: dozens of operations happen in milliseconds, blocks are allocated, journals written, inodes updated. Your OS handles it all.
