How Compilers Work

March 15, 2026

Every program you run — from a Python script to a C++ game — goes through the same transformation: human-readable source code becomes machine-executable instructions. That transformation is the compiler's job. It's one of the most underrated pieces of software engineering, and understanding it changes how you think about code.

The Big Picture

A compiler isn't a single step. It's a pipeline:

  1. Lexical Analysis (Lexing) — Raw text → tokens
  2. Parsing — Tokens → Abstract Syntax Tree (AST)
  3. Semantic Analysis — AST → typed, validated AST
  4. Intermediate Representation (IR) → Platform-agnostic code
  5. Optimization — IR gets faster, smaller, better
  6. Code Generation → Target machine code

Lexing: Breaking Text into Tokens

The first step is turning a stream of characters into meaningful chunks. This is lexing (short for lexical analysis). A lexer takes input like:

let x = 42 + y;

And produces tokens:

LET, IDENTIFIER("x"), EQUALS, NUMBER(42), PLUS, IDENTIFIER("y"), SEMICOLON

The lexer applies regex-like rules to identify keywords, operators, identifiers, and literals. It discards whitespace and comments — they're for humans, not the machine.

Tools: lex, flex, ANTLR lexer generators.

Parsing: Building Structure

Tokens are flat. Programs have structure — nested expressions, statements, blocks. The parser takes tokens and arranges them into a tree.

The tree is called an Abstract Syntax Tree (AST). Each node represents a construct:

       =
      / \
     x   +
        / \
       42  y

Parsers use grammars — formal rules that define valid syntax. Context-free grammars (CFGs) are the standard. The parser validates that the token stream follows these rules and errors out if you write syntactically invalid code.

Tools: yacc, bison, ANTLR parser generators.

Semantic Analysis: Meaning Matters

Syntactically correct doesn't mean correct. The parser accepts:

x + "hello"

If x is a number, this is a type error — but the parser doesn't know about types yet. That's the semantic analyzer's job.

It performs:

  • Type checking — Are operations valid for the types involved?
  • Scope resolution — What does each identifier refer to?
  • Symbol management — Building the symbol table

The output is an annotated AST with type information attached to every node.

Intermediate Representation: The Universal Language

Here's the key insight: compiling to machine code directly is hard. Instead, most compilers translate to an Intermediate Representation (IR) — a platform-agnostic, low-level format.

IR looks something like this (three-address code):

t1 = 42
t2 = load y
t3 = t1 + t2
store t3, x

Why bother? Because IR lets you:

  • Optimize code regardless of the target CPU
  • Support multiple source languages → one compiler backend
  • Support multiple targets (x86, ARM, WASM) from one frontend

Examples: LLVM IR, GCC's GIMPLE, Java bytecode.

Optimization: Making It Better

This is where compilers earn their keep. The optimizer takes IR and applies transformations:

  • Constant folding — 2 + 3 → 5
  • Dead code elimination — Remove unused variables/functions
  • Loop unrolling — Expand small loops to reduce branch overhead
  • Inlining — Replace function calls with the function body
  • Common subexpression elimination — Don't recompute a + b twice

Modern compilers perform hundreds of optimization passes. LLVM's pass pipeline is massive. Some are machine-independent (works on IR), others are machine-dependent (exploit CPU-specific instructions).

Hot topic: Link-time optimization (LTO) — optimizing across compilation units after linking. Rust and C++ support this for whole-program optimization.

Code Generation: The Final Step

Now we translate IR to actual machine code for the target architecture. This involves:

  • Instruction selection — Which CPU instructions correspond to IR operations?
  • Register allocation — How do we fit variables into finite CPU registers?
  • Instruction scheduling — Reorder to avoid pipeline stalls

Output is either:

  • Assembly — human-readable mnemonics (then assembled to machine code)
  • Object files — directly in machine code, with relocations

For interpreted languages (Python, JS), the "compiler" stops at bytecode or IR. The interpreter then runs that.

Why This Matters

You don't need to write a compiler to benefit from understanding them. But you will:

  • Write better code — Know what your language actually does
  • Debug better — Understand why compile errors look the way they do
  • Choose tools wisely — LLVM-backed languages (Rust, Swift, Clang) share optimization passes
  • Appreciate tradeoffs — Compilation speed vs. runtime speed, interpreted vs. compiled

The Compiler Landscape Today

Modern compilers are vast ecosystems:

  • LLVM — The dominant framework. Clang (C/C++), Rust (via rustc), Swift, Julia all use it. It provides the optimizer and code generator; languages just write a frontend.
  • GCC — The GNU Compiler Collection. Older, massive, supports many languages. Its middle-end is called GIMPLE.
  • rustc — Uses LLVM backend but has its own sophisticated borrow checker and type system.
  • GraalVM — Polyglot: compile Java, JS, Python, Ruby to native binaries or run as JVM languages.

JIT (Just-In-Time) compilers are a different beast — they compile at runtime, often using profiling data to guide optimization. V8 (Chrome's JS engine) is the famous example.
