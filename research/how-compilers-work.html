How Compilers Work

March 11, 2026

The magic trick: you write printf("hello"), press run, and somehow the machine understands. But between your code and the CPU is a pipeline of transformation so elegant it's easy to forget it exists. Here's how a compiler actually works.

Source → Lexer → Parser → IR → Optimizer → ASM → Binary

The Four Stages

A compiler isn't one program — it's four programs chained together. Each one transforms the representation of your code into something the next one can work with.

1. Lexical Analysis (Lexer)

The lexer reads your source code as a stream of characters and groups them into tokens. Think of it as tokenizing a sentence into words.

// Source
if (x > 5) { return x; }

// Tokens
IF, LPAREN, IDENTIFIER(x), GT, NUMBER(5), RPAREN, LBRACE, 
KEYWORD(return), IDENTIFIER(x), SEMICOLON, RBRACE

The lexer throws away whitespace and comments — they only matter to humans. It also tracks line and column numbers for error reporting.

2. Parsing

The parser takes tokens and builds a parse tree (or Abstract Syntax Tree, AST) that represents the grammatical structure of your program. This is where syntax errors are caught.

// AST for: if (x > 5) { return x; }

IfStatement
├── Condition: BinaryOp(GT)
│   ├── Left: Identifier(x)
│   └── Right: Literal(5)
└── Then: Block
    └── ReturnStatement
        └── Identifier(x)

The parser uses a grammar — a formal specification of what's valid. Languages are defined by context-free grammars (CFGs), which is why you can't have unmatched parentheses in valid code.

3. Semantic Analysis

The AST is syntactically valid but might still be meaningless. Semantic analysis catches that:

  • Type checking — is x + 5 valid when x is a string?
  • Scope resolution — is this variable defined?
  • Constant folding — can we compute 2 + 3 at compile time?

This is where symbol tables come in — a data structure that tracks every variable, function, and type in your program.

4. Code Generation

Now we generate target code. In older compilers, this was directly to assembly or machine code. Modern compilers go through an intermediate step:

Intermediate Representation (IR)

IR is a platform-independent representation of your program. It's the secret sauce that lets GCC target x86, ARM, RISC-V, and dozens of other architectures from the same codebase.

; x86 Assembly (target-specific)
mov eax, [ebp-4]
cmp eax, 5
jle .L2

; IR (platform-independent)
%1 = load i32 %x
%2 = icmp sgt i32 %1, 5
br i1 %2, label %then, label %else

LLVM IR is the most famous example. It's written in SSA (Static Single Assignment) form — every value is assigned exactly once, which makes optimization easier.

The Optimizer

Between IR and final code generation, there's the optimizer. This is where compilers earn their keep:

  • Constant propagation — replace variables with known values
  • Dead code elimination — remove code that does nothing
  • Loop unrolling — reduce branch overhead
  • Inline expansion — replace function calls with function body
  • Register allocation — map variables to CPU registers
Optimization levels: -O0 (none), -O1 (basic), -O2 (aggressive), -O3 (very aggressive), -Os (size), -Ofast (fast but may break floating point)

Code Generation

Finally, the IR is lowered to target assembly. This involves:

  • Instruction selection — pick the right CPU instruction
  • Register allocation — Graph coloring (NP-complete, solved heuristically)
  • Calling conventions — how to pass parameters (registers vs stack)
  • Peephole optimization — local improvements at the assembly level

Linking

Your compiler outputs an object file. But programs are built from multiple source files and libraries. That's the linker's job:

  • Symbol resolution — match function calls to their definitions
  • Address relocation — fix up addresses after combining object files
  • Symbol stripping — remove debug info for production
# Compilation pipeline
gcc -c main.c -o main.o    # Compile to object
gcc -c foo.c -o foo.o      # Compile to object  
gcc main.o foo.o -o app    # Link together

Interpreter vs Compiler

Here's the distinction people get wrong:

  • Compiler — translates entire program before execution
  • Interpreter — executes program statement-by-statement

Most "interpreted" languages (Python, Ruby) actually compile to bytecode first. Most "compiled" languages (C, Rust) have JIT compilers now. The line is blurry.

JIT compilation — compiles at runtime, can optimize based on actual runtime data. V8 (JavaScript), HotSpot (JVM), and .NET CLR all use JIT. It's why JavaScript got fast.

Why It Matters

Understanding compilers isn't just for language implementers. It helps you:

  • Write faster code (data-oriented design, cache friendliness)
  • Debug mysterious crashes (knowing the calling convention)
  • Choose languages wisely (compilation time vs runtime performance)
  • Build DSLs and tools
"The compiler is the programmer's best friend. It tells you exactly what's wrong, in language you can understand." — somewhere in every error message

Next time you hit compile, remember: your human-readable code just went through lexing, parsing, semantic analysis, optimization, code generation, and linking. In seconds. That's not magic — it's forty years of compiler engineering.
