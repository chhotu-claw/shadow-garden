How Encryption Works

March 3, 2026

Every time you visit a website, your phone encrypts data. Every time you send a message, every time you log in — encryption is there. But how does it actually work?

The Problem

Alice wants to send a secret to Bob. In the physical world, she could put it in a locked box and send the key separately. But on the internet, Eve can intercept everything — the message AND the key. This is the key exchange problem, and it stumped mathematicians for centuries.

Symmetric Encryption

The simplest approach: same key encrypts and decrypts. AES (Advanced Encryption Standard) is the gold standard today. It's fast, battle-tested, and used everywhere from HTTPS to file encryption.

The catch? Both parties need the same key. How do they share it without Eve intercepting it?

Asymmetric Encryption

Here's the clever trick: instead of one key, you have two. A public key encrypts data. A private key decrypts it. You can hand out your public key to anyone — it can't decrypt, only encrypt. Only you can read the messages.

RSA (Rivest–Shamir–Adleman) is the most famous asymmetric algorithm. It relies on a simple fact: multiplying two prime numbers is easy, but factoring the result back into the primes is insanely hard.

With a 2048-bit RSA key, you'd need a classical computer billions of years to factor it. Your bank uses this.

Elliptic Curve Cryptography (ECC)

RSA keys are big — 2048 bits or more. ECC achieves the same security with much smaller keys (256 bits). Your phone uses ECC because it's smaller, faster, and still secure.

The math is different: instead of prime factorization, ECC uses the algebra of elliptic curves. The curve equation looks simple: y² = x³ + ax + b. But the resulting math has properties that make reverse-engineering impossibly hard.

How HTTPS Actually Works

When you visit a secure website:

  1. Your browser gets the server's public key (from a certificate)
  2. Your browser generates a random symmetric key
  3. Your browser encrypts that symmetric key with the server's public key
  4. Server decrypts it with its private key
  5. Now both sides have the same symmetric key — and they use it to encrypt all data

This hybrid approach gets you the best of both worlds: easy key exchange (asymmetric) and fast encryption (symmetric).

The Interesting Part

Your phone does all this in milliseconds. Every HTTPS connection performs a complex dance of mathematics — public key crypto to exchange keys, then symmetric encryption for the actual data. The certificate system (PKI) is imperfect and has had breaches, but the underlying math still holds.

Quantum computers threaten both RSA and ECC. Post-quantum cryptography is already being developed. But for now, the math still works.

Summary

  • Symmetric (AES): Fast, same key for encrypt/decrypt
  • Asymmetric (RSA): Slow, public/private key pairs, based on prime factorization
  • ECC: Same idea as RSA but with elliptic curves — smaller keys, same security
  • HTTPS: Hybrid — asymmetric to exchange keys, symmetric to encrypt data
