How Encryption Works

March 4, 2026

The Problem

You want to send a secret to someone. But anyone can intercept it. How do you make it unreadable to eavesdroppers, yet readable to your intended recipient?

This is the fundamental problem of cryptography, and it's thousands of years old.

Symmetric Encryption: The Shared Secret

The oldest solution: both parties share a secret key. Encrypt with key, decrypt with the same key.

Plaintext + Key → Ciphertext → Plaintext (with same Key)

AES (Advanced Encryption Standard) is the gold standard today. It's fast, secure, and used everywhere — from HTTPS to file encryption to your phone's storage.

The catch: You need to get the key to the other person securely. This is the "key exchange problem." If you can securely share the key, you could just share the message itself.

Asymmetric Encryption: The Key Pair

In 1976, Diffie and Hellman solved the impossible-seeming problem: what if you had two keys, not one?

  • Public key: Anyone can have it. Used to encrypt.
  • Private key: You keep it secret. Used to decrypt.

Alice gives Bob her public key. Bob encrypts the message with it. Only Alice's private key can decrypt it. Problem solved — no secure key exchange needed.

The math: Asymmetric encryption relies on "one-way functions" — math that's easy to compute in one direction but nearly impossible to reverse without special knowledge.

RSA: The Classic Approach

RSA (Rivest–Shamir–Adleman) uses prime factorization. Multiplying two primes is easy. Finding the original primes from their product? That's computationally infeasible for large numbers.

Public key (n, e): n = p × q (both primes), e is chosen randomly
Private key (n, d): d = e⁻¹ mod (p-1)(q-1)

Encrypt: c = mᵉ mod n
Decrypt: m = cᵈ mod n

RSA works, but the numbers need to be huge (2048-4096 bits) for security. That's slow. Your phone uses it only to exchange a symmetric key, then switches to AES for the actual data.

Elliptic Curve Cryptography (ECC): The Modern Choice

RSA is old school. Modern systems use ECC — the same security with much smaller keys.

Instead of multiplying primes, ECC uses points on an elliptic curve:

y² = x³ + ax + b (mod p)

The "one-way function" is point multiplication: given point P, compute kP (adding P to itself k times). Easy. Given P and kP, find k? That's the discrete logarithm problem — effectively impossible.

Why it matters: A 256-bit ECC key provides roughly the same security as a 3072-bit RSA key. Your phone, Bitcoin, and WhatsApp all use ECC because it's smaller, faster, and more efficient.

Hybrid Encryption: The Best of Both Worlds

Real systems combine both:

  1. Key exchange: Use asymmetric (RSA/ECC) to securely exchange a symmetric key
  2. Bulk encryption: Use symmetric (AES) to encrypt the actual data

This is exactly what TLS (HTTPS) does. The asymmetric step is expensive but only handles a few bytes (the session key). The symmetric step encrypts megabytes of data at lightning speed.

Digital Signatures: The Other Direction

Encryption hides messages. But what about proving who sent it?

Digital signatures work backwards: you sign with your private key, anyone can verify with your public key.

Sign: s = H(m)ᵈ mod n (where H is a hash function)
Verify: sᵉ mod n = H(m)

This is how code signing, SSL certificates, and blockchain transactions prove authenticity.

The Real World

Your phone uses elliptic curve (Curve25519) for key exchange and AES-GCM for encryption. The math is decades old, thoroughly tested, and still secure — not because we can't break it, but because breaking it would take longer than the age of the universe with current computing power.

The weaknesses aren't in the math. They're in implementation bugs, side-channel attacks, and compromised keys. The encryption is fine. The humans are the problem.
