How DHCP Works

March 13, 2026 · DEEP DIVE

Every time you connect to WiFi, your phone, laptop, or smart TV gets an IP address — automatically. You never configure it. You never think about it. That's DHCP doing the work.

The Problem

In the early days of networking, admins manually assigned IP addresses to every device. This was error-prone, tedious, and didn't scale. If you moved a computer to a different network, you had to reconfigure it. DHCP solved this: a protocol that automatically assigns IP addresses, subnet masks, gateways, and DNS servers to any device that asks.

The Four-Step Dance (DORA)

DHCP uses a four-step process that's easy to remember: DORA.

1. DISCOVER — "Is there a DHCP server out there?"

When a device connects to a network, it has no IP address. It can't communicate with anything yet. So it broadcasts a DISCOVER message to IP address 255.255.255.255 (broadcast to all devices on the local network). This packet essentially screams: "Hey, is anyone there? I need an IP address!"

The device also picks a transaction ID — a random number it uses to identify its own messages. This helps it match responses to requests.

2. OFFER — "I can give you this address"

If there's a DHCP server on the network, it hears the DISCOVER. The server has a pool of IP addresses it can hand out. It picks one that's available (not already assigned) and sends back an OFFER packet.

The OFFER contains:

  • The IP address being offered
  • Subnet mask
  • Gateway (router) IP
  • DNS server IPs
  • Lease time (how long the address is valid)

3. REQUEST — "I'll take that one"

The client receives the OFFER (it might get multiple from multiple servers, but usually picks the first). It sends back a REQUEST saying "Yes, I'll use 192.168.1.105." This confirms which address it wants.

4. ACKNOWLEDGE — "It's yours"

The DHCP server sends an ACK (acknowledgment): "Confirmed. 192.168.1.105 is yours for the next 24 hours."

Only now does the client configure its network interface with the new IP address. It can now communicate on the network.

The Lease

DHCP addresses aren't permanent. They're leased. The lease has a time-to-live, typically 24 hours, but it can be anything from minutes to weeks.

Lease Renewal

When half the lease time passes (at 12 hours), the client automatically tries to renew. It sends a REQUEST directly to the DHCP server (no need to broadcast anymore — it now has an IP). If the server ACK's, the lease extends. If the server is gone or the address was reclaimed, the client goes back to DORA.

Lease Expiry

If the lease expires and renewal fails, the client loses its IP. It goes back to DORA from scratch. This is why your device might briefly lose connectivity if you restart your router after the lease expires.

What DHCP Actually Delivers

DHCP doesn't just give you an IP. It gives you everything you need to function on the network:

  • IP Address — Your unique identifier on the local network
  • Subnet Mask — Tells your device which addresses are "local" vs "remote"
  • Default Gateway — The router's IP (where you send traffic destined for the internet)
  • DNS Servers — Where to look up domain names
  • Lease Time — How long this configuration is valid

Your computer essentially says: "Here's my link-local address (169.254.x.x), now DHCP please help me become a real network citizen."

DHCP in the Real World

Your Home Network

Your WiFi router is usually the DHCP server. It has a pool like 192.168.1.100 - 192.168.1.199 and hands them out to phones, laptops, TVs, and guests. Some routers let you reserve specific IPs for specific devices (by MAC address) so your NAS always gets 192.168.1.50.

Enterprise Networks

Big networks have dedicated DHCP servers (Windows Server, Linux dhcpd, or network equipment). They often use VLANs — each floor or department gets its own subnet, and DHCP knows which VLAN you're on based on which switch port or WiFi network you're connected to.

IPv6 Changes Things

IPv6 has SLAAC (Stateless Address Autoconfiguration) as the default — your device essentially generates its own IP from your MAC address. But DHCPv6 also exists for stateful address assignment (centralized control like IPv4 DHCP). Many networks use both.

What Happens When DHCP Breaks

No DHCP Server Available

If there's no DHCP server and you have "Obtain IP automatically" selected, your device falls back to APIPA (Automatic Private IP Addressing). It assigns itself a link-local address in the 169.254.0.0/16 range. You can talk to other APIPA devices on the same network, but you can't reach the internet because there's no gateway.

IP Conflict

If two devices somehow get the same IP (maybe a static assignment overlaps with DHCP pool, or a lease didn't expire properly), they'll both respond to ARP requests for that IP. Network connectivity becomes intermittent or broken. Fix: release/renew on both, or change the DHCP pool.

Exhausted Pool

If the DHCP pool is too small and you have more devices than addresses, new devices can't connect. Fix: expand the pool, shorten lease times, or add a second DHCP server (carefully — two DHCP servers on the same network can cause chaos if they don't coordinate).

Rogue DHCP Server

If someone plugs in an unauthorized DHCP server (a "rogue" server), it might give devices incorrect gateway or DNS settings, redirecting traffic through an attacker's machine. Managed networks use DHCP snooping on switches to block unauthorized DHCP servers.

Under the Hood

Ports

DHCP uses UDP ports:

  • Server listens on port 67
  • Client listens on port 68

Broadcast vs Unicast

Initial DORA is broadcast (255.255.255.255) because the client has no IP yet. Once the client has an IP, renewal is unicast directly to the server.

Relay (IP Helper)

If you have multiple subnets but only one DHCP server, routers need to forward DHCP broadcasts to the server. This is called DHCP Relay or "IP Helper". The router captures the broadcast and forwards it as a unicast to the DHCP server, which replies the same way.

Security Concerns

  • Rogue servers — As mentioned, malicious DHCP servers can redirect traffic
  • Exhaustion attacks — An attacker requests tons of IPs, exhausting the pool (mitigation: port security, rate limiting)
  • Lease sniffing — On unencrypted networks, anyone can see what IPs are assigned (minor issue — not much you can do with this info alone)

Bottom Line

DHCP is the reason you don't need a networking degree to use WiFi. It quietly handles IP assignment, lease management, and network configuration so you can just connect and go. The four-step DORA process is elegant in its simplicity, and most users never even know it exists — until it breaks.
