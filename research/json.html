How JSON Works

March 3, 2026

JSON won. Not because it was the best, but because it was simple enough.

The Problem

In the early 2000s, data serialization was a mess. XML was dominant but verbose. SOAP was bloated. Every platform had its own binary format. There was no common language for exchanging data between systems.

What JSON Solved

JSON (JavaScript Object Notation) emerged from JavaScript's object literal syntax. Douglas Crockford formalized it in 2002 and specs'd it in 2006. The timing was perfect: browsers were becoming platforms, AJAX was replacing page reloads, and APIs needed something lightweight.

JSON's rules are trivial:

  • Objects: {"key": "value"}
  • Arrays: ["a", "b", "c"]
  • Strings: double-quoted Unicode
  • Numbers: integers or floats
  • Booleans: true or false
  • Null: null

That's it. No attributes, no namespaces, no processing instructions, no DTDs. Just data.

Why It Won

JSON succeeded for boring reasons:

  • Ubiquity: Every language parses JSON. It's the lingua franca of APIs.
  • Readability: Humans can read it. Engineers can debug it.
  • Simplicity: The parser is trivial. There's almost nothing to spec.
  • JavaScript native: JSON.parse() was built into browsers. No library needed.

The "Good Enough" Principle

JSON isn't the most efficient format. Protocol Buffers are smaller. MessagePack is faster. CBOR handles more types. But JSON is "good enough" everywhere.

This is a recurring pattern in tech:

  • HTML beat SGML
  • USB beat FireWire (sometimes)
  • JPG beat PNG for photos
  • MP3 beat FLAC for portability

The winner is rarely the best technically. It's the one that's "good enough" and everywhere.

Limitations

JSON has real problems:

  • No comments: Minor but annoying.
  • No dates: Just strings. Every API invents its own format.
  • No types: Everything is loosely typed. Schemas are separate.
  • Verbosity: Repetitive keys bloat payloads.
  • No streaming: Must parse the entire document.

What Comes Next?

JSON is losing ground slowly:

  • MessagePack: Binary JSON, smaller payloads
  • Protocol Buffers: Google's schema-driven binary format
  • CBOR: IETF standard binary format
  • JSONC/JSON5: JSON with comments and relaxed syntax

But none have replaced JSON for general-purpose APIs. The network effect is too strong. Every tool speaks JSON. Every developer knows JSON. That inertia is hard to beat.

The Takeaway

JSON won because it was "good enough" at the right time. Not the best technically, but simple, universal, and built-in. Sometimes the boring solution wins.

The lesson: perfection is the enemy of adoption. Build something that works, make it ubiquitous, and you've won—even if something better exists in a lab somewhere.
