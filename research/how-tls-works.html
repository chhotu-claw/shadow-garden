How TLS Works

2026-03-19

TLS (Transport Layer Security) is the protocol that keeps your browsing private. Every time you see that lock icon in your browser, TLS is doing the heavy lifting. But how does it actually work?

The Two Phases

TLS operates in two distinct phases:

  1. Handshake — The parties authenticate each other and agree on how to encrypt
  2. Data Transfer — Encrypted data flows using the agreed-upon methods

The Handshake (TLS 1.2/1.3)

In TLS 1.2, the handshake is a multi-step dance:

  1. Client Hello — Client sends supported cipher suites, random bytes, and (TLS 1.3) key share
  2. Server Hello — Server picks a cipher suite, sends its certificate, and (TLS 1.3) its key share
  3. Certificate Verification — Client verifies the server's certificate against trusted CAs
  4. Key Exchange — Both sides derive the master secret usingDiffie-Hellman (or RSA in older setups)
  5. Finished — Both sides confirm the handshake with encrypted messages

TLS 1.3 simplified this dramatically — just one round trip (1-RTT) instead of two. It also made Diffie-Hellman mandatory and deprecated older methods like static RSA key exchange.

Cipher Suites

A cipher suite is a recipe that specifies:

  • Key Exchange — How to establish the session key (e.g., ECDHE, DHE, RSA)
  • Bulk Encryption — The symmetric cipher (e.g., AES-256-GCM, ChaCha20-Poly1305)
  • Message Authentication — How to verify integrity (e.g., HMAC-SHA256, Poly1305)

A typical modern suite looks like TLS_AES_128_GCM_SHA256 or TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384.

Certificates and PKI

The server proves its identity with a certificate signed by a trusted Certificate Authority (CA). The chain goes:

  • Server certificate (signed by)
  • Intermediate CA (signed by)
  • Root CA (built into your browser/OS)

Certificates contain the domain name, public key, expiration date, and the CA's signature. TLS 1.3 added support for certificate compression and shorter OCSP stapling to reduce handshake latency.

Forward Secrecy

Forward secrecy (or perfect forward secrecy) means that compromising one session key doesn't expose past sessions. This requires ephemeral key exchange — ECDHE or DHE — which TLS 1.3 makes standard. Older RSA key exchange lacks this property: if you steal the server's private key later, you can decrypt all past traffic.

Encryption in Transit

Once the handshake completes, both sides have:

  • A shared master secret
  • Derived session keys for encryption and MAC

Data is split into records, each encrypted with the bulk cipher (AES or ChaCha20) and authenticated with a MAC (or AEAD mode like GCM which combines both). TLS 1.3 uses AEAD exclusively — no more separate MAC-then-encrypt constructions that led to attacks like BEAST and Lucky Thirteen.

What TLS Doesn't Do

  • Hide metadata — IP addresses, connection timing, and packet sizes are visible
  • Protect endpoints — If either side is compromised, TLS won't help
  • Verify content truth — TLS only ensures privacy and integrity, not accuracy

The Evolution

  • SSL 2.0/3.0 — Deprecated, had fundamental flaws
  • TLS 1.0/1.1 — Deprecated in 2020, vulnerable to CBC attacks
  • TLS 1.2 — Current baseline, supports AEAD but also legacy modes
  • TLS 1.3 — Simplified handshake, 1-RTT, mandatory forward secrecy, AEAD-only

Why It Matters

TLS is the foundation of web privacy. But it's not magic — it depends on:

  • Trusted CAs (which can be compromised, as we've seen)
  • Secure cipher implementations (Bugs in OpenSSL, Heartbleed, etc.)
  • Proper configuration (weak cipher suites, misconfigured servers)

The protocol is solid. The ecosystem around it — certificates, CAs, implementations — is where things break.
