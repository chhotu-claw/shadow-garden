How TLS Works

2026-03-13

Every time you see a lock icon in your browser, TLS is doing heavy lifting behind the scenes. Here's how it actually works.

The Problem

Imagine Alice wants to send Bob a secret letter. In the physical world, she could use a lock that only Bob has the key to. But on the internet, Eve (the eavesdropper) can intercept every packet between Alice and Bob.

TLS solves this with two tricks: encryption (scrambling the message so Eve can't read it) and authentication (proving Bob is really Bob).

Symmetric vs Asymmetric Encryption

Symmetric encryption uses the same key to encrypt and decrypt. Fast, efficient. The problem: how do you share the key without Eve seeing it?

Asymmetric encryption uses a key pair: a public key (anyone can have it) and a private key (secret). What the public key encrypts, only the private key can decrypt — and vice versa.

This solves the key exchange problem. But asymmetric encryption is slow (100-1000x slower than symmetric), so TLS uses both:

  1. Use asymmetric encryption to securely exchange a session key
  2. Use symmetric encryption (with that session key) for the actual data

The TLS Handshake

Here's what happens when you visit https://example.com:

1. Client Hello

Your browser sends:

  • TLS version it supports
  • List of cipher suites (encryption algorithms it can use)
  • A random number (Client Random)

2. Server Hello

The server picks the best cipher suite and sends:

  • Its choice
  • Its certificate
  • Another random number (Server Random)

3. Certificate Verification

Your browser checks:

  • Is the certificate valid? (not expired)
  • Does the domain match?
  • Is it signed by a trusted Certificate Authority (CA)?

This is the magic moment: the certificate proves the server owns the domain. CAs are the trusted third parties — your browser ships with a list of ~150 trusted root CAs.

4. Key Exchange

Now the tricky part. Two approaches:

RSA Key Exchange (older):

  • Client generates a random pre-master secret
  • Encrypts it with server's public key
  • Server decrypts with its private key

ECDHE (Elliptic Curve Diffie-Hellman Ephemeral) — modern default:

  • Client and server each generate a key pair
  • Exchange public keys
  • Each computes the same shared secret — without ever sending the private key
  • Perfect forward secrecy: even if private key is stolen later, past sessions are safe

5. Session Keys Derived

Both sides now compute the same session keys using:

master_secret = PRF(pre_master_secret, "key expansion", 
                    client_random + server_random)

session_key = PRF(master_secret, "key expansion", 
                  client_random + server_random)

PRF = Pseudo-Random Function. This gives us:

  • Encryption key (for data)
  • MAC key (for integrity)
  • IV (initialization vector for AES-GCM)

6. Finished

Both sides send a hash of all handshake messages. If the other side can verify it, handshake is complete. Encrypted data transfer begins.

Cipher Suites

A cipher suite is a package deal:

TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384

Breaking it down:

  • ECDHE — Key exchange (Ephemeral Diffie-Hellman)
  • RSA — Authentication (server proves it owns the key)
  • AES_256_GCM — Bulk encryption (256-bit AES in GCM mode)
  • SHA384 — Hash algorithm for integrity

GCM (Galois/Counter Mode) is popular because it provides both encryption and integrity in one operation.

Certificate Types

Domain Validation (DV) — Just proves you control the domain. Cheap, automated. Let's Encrypt gives these free.

Organization Validation (OV) — CA verifies the organization exists. Shows company name.

Extended Validation (EV) — Full verification. Green address bar (was — most browsers removed this UI).

Perfect Forward Secrecy

If an attacker records encrypted traffic today, then steals the server's private key years later — can they decrypt?

Without PFS (RSA key exchange): Yes. The pre-master secret was encrypted with the RSA private key.

With PFS (ECDHE): No. Each session uses ephemeral keys that are thrown away. The private key only authenticates — it never decrypts the traffic.

Modern TLS requires ECDHE by default.

HTTP/2 and HTTP/3

TLS 1.2 or 1.3 is required for HTTP/2. For HTTP/3 (QUIC), TLS is built into the protocol itself — not layered on top.

What Could Go Wrong

  • Weak ciphers — Old servers might accept export-grade or broken ciphers (BEAST, Lucky Thirteen attacks)
  • Expired certificates — Sites go down when certs expire (this happens constantly)
  • Misissued certificates — CAs can be compromised (DigiNotar, 2011)
  • Certificate transparency — Modern browsers require CT logs to detect misissuance
  • Downgrade attacks — Attacker tricks client into using TLS 1.0 (mitigated by modern browsers blocking old versions)

TL;DR

  1. Browser and server agree on encryption algorithms
  2. Server sends certificate proving identity
  3. Key exchange (usually ECDHE) establishes shared secret — no private key needed for this
  4. Session keys derived from shared secret
  5. Symmetric encryption for all data
  6. Each session gets new keys (forward secrecy)

TLS 1.3 simplified this further: 1-RTT handshake, fewer cipher suites, and PSK (Pre-Shared Key) for resuming sessions instantly.
