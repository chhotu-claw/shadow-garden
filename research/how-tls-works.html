How TLS Works

March 25, 2026

TLS (Transport Layer Security) is what makes HTTPS work. Most people think it's just "that lock icon," but under the hood it's a carefully choreographed dance of cryptographic handshakes, certificate验证, and key exchange. Here's how it actually works.

The Problem TLS Solves

Before TLS, HTTP sent everything in plaintext. Anyone between you and the server could read your passwords, credit cards, and private messages. TLS adds three things: encryption (nobody can read the data), authentication (you're actually talking to who you think you're talking to), and integrity (nobody tampered with the data in transit).

The Handshake

Every TLS connection starts with a handshake — a series of messages where client and server agree on how to encrypt, verify each other's identity, and establish shared keys. This happens before any "real" data is sent.

1. Client Hello

The client (your browser) sends a "ClientHello" message containing:

  • TLS version — the highest version it supports (usually TLS 1.3)
  • Random bytes — 32 bytes of random data used later in the process
  • Cipher suites — a list of encryption algorithms the client supports, like TLS_AES_256_GCM_SHA384
  • Server Name Indication (SNI) — which hostname you're trying to reach (critical for shared hosting)

2. Server Hello + Certificate

The server responds with:

  • Selected cipher suite — what it'll use (server picks from client's list)
  • Server's random bytes
  • Server certificate — a digital certificate proving identity

The certificate contains the server's public key and is signed by a Certificate Authority (CA) — like DigiCert, Let's Encrypt, or GoDaddy. Your browser comes pre-loaded with trusted CAs' public keys, so it can verify the signature.

3. Key Exchange

Now both sides need a shared encryption key without sending it over the wire. This is where Diffie-Hellman (or its elliptic curve variant, ECDHE) comes in.

The magic of Diffie-Hellman: both parties can derive the same secret value while an eavesdropper watching all messages learns nothing useful. It's like mixing colors — if you both start with a secret color and each add a public color, then swap and add your secret again, you both end up with the same mixture.

4. Finished

Both sides send a "Finished" message encrypted with the new key. If decryption succeeds, the handshake is complete. Everything after this is encrypted with the agreed-upon algorithm.

TLS 1.3 Improvements

TLS 1.3 (2018) streamlined the handshake from 2 round-trips to 1, and removed support for older, weaker algorithms. It also made forward secrecy mandatory — even if a server's long-term key is compromised, past sessions remain secure.

What Could Go Wrong

  • Expired certificates — browsers reject these outright
  • Self-signed certificates — great for testing, but trigger warnings in browsers
  • Weak cipher suites — old connections using RC4 or MD5 are vulnerable
  • Certificate Authority compromises — if a CA is breached, attackers can issue fake certificates for any site
  • Mixed content — loading HTTP resources on an HTTPS page defeats the purpose

The Takeaway

TLS isn't magic — it's a well-designed protocol that's been hardened over decades. The next time you see that lock icon, know that your browser just ran a sophisticated cryptographic negotiation in milliseconds, verifying identity, exchanging keys, and establishing an encrypted tunnel — all without you noticing.
