How TLS Works

March 11, 2026

TLS (Transport Layer Security) is the protocol that makes HTTPS possible. It's the reason your browser shows a lock icon instead of "this connection is insecure." But TLS is more than just encryption — it's a complex dance of key exchange, authentication, and negotiated security. Here's how it actually works.

The Problem TLS Solves

Imagine Alice wants to send a secret message to Bob over the internet. The problem: every server between them can read the message. Encryption solves this, but there's a catch — how do they agree on the encryption key without someone eavesdropping?

This is the fundamental problem TLS solves: secure key exchange over an insecure channel.

The TLS Handshake

Before any data is sent, TLS performs a handshake. This is where the magic happens:

1. Client Hello

The client (your browser) sends a "Hello" message that includes:

  • TLS version (TLS 1.3 is current)
  • A random number (client random)
  • A list of cipher suites it supports
  • Extension requests (like SNI — Server Name Indication)

2. Server Hello

The server responds with:

  • Chosen cipher suite
  • Its random number (server random)
  • Its digital certificate

3. Certificate Verification

This is where authentication happens. The server sends its certificate, and the client verifies it:

  • Is the certificate signed by a trusted Certificate Authority (CA)?
  • Is the certificate not expired?
  • Does the certificate domain match the server?

The browser ships with a root store of trusted CAs. When you visit example.com, its certificate is signed by Let's Encrypt or DigiCert, whose certificates are signed by... more CAs, eventually reaching a root CA that your browser trusts.

4. Key Exchange

Now both parties need to derive the same encryption key. TLS 1.3 uses Diffie-Hellman (DH) key exchange:

  1. Client generates an ephemeral key pair
  2. Server generates an ephemeral key pair
  3. They exchange public keys
  4. Each computes the shared secret using their private key and the other's public key

The math: given g and p, it's easy to compute gab from ga and gb, but practically impossible to compute a from ga. This is the discrete logarithm problem.

Perfect Forward Secrecy (PFS): Because ephemeral keys are generated per-session, compromising one session's keys doesn't compromise past sessions. TLS 1.3 makes PFS mandatory.

5. Finished

Both parties now have a shared master key. They send encrypted "Finished" messages to verify everyone got the same keys. Handshake complete — encrypted data flows.

Cipher Suites

A cipher suite defines how encryption happens. Example: TLS_AES_256_GCM_SHA384

  • TLS — protocol
  • AES_256_GCM — symmetric encryption (256-bit key, Galois/Counter Mode)
  • SHA384 — hash function for authentication

TLS 1.3 simplified this drastically — only 5 cipher suites, all with authenticated encryption (AEAD), which means encryption and authentication happen together. No more separate MAC-then-Encrypt.

Certificate Types

Domain Validation (DV)

验证 you control the domain. Let's Encrypt issues these for free. Fast, automated, but only proves you own the domain — not who you are.

Organization Validation (OV)

The CA verifies the organization exists. Shows organization name in the certificate. Better for business sites.

Extended Validation (EV)

Highest verification. Shows green bar with organization name. Requires extensive documentation. Mostly used by banks and large companies. (Browsers are deprecating the green bar — EV effectiveness is debatable.)

Self-Signed Certificates

Signed by the same entity that uses it. Browsers don't trust these by default (unless you add them to the trust store). Useful for development or internal networks.

The Trust Chain

Your browser doesn't just trust one certificate. It trusts a chain:

example.com certificate
  → Signed by Let's Encrypt Authority X3
  → Signed by ISRG Root X1 (in browser trust store)

If any link in this chain is broken — expired CA cert, revoked intermediate, mismatched signature — the browser shows a security warning.

Common Attacks

Man-in-the-Middle (MITM)

An attacker intercepts communication, impersonating both sides. TLS prevents this because: (1) the server must prove it owns the private key, and (2) the key exchange is authenticated.

Certificate Pinning Bypass

Apps can "pin" a specific certificate or public key. If an attacker replaces the certificate, the app rejects it. But pinning can be bypassed if the app is compromised or the pinned key rotates poorly.

Downgrade Attacks

Attacker tricks parties into using older, weaker TLS versions. TLS 1.3 prevents this by requiring forward-secure key exchange in all cipher suites. TLS 1.0 and 1.1 are deprecated.

REX-3 / ROBOT

Timing attacks on RSA key exchange (in TLS 1.2 and earlier). If a server uses PKCS#1 v1.5 padding, an attacker can decrypt ciphertexts by measuring server response times. TLS 1.3 eliminates this by using DH key exchange by default.

TLS 1.3 is faster (1 round trip vs 2-3), more secure (PFS mandatory, removed insecure features), and simpler. There's no reason to use TLS 1.2 anymore except for compatibility with very old systems.

What's Actually Happening When You See the Lock

  1. Browser connects to example.com:443
  2. Server sends its certificate chain
  3. Browser verifies: not expired, trusted CA, domain matches
  4. Browser and server do DH key exchange (each generates temp keys)
  5. Both derive master key
  6. Encrypted tunnel established

All of this happens in ~100ms. The lock icon represents hundreds of milliseconds of cryptographic operations, dozens of trust decisions, and a global infrastructure of certificate authorities.

The Bigger Picture

TLS is invisible until it breaks. That's by design — security that announces itself is failing. But understanding what's under the hood reveals something: the web's security model is built on trust in third parties (CAs), and that trust has been repeatedly betrayed (DigiNotar, Symantec, etc.).

Certificate Transparency (CT) logs — public append-only logs of all certificates issued — are one response. But the fundamental model hasn't changed: we trust CAs to tell us who owns a domain. DNS-based Authentication (DANE) and OCSP stapling are attempts to reduce CA dependence.

The lock isn't a guarantee. It's a negotiation. And like all negotiations, the outcome depends on who's actually at the table.
