How TLS Actually Works

2026-03-16

TLS (Transport Layer Security) is the protocol that keeps the internet secure. Every time you see a lock icon in your browser, TLS is working. But how does it actually work? Not the marketing version — the real mechanics.

The Problem TLS Solves

Imagine Alice and Bob want to communicate over the internet. Mallary (an attacker) sits between them. Without TLS:

  • Alice sends "Hello Bob"
  • Mallary intercepts, reads it, forwards to Bob
  • Bob replies "Hi Alice"
  • Mallary reads that too

This is a man-in-the-middle attack. TLS prevents this by establishing an encrypted tunnel that Mallary cannot peer into.

TLS in One Sentence

TLS uses asymmetric cryptography to exchange keys, then symmetric encryption to protect data, with certificates proving identity.

The TLS Handshake (The Important Part)

Before any data moves, client and server do a dance called the handshake:

1. Client Hello

Browser sends: "Hey server, here are the TLS versions and cipher suites I support." Also sends a random number.

2. Server Hello

Server picks the best cipher suite, sends its certificate, and its own random number.

3. Certificate Verification

Browser verifies the certificate:

  • Is it not expired?
  • Does the domain match?
  • Is it signed by a trusted Certificate Authority (CA)?

Your OS ships with ~150 root CA certificates. If a cert is signed by one of those, it's trusted.

4. Key Exchange

This is where it gets interesting. The server's certificate contains a public key. The client uses this to encrypt a new random number (the pre-master secret). Only the server can decrypt it (with its private key).

Both sides now derive the same session key using a key derivation function (KDF).

5. Finished

Both sides confirm handshake is done. Now they encrypt everything with the session key.

Symmetric vs Asymmetric Encryption

TLS uses asymmetric crypto only for the handshake. After that, everything is encrypted with symmetric encryption (usually AES). This is why TLS is fast after connection establishment.

Cipher Suites

A cipher suite defines the combo of algorithms used:

TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
  • ECDHE: Key exchange (Elliptic Curve Diffie-Hellman Ephemeral)
  • RSA: Certificate type / signing
  • AES_256_GCM: Bulk encryption (256-bit AES in GCM mode)
  • SHA384: Hash algorithm for authentication

Why GCM Matters

GCM is an AEAD (Authenticated Encryption with Associated Data) mode. It provides:

  • Confidentiality: Can't read data without key
  • Integrity: Can't modify data undetected
  • Authentication: Proves who sent the data

Older modes like CBC required separate MAC (HMAC) which was often implemented incorrectly (POODLE, BEAST attacks).

Certificate Authorities (CAs)

Here's where TLS gets flaky. You trust a site because a CA vouched for it. But:

  • 2011: DigiNotar hacked, fake Google certs issued
  • 2015: CNNIC added to blocklists after issuing unauthorized certs
  • 2020: 20+ intermediate CA certificates revoked

The CA system is a trust on first use model — you trust your OS/browser's pre-installed Certs. It's the best we have, but it's not bulletproof.

What TLS Doesn't Do

  • Doesn't hide metadata: Who you connected to, when, how much data — all visible (see: traffic analysis)
  • Doesn't hide DNS: DNS queries still leak (use DNS over HTTPS or Tor)
  • Doesn't hide IP address: The server's IP is always visible

TLS 1.3: The Modern Version

TLS 1.3 (2018) simplified things:

  • 1-RTT handshake: Down from 2-RTT (or 3 in older TLS)
  • Dropped deprecated ciphers: No more RC4, MD5, SHA-1
  • Forward secrecy mandatory: Every session uses new keys (even if server private key leaks)
  • 0-RTT mode: Reconnect instantly (but with replay attack risks)

The Takeaways

  1. TLS uses asymmetric crypto only to exchange keys — after that, it's fast symmetric encryption
  2. Certificates prove identity — but the CA system has a spotty history
  3. GCM is the gold standard — it bundles encryption and authentication
  4. TLS 1.3 is much simpler and safer — enable it everywhere
  5. TLS isn't magic privacy — metadata, DNS, and IPs still leak

Next time you see that lock icon, you know what's actually happening underneath.
