How TLS Works

March 13, 2026

TLS (Transport Layer Security) is the protocol that keeps the internet secure. Every time you see 🔒 in your browser, TLS is doing the heavy lifting. Here's how it actually works.

The Problem TLS Solves

Imagine Alice wants to send a secret letter to Bob through an unsecured postal system. Anyone who handles the letter could read it, or even change its contents. TLS solves this with three properties:

  • Encryption — Nobody can read the data
  • Authentication — You know who you're talking to
  • Integrity — Data hasn't been tampered with

TLS Handshake: The Dance

Before any data flows, TLS performs a "handshake" to establish a secure connection. This happens in milliseconds.

1. Client Hello

The client (your browser) sends:

  • TLS version it supports
  • List of cipher suites (encryption algorithms it can use)
  • A random number

2. Server Hello

The server responds with:

  • Chosen cipher suite
  • Its certificate (contains public key)
  • Another random number

3. Certificate Verification

The client verifies the server's certificate against trusted Certificate Authorities (CAs). If valid, it trusts the server. This is why self-signed certs show warnings — they're not signed by a trusted CA.

4. Key Exchange

Both parties derive the same "master secret" using:

  • The two random numbers
  • Either RSA key exchange or Diffie-Hellman (DH/ECDH)

RSA: Client generates random master secret, encrypts with server's public key, sends to server.

Diffie-Hellman: Both exchange parameters and compute the master secret together. Even if intercepted, the secret can't be derived. (This provides "forward secrecy" — if a key is compromised later, past sessions stay secure.)

5. Handshake Complete

Both sides now have a shared master secret. They derive session keys for symmetric encryption (much faster than asymmetric). All subsequent data uses these keys.

Certificates: The ID Card

TLS certificates are digital identity documents. They bind a domain name to a public key, signed by a trusted CA.

Certificate structure:
├── Domain name (subject)
├── Public key
├── Issuer (CA who signed it)
├── Validity period
└── CA's digital signature

Certificate Chain

Your certificate is signed by an Intermediate CA, which is signed by a Root CA (trusted by your OS/browser). The chain looks like:

Root CA (trusted by browser)
  └── Intermediate CA
    └── your-domain.com

Types of Validation

  • DV (Domain Validation) — CA verifies you control the domain. Fast, cheap.
  • OV (Organization Validation) — CA verifies the organization exists. Shows company name.
  • EV (Extended Validation) — Thorough verification. Green bar in browsers. Rare now.

Cipher Suites

A cipher suite defines the algorithms used. Format: TLS_KE_KEYEXCHANGE_AUTH_WITH_CIPHER_HASH

Example: TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384

ECDHE  — Key exchange (Elliptic Curve Diffie-Hellman Ephemeral)
RSA    — Authentication (server's key type)
AES_256_GCM — Bulk encryption (256-bit, Galois/Counter mode)
SHA384 — Hash algorithm for HMAC

Important: "Ephemeral" (E) in ECDHE means forward secrecy — new keys per session.

What Actually Gets Encrypted?

+------------------+-------------------+
|    TLS 1.2       |     TLS 1.3       |
+------------------+-------------------+
| Handshake: plain | Handshake: encrypted |
| Application: encrypted | Application: encrypted |
+------------------+-------------------+

TLS 1.3 encrypts the handshake itself (except Client/Server Hello). Less metadata leaked.

Common Vulnerabilities

POODLE (2014)

Attacked SSLv3's CBC mode. Fix: Disable SSLv3.

Heartbleed (2014)

Bug in OpenSSL's heartbeat extension leaked memory. Fix: Update OpenSSL.

BEAST (2011)

Attacked TLS 1.0's CBC. Fix: Use TLS 1.2+.

Renegotiation Attack

Attacker injects data into legitimate connection. Fix: Secure renegotiation (TLS 1.1+).

TLS 1.3 Improvements

  • Faster handshake — 1 round trip vs 2 in TLS 1.2
  • Encrypted everything — Even handshake messages
  • Removed weak ciphers — No MD5, SHA-1, RC4, 3DES
  • Mandatory forward secrecy — DH/ECDHE only
  • 0-RTT mode — Resumes sessions instantly (but with replay risk)

How to Check

# Test TLS support
openssl s_client -connect example.com:443 -tls1_2
openssl s_client -connect example.com:443 -tls1_3

# Check certificate
openssl x509 -in cert.pem -text -noout

# Check supported ciphers
nmap --script ssl-enum-ciphers -p 443 example.com

The Takeaway

TLS isn't magic — it's carefully designed cryptography that runs every time you visit an HTTPS site. The handshake establishes trust, certificates prove identity, and symmetric encryption protects your data. Modern TLS (1.3) is fast and secure, but legacy support for older versions still creates vulnerabilities.

The next time you see that 🔒, know that hundreds of milliseconds of cryptographic negotiation just happened to keep you safe.
