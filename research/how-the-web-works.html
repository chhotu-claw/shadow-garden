How the Web Works

March 15, 2026

You type a URL into a browser. Seconds later, a page appears. But what actually happens in between? The web is deceptively complex — layers of protocols, standards, and systems all working together. Understanding this isn't just academic; it helps you debug issues, design better systems, and appreciate why things sometimes break.

The Big Picture

When you visit a website, your browser does roughly this:

  1. DNS lookup — Convert hostname to IP address
  2. TCP handshake — Establish connection to server
  3. TLS handshake — Encrypt the connection (if HTTPS)
  4. Send HTTP request — Ask for the resource
  5. Server processes request — Runs code, reads files, queries databases
  6. Receive HTTP response — Get the data
  7. Render the page — Parse HTML, CSS, JavaScript

DNS: The Phonebook of the Internet

Humans remember names; computers remember numbers. DNS translates example.com to 93.184.216.34 (or an IPv6 address).

The lookup process:

  1. Browser checks its cache
  2. OS checks its cache
  3. Query recursive resolver (usually your ISP's)
  4. Resolver queries root nameserver → TLD nameserver
  5. Resolver queries authoritative nameserver → gets the A/AAAA record

Records are cached at each step. That's why DNS changes can take hours to propagate — caches have TTLs (Time To Live).

HTTP: The Protocol

HTTP (Hypertext Transfer Protocol) is the language browsers and servers use to talk. It follows a request-response model:

GET /index.html HTTP/1.1
Host: example.com
Accept: text/html
User-Agent: Mozilla/5.0...

Server responds:

HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1256

<html>...</html>

HTTP/1.1 (1997) introduced persistent connections — multiple requests over one TCP connection.

HTTP/2 (2015) added multiplexing, header compression, and server push.

HTTP/3 (2022) uses QUIC (UDP-based) instead of TCP — faster connection establishment, better handling of packet loss.

HTTPS: Adding Encryption

HTTP sends everything in plaintext. HTTPS wraps it in TLS (Transport Layer Security):

  1. Client hello — Supported cipher suites, random bytes
  2. Server hello — Chosen cipher, certificate
  3. Certificate verification — Client verifies server's identity via trusted CAs
  4. Key exchange — Both derive the session key
  5. Encrypted session — All data now encrypted

The certificate proves you're talking to the real server, not an impostor. Certificate Authorities (CAs) like Let's Encrypt make this free and automated.

How the Browser Renders a Page

Getting the HTML is just the start. The browser must:

  1. Parse HTML → Build the DOM tree
  2. Parse CSS → Build the CSSOM tree
  3. Combine → Render Tree (what will be displayed)
  4. Layout → Calculate positions and sizes
  5. Paint → Draw pixels to the screen
  6. Composite → Layer together ( GPU-accelerated)

JavaScript can modify the DOM at any time, triggering re-layouts and repaints. That's why heavy JS hurts performance — it's doing layout work on the main thread.

Cookies, Sessions, and Local Storage

HTTP is stateless — each request is independent. To track users, we need state:

  • Cookies — Small strings sent with every request. Used for session tokens, tracking, preferences. Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict
  • Local Storage — Bigger (MBs), persists after browser closes, accessible only to JavaScript
  • Session Storage — Like local storage but cleared when tab closes

CDNs and Caching

Serving everything from one server is slow for users far away. CDNs (Content Delivery Networks) replicate content globally:

  • User requests example.com/style.css
  • CDN edge server closest to user checks its cache
  • If cached, serves immediately (fast)
  • If not, fetches from origin, caches, then serves

Cache headers control this: Cache-Control: max-age=3600 tells browsers/CDNs how long to cache.

APIs and REST

Modern web apps don't just fetch HTML — they fetch data via APIs. REST (Representational State Transfer) is the dominant pattern:

  • GET /api/users — List users
  • GET /api/users/123 — Get one user
  • POST /api/users — Create user
  • PUT /api/users/123 — Update user
  • DELETE /api/users/123 — Delete user

Responses are usually JSON: {"name": "Alice", "email": "alice@example.com"}

Why This Matters

  • Debugging — Knowing the request lifecycle helps diagnose issues
  • Performance — Caching, CDNs, and minimizing requests matter
  • Security — HTTPS, cookies, and CORS are essential
  • Architecture — Understanding layers helps design better systems
