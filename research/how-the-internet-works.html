How the Internet Works

2026-03-26

You type google.com. Three seconds later, Google's homepage appears. That magic hides a lot of machinery. Here's how it actually works.

DNS: The Phonebook of the Internet

Computers don't understand "google.com." They understand IP addresses like 142.250.190.46. DNS is what translates between them.

When you type a URL, your browser first checks its cache. If not there, it asks the operating system. The OS checks its cache, then asks a recursive DNS resolver (usually your ISP's). That resolver walks the DNS hierarchy — root server, TLD server (.com), authoritative nameserver — until it gets an answer. All in milliseconds.

DNS queries are typically UDP port 53. The protocol is simple: "What's the IP for google.com?" → "Here's the IP." But there's a dark side: DNS is unencrypted. Your ISP sees every site you visit. That's why DNS over HTTPS (DoH) exists.

Packets: Breaking Things Up

Data doesn't travel as one big chunk. It gets sliced into packets — typically around 1,500 bytes each. Each packet has:

  • Header: Source IP, destination IP, protocol, sequence number
  • Payload: The actual data

Why break it up? Because networks are shared. If you sent a 10MB file as one chunk, everyone else would wait. Packets let different flows share bandwidth dynamically.

This is called "packet switching" — the foundation of the internet. No dedicated circuit, just efficient sharing.

Routing: Finding the Path

Every packet has a destination IP. Routers along the way look at that IP and decide: "Where should I send this next?"

They use routing tables — essentially maps built from routing protocols like BGP (Border Gateway Protocol). BGP is the glue that holds the internet together. It lets autonomous systems (ISPs, cloud providers, big companies) announce which IP ranges they control.

When you send a packet to Google, it passes through maybe 10-20 routers. Each router makes a local decision. The path isn't fixed — it depends on current congestion, failures, and policy.

This is why "traceroute" shows different paths at different times.

TCP: The Reliable Conveyor

Packets can arrive out of order, get duplicated, or disappear. TCP fixes this:

  • Sequence numbers: Reassemble packets in the right order
  • Acknowledgments: Sender waits for ACKs before sending more
  • Retransmission: Lost packets get resent
  • Flow control: Don't overwhelm the receiver
  • Congestion control: Don't overwhelm the network

TCP is why downloads are reliable. It adds overhead (latency) but guarantees delivery.

SSL/TLS: The Handshake

Once TCP connects, if you're using HTTPS, TLS kicks in. The client and server:

  1. Agree on a cipher suite (e.g., TLS 1.3 with AES-256-GCM)
  2. Server sends its certificate
  3. Client verifies the certificate against trusted CAs
  4. They do a key exchange (Diffie-Hellman or RSA) to establish session keys
  5. Everything after is encrypted

TLS 1.3 reduced the handshake from 2 round trips to 1. That's why HTTPS is fast now.

The Full Trip

So when you visit a site:

  1. DNS resolves the hostname → IP
  2. TCP SYN → SYN-ACK → ACK (three-way handshake)
  3. TLS handshake (if HTTPS)
  4. HTTP request: GET / HTTP/1.1
  5. Server responds with HTML, CSS, JS
  6. Browser parses, renders, fetches more assets

This happens dozens of times per page load. The average page makes 80+ requests.

The miracle isn't that it works. It's that it works so well, so fast, with so little visible infrastructure. The internet is a collaboration between thousands of organizations, billions of devices, and countless protocols — all agreeing to talk to each other.
