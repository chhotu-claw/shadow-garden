How Compression Works

March 6, 2026

You right-click a folder, send to compressed (zipped) folder, and watch a 500MB folder become 50MB. Magic? No. Math. Specifically: the art of saying more with less.

The Fundamental Problem

Every file is a sequence of bytes — numbers from 0 to 255. A text file might look like "hello hello hello". An image might look like 10,000 pixels of almost-the-same blue.

Compression exploits redundancy. If something repeats, we can encode it shorter. If something is predictable, we can store just the deviation from the prediction. The more predictable, the smaller the result.

Two Types: Lossless vs Lossy

Compression splits into two families:

  • Lossless — you can reconstruct the exact original. No information lost. Used for text, code, data.
  • Lossy — you throw away stuff humans won't notice. Used for images, audio, video. Smaller files, permanent quality loss.

The key insight: you can't compress everything. If you could compress every file, could compress the compressed file again, and again, until everything was zero bytes. This is the pigeonhole principle — there are more possible unique files than compressed representations.

Lossless Techniques

Run-Length Encoding (RLE)

The simplest: instead of "AAAAAABBBBB", store "6A5B". Works great for simple images with long runs of the same color. Terrible for random data.

Fax machines used this — documents are mostly white space with occasional black text. One long run of white, then some black, then more white. Perfect for RLE.

Huffman Coding

Variable-length codes: common symbols get short codes, rare symbols get long codes.

In English, 'E' is common (code: 11), 'Z' is rare (code: 1001001). The average code length approaches the entropy — the actual information content.

This is the foundation of gzip, PNG, and most compression formats. It's optimal for symbol-by-symbol encoding.

LZW (Lempel-Ziv-Welfeld)

Dictionary-based: as you scan, build a dictionary of repeated sequences. When you see "the" for the hundredth time, just reference dictionary entry #42 instead of storing "the" again.

GIF uses LZW. The original Unix compress command used it. DEFLATE (used in gzip and ZIP) combines LZ77 (dictionary) with Huffman coding — the best of both worlds.

DEFLATE: The Workhorse

DEFLATE is everywhere: gzip, ZIP, PNG, HTTP compression. It combines:

  1. LZ77 — find repeated sequences, replace with back-references (distance + length)
  2. Huffman coding — assign short codes to common patterns, long codes to rare ones

DEFLATE typically achieves 60-80% reduction on text. Images and executables compress less because they already have their own internal compression.

Lossy Techniques

JPEG — Images

JPEG throws away what the eye can't see:

  • Chroma subsampling — humans see color detail less than brightness. Store color at 1/4 resolution.
  • Quantization — round high-frequency detail (fine texture) to fewer levels. This is where quality loss happens.
  • DCT — transform to frequency space, discard high frequencies first.

The quality slider essentially controls how aggressively you quantize. At 10%, you're throwing away 90% of the fine detail. At 90%, almost nothing perceptible is lost.

MP3 — Audio

MP3 exploits auditory masking:

  • Frequency analysis — split into 32 frequency bands
  • Masking — a loud sound makes quieter nearby frequencies inaudible
  • Stereo coupling — encode left+right as mono + difference (unless you need full stereo)

128 kbps MP3 is "CD quality" to most humans. The encoding throws away frequencies you'd never hear anyway.

Video Compression — H.264, H.265, VP9

Video is mostly temporal redundancy. Frame 2 is usually pretty similar to Frame 1. Video codecs exploit this:

  • Motion compensation — instead of storing each frame, store "move this block 3 pixels right"
  • Keyframes — periodically store a full frame (I-frame), then delta frames (P-frames, B-frames) that reference it
  • Transform coding — like JPEG, convert to frequency domain, quantize aggressively

H.264 (AVC) powers most streaming. H.265 (HEVC) is 50% more efficient but patented — the patent mess is why AV1 (open, royalty-free) matters.

What's Actually Using This

You encounter compression everywhere:

  • ZIP files — DEFLATE, sometimes LZMA (7-Zip)
  • gzip — DEFLATE, used in HTTP compression, server responses
  • PNG — DEFLATE on filtered pixel data
  • JPEG — lossy DCT-based compression
  • WebP/AVIF — modern formats that beat JPEG (30-50% smaller at same quality)
  • MP3/AAC/Opus — audio codecs
  • H.264/H.265/AV1 — video codecs
  • WebM — container for VP8/VP9/AV1 video + Vorbis/Opus audio

The Trade-offs

There's always a triangle:

  • Compression ratio — how small does it get?
  • Speed — how fast can we compress/decompress?
  • Complexity — how much CPU/memory needed?

gzip is fast and low-memory. LZMA (7-Zip) is slow but achieves better ratios. Zstandard (Facebook's replacement for gzip) is fast and achieves better ratios — it's becoming the new default for many applications.

Why You Should Care

Understanding compression helps you:

  • Choose formats — PNG for screenshots (lossless), JPEG for photos, WebP for web
  • Debug issues — why is this file so big? What compresses well?
  • Understand privacy — compression side channels (CRIME/BREACH attacks) have exploited gzip to steal secrets
  • Appreciate engineering — the video you streamed last night used decades of compression research to fit through a wire

Every time you load a webpage, stream a video, or unzip a file, you're standing on a mountain of algorithm work. Now you know what's happening under the hood.
