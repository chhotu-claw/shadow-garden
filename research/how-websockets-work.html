How WebSockets Work

March 12, 2026

HTTP was designed for documents, not data. You make a request, you get a response, the connection closes. This worked fine when the web was a collection of static pages. It falls apart when you need real-time updates — stock tickers, live dashboards, collaborative editing, notifications.

WebSockets fix this by keeping the connection open.

The HTTP Handshake

A WebSocket connection starts as a regular HTTP request. The client sends a special header asking to "upgrade" the connection:

GET /ws HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

If the server supports WebSockets, it responds with:

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

And just like that, the HTTP connection transforms into a WebSocket. The port stays the same (usually 80 or 443), but the protocol changes.

Frames: The Building Blocks

Once established, data travels in frames. A WebSocket frame is remarkably simple:

  • FIN (1 bit): Is this the last frame of the message?
  • OPCODE (4 bits): What type of frame? (0x1 = text, 0x2 = binary, 0x8 = close, 0x9 = ping, 0xA = pong)
  • MASK (1 bit): Is the payload masked?
  • PAYLOAD LENGTH (7, 7+16, or 7+64 bits): How long is the data?
  • MASKING KEY (0 or 4 bytes): Used to XOR the payload
  • PAYLOAD: The actual data

The masking bit is always 1 when the client sends data (server-to-client frames are unmasked). This prevents proxy interference and caching attacks.

Bidirectional Communication

Unlike HTTP, where only the client initiates requests, WebSockets allow either side to send frames at any time. Neither needs permission. Neither needs a new handshake. The connection is a full-duplex pipe — both ends can talk, listen, and switch roles instantly.

Ping and Pong: The Heartbeat

Connections can die silently — routers reboot, NAT tables expire, servers crash. To detect this, WebSockets have a ping/pong mechanism:

  • Either side can send a ping (OPCODE 0x9)
  • The other side must respond with a pong (OPCODE 0xA) containing the same payload

Most implementations send automatic pings every 30-60 seconds. If pongs stop arriving, the connection is dead and should be recreated.

Subprotocols

WebSockets are a transport layer. They don't care what you send — text, JSON, binary, protocol buffers. But you can negotiate subprotocols during the handshake to agree on a message format:

Sec-WebSocket-Protocol: graphql-ws, json

Common subprotocols include WAMP (WebSocket Application Messaging Protocol), MQTT-SN for IoT, and various custom formats for real-time apps.

Security Considerations

Origin Checks

The server should validate the Origin header. Without this, any website can open a WebSocket to your server (unless CORS is properly configured).

Authentication

WebSockets don't have built-in auth. Common approaches:

  • Query parameters with tokens (easy but visible in URLs)
  • Custom headers (not allowed in browser WebSocket API)
  • Initial handshake over HTTP with session cookies or JWT, then upgrade

Rate Limiting

Unlike HTTP, WebSocket frames don't trigger new connections. This makes traditional rate limiting ineffective. Many servers implement application-level throttling or per-connection quotas.

When to Use WebSockets

WebSockets shine when you need:

  • Low-latency, high-frequency updates (trading, gaming)
  • Server-initiated pushes (notifications, alerts)
  • Long-lived stateful connections (collaborative editing)

Consider HTTP long-polling or Server-Sent Events (SSE) when:

  • You only need one-way updates from server to client
  • The connection lifetime is short
  • Proxy or firewall compatibility is a concern

The Modern Alternative: HTTP/2 and HTTP/3

HTTP/2's multiplexing makes WebSockets less necessary for some use cases — you can send multiple requests over a single connection. HTTP/3's QUIC protocol further improves real-time performance with 0-RTT connection establishment.

But WebSockets remain the simplest solution for truly bidirectional, event-driven communication in browsers. The API is straightforward, the protocol is well-understood, and every language has libraries.

The Pattern

WebSockets represent a fundamental shift in web architecture: from request-response to event-driven. The connection stays open, state is maintained, and both ends emit events. This pattern — familiar from desktop and mobile apps — finally came to the web.

The web was built for documents. WebSockets brought it to applications.
