How WebSockets Work

2026-03-13 · DEEP DIVE

The Problem with HTTP

HTTP is request-response. Client asks, server answers, connection closes. This works fine for loading pages — you ask for HTML, you get HTML, done. But what happens when the server has new data to push? Polling. Constantly asking "any new data yet? any new data yet?"

That's waste. WebSockets fix this.

The Upgrade

A WebSocket connection starts as a normal HTTP request:

GET /ws HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

Server responds:

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

That's it. Protocol switch. The TCP connection stays open. Now both sides can send data anytime.

The Frames

WebSocket data travels in frames. Not HTTP messages — frames. Each frame has:

  • FIN (1 bit) — is this the last fragment?
  • Opcode (4 bits) — what kind of message? 0x1 = text, 0x2 = binary, 0x8 = close, 0x9 = ping, 0xA = pong
  • Mask (1 bit) — is the payload masked?
  • Payload length (7, 7+16, or 7+64 bits)
  • Masking key (0 or 4 bytes) — used to obfuscate data
  • Payload — the actual data

Why mask? Originally required for security when proxied. Payload is XOR'd with a random key per frame. Receiver XOR's again to get original data.

Bidirectional Magic

Unlike HTTP where client always initiates, WebSocket messages can go either direction. Server can push. Client can push. No asking permission.

Both sides can send:

  • TEXT frames — UTF-8 text
  • BINARY frames — raw bytes
  • CLOSE frames — graceful shutdown
  • PING/PONG frames — keepalive and latency check

Ping-Pong: The Heartbeat

WebSockets have no built-in timeout, but middleboxes (load balancers, firewalls) may close idle connections. Solution: PING/PONG.

Send a PING (opcode 0x9), receive a PONG (opcode 0xA). This keeps the connection alive and detects if the other side is dead.

Subprotocols

WebSocket is a transport. The application protocol on top is up to you. Common ones:

  • SOAP — XML over WebSocket
  • MQTT — lightweight messaging
  • WAMP — RPC + pub/sub
  • GraphQL subscriptions — often over WebSocket

You negotiate during the handshake:

Sec-WebSocket-Protocol: graphql-ws, mqtt

Security

Origin Checking

Server should validate the Origin header. Don't accept connections from untrusted origins.

WSS (WebSocket Secure)

Always use TLS:

wss://example.com/socket

Not ws://. Unencrypted WebSockets send everything in plaintext including the masking key.

Input Validation

Just because it's WebSocket doesn't mean you skip validation. Validate every message. Don't trust the client.

Rate Limiting

Without rate limits, a compromised client can flood the server. Implement throttling per connection.

When to Use WebSockets

Good for:

  • Real-time chat
  • Live dashboards (stock tickers, metrics)
  • Multiplayer games
  • Collaborative editing
  • Notifications

Bad for:

  • Simple request-response (just use HTTP)
  • High-frequency trading (use binary protocols, co-location)
  • When you need caching (WebSockets don't cache well)

HTTP/2 and HTTP/3

WebSockets predate HTTP/2. With HTTP/2's streams, you can achieve bidirectional communication too. HTTP/3 with QUIC makes it even better. But WebSockets still win for true server push without request overhead.

The Handshake Math

The Sec-WebSocket-Key is base64 of a 16-byte random value. The Sec-WebSocket-Accept is:

base64(SHA1(sec-websocket-key + "258EAFA5-E914-47DA-95CA-C5AB0DC85B11"))

That magic string is the WebSocket protocol GUID. Client proves it understands the protocol by generating the correct response.

Closing Connections

To close gracefully, send a CLOSE frame (opcode 0x8) with:

  • Close code (1000 = normal, 1001 = going away, etc.)
  • Optional reason string

Always close properly. Don't just kill the TCP connection — send the close frame and wait for response.

Summary

  • WebSockets start as HTTP upgrade (101 Switching Protocols)
  • Bidirectional, frame-based communication
  • Text and binary frames, plus PING/PONG for keepalive
  • Use WSS, validate input, rate limit
  • Great for real-time, bad for simple request-response
