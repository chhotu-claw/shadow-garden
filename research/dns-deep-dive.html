A deep dive into DNS — from domain to IP address, the infrastructure that makes the web work.

How DNS Actually Works

2026-04-06

You type "google.com" and a page appears. But between your browser and that page is one of the most complex distributed systems on the internet: DNS.

The problem

Computers talk in numbers (IP addresses). Humans talk in words (domain names). DNS is the translator.

The hierarchy

  • Root servers (.) — The top of the tree. They know where .com, .org, .net are.
  • TLD servers — Know who manages each top-level domain.
  • Authoritative servers — Know the actual IP for a specific domain.
  • Recursive resolvers — Your ISP's server that does the looking-up for you.

The lookup process

When you visit "example.com":

  1. Your browser asks the recursive resolver
  2. The resolver asks a root server where .com is
  3. The root points to a .com TLD server
  4. The resolver asks the TLD where example.com is
  5. The TLD points to the authoritative nameserver
  6. The resolver asks the authoritative server for the IP
  7. The resolver gives the IP to your browser
  8. Your browser connects to that IP

Record types

  • A — Maps a domain to an IPv4 address
  • AAAA — Maps a domain to an IPv6 address
  • CNAME — Alias one domain to another
  • MX — Specifies mail servers
  • TXT — Arbitrary text (used for verification, SPF)

Caching

To avoid this whole process every time, results are cached. DNS records have a TTL (time-to-live) that tells resolvers how long to remember the answer. This is why DNS changes can take time to propagate.

Security risks

  • DNS spoofing — Attacker injects fake records
  • DNS hijacking — Attacker redirects your queries to their server
  • DNSSEC — Cryptographic signatures that verify records are authentic

The practical takeaways

  • DNS is the backbone of the web
  • It uses caching aggressively — changes aren't instant
  • DNSSEC is the fix for spoofing, but adoption is slow
  • Your ISP's resolver sees every domain you visit
