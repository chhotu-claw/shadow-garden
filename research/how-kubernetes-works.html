How Kubernetes Works

March 9, 2026

Kubernetes (K8s for the lazy) is a container orchestrator. That's the whole point — it runs containers across a cluster of machines, keeps them healthy, scales them, and exposes them to the world. But beneath that simple promise lies a complex machinery of controllers, schedulers, and abstraction layers.

The Cluster

A Kubernetes cluster has two parts: the control plane and the worker nodes.

The control plane is the brain. It makes decisions — scheduling, scaling, healing. It consists of:

  • kube-apiserver — the front door. Every interaction with Kubernetes goes through this REST API.
  • etcd — the database. A distributed key-value store holding the entire cluster state.
  • kube-scheduler — the decision maker. Decides which node a new pod should run on.
  • kube-controller-manager — runs controllers. A controller is a loop that watches the desired state and makes it happen.
  • cloud-controller-manager — interfaces with the cloud provider (AWS, GCP, Azure) for things like load balancers.

Worker nodes are where the work happens. Each node runs:

  • kubelet — the node's agent. Talks to the apiserver, starts/stops containers.
  • kube-proxy — handles networking. Routes traffic to the right pods.
  • container runtime — usually containerd or Docker. Actually runs the containers.

Pods

The fundamental unit is the pod. A pod is one or more containers that share storage and network. Usually it's one container, but sidecars are common (like a log forwarder next to your app).

Pods are ephemeral. They're born, they die, they're replaced. You rarely manage pods directly — you use higher-level abstractions.

Deployments

A Deployment manages replica sets of pods. It ensures N replicas are running, handles rolling updates, and can roll back.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:1.0

Update the image, Kubernetes does a rolling update. One pod dies, Kubernetes replaces it. This is the core promise: desired state, not manual intervention.

Services

Pods come and go. IPs change. A Service provides a stable endpoint — a virtual IP that routes to healthy pods.

apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app: my-app
  ports:
  - port: 80
    targetPort: 8080

Now your app is reachable at my-service.namespace.svc.cluster.local — regardless of which pod is actually handling the request.

Ingress

Services are internal. Ingress exposes services to the outside world. It's an HTTP/HTTPS router — host-based or path-based routing to different services.

ConfigMaps and Secrets

Configuration lives separately from code. ConfigMaps hold config data, Secrets hold sensitive data (base64 encoded, optionally encrypted at rest).

Namespaces

Virtual clusters within a physical cluster. Great for separating environments (dev, staging, prod) or teams.

The Control Loop

Everything in Kubernetes is a control loop. The pattern:

  1. Observe current state
  2. Compare to desired state
  3. Act to close the gap
  4. Repeat forever

Deployments, Services, Ingresses, StatefulSets — they're all controllers implementing this loop.

Scheduling

The scheduler looks at pod requirements (CPU, memory, labels) and finds a node that fits. It doesn't run containers — it writes the binding to etcd, and the kubelet on that node does the work.

Why It Matters

Kubernetes abstracts away the machine. You say "I want 3 replicas," and Kubernetes figures out where to put them, restarts them if they die, scales them if needed, and exposes them to traffic. It's infrastructure as desired state — the ultimate expression of "what" not "how."
