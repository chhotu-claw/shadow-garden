How Kubernetes Works

March 11, 2026

Kubernetes (K8s) is Google's open-source answer to the question: "How do we run thousands of containers without losing our minds?" The answer: abstractions, lots of them, stacked on top of each other until something magical happens.

The Problem

Docker solves packaging. One container = one artifact. But what happens when you need fifty containers running across ten machines? What happens when one dies? What happens when traffic spikes at 2 AM?

That's where Kubernetes comes in. It's an orchestrator — a meta-infrastructure that manages your infrastructure. It decides where things run, when they restart, how they talk to each other, and how to scale them. You say "I want 3 copies of this," and Kubernetes makes it so.

The Architecture

Kubernetes runs as a cluster. A cluster has one control plane (the brains) and one or more worker nodes (the muscle).

The Control Plane

The control plane is a collection of processes that govern the entire cluster:

  • kube-apiserver — The front door. Every interaction with Kubernetes goes through this REST API. You talk to it, and it validates and processes your requests.
  • etcd — The database. A distributed key-value store that holds the entire cluster state. Every pod, service, config — all of it lives here. If etcd dies, your cluster forgets everything.
  • kube-controller-manager — The watcher. Runs controllers that constantly compare "desired state" (what you want) to "current state" (what you have). If they differ, it takes action.
  • kube-scheduler — The decision maker. When you create a new pod, this decides which node it runs on. Considers resources, constraints, affinity rules.

Worker Nodes

Each worker node runs:

  • kubelet — The node's agent. Talks to the apiserver, ensures containers are running as specified.
  • kube-proxy — The network proxy. Handles pod-to-pod communication and load balancing within the cluster.
  • Container runtime — Usually containerd or CRI-O, the actual thing that runs containers.

The Objects

Kubernetes is declarative. You describe what you want, and Kubernetes makes it happen. That description uses objects:

Pod

The smallest deployable unit. A pod is one or more containers that share network and storage. Usually, it's just one container — but they can share. Pods are ephemeral. They'll die. Kubernetes will restart them. Don't get attached.

ReplicaSet

Ensures N copies of a pod are running. If one dies, ReplicaSet notices and spawns a replacement. You usually don't create these directly — Deployment manages them.

Deployment

The thing you actually use. A Deployment manages ReplicaSets, which manage Pods. It handles rolling updates (gradually replace old pods with new ones), rollbacks, and scaling. You say "here's my container image" and "I want 5 replicas" — Deployment handles the rest.

Service

Pods come and go. Their IP addresses change. Services provide stable network endpoints. A Service is a stable IP and DNS name that routes to whatever pods match its selector. The service abstraction means your code doesn't care which pod it's talking to.

Ingress

How do you get traffic from the outside world into your cluster? Ingress. It's an HTTP/HTTPS router — host-based and path-based routing, TLS termination, the works.

ConfigMap & Secret

Configuration data and sensitive data (passwords, tokens). Mounted as environment variables or files. Separates config from code.

PersistentVolume

Containers are ephemeral. Pods die, their storage dies. PersistentVolume provides durable storage that survives pod restarts. You claim a chunk of storage, Kubernetes binds it to your pod.

The Control Loop

The heart of Kubernetes is the reconciliation loop. Every controller runs this:

  1. Observe current state (from etcd)
  2. Compare to desired state (from your YAML)
  3. Take action to make them match
  4. Repeat forever

This is why Kubernetes is resilient. Something breaks? The loop notices. Traffic spikes? The horizontal pod autoscaler notices. The loop fixes it.

What's Hard

Kubernetes is not simple. It's a tool for managing complexity — but it creates its own:

  • Learning curve — YAML, concepts, debugging distributed systems
  • Overhead — Running K8s costs resources. Not for small projects.
  • Debugging — When something breaks in a distributed system, figuring out where is its own skill.
  • Security — RBAC, network policies, secrets management — easy to get wrong.

When to Use It

Kubernetes makes sense when:

  • You have multiple services that need to communicate
  • You need automatic scaling
  • You need zero-downtime deployments
  • You have the team size to manage it

It doesn't make sense when:

  • You have one app running on one server
  • Your team is one person
  • You're just starting out

The Bottom Line

Kubernetes is a orchestration platform for containerized applications. It provides self-healing, scaling, service discovery, and deployment management. The cost is complexity — lots of it. But when you need to run many services at scale, it's the standard for a reason.

You don't need Kubernetes until you do. And when you do, you'll know.
