How Kubernetes Works

March 23, 2026

Kubernetes (K8s) is the operating system of the cloud-native world. But unlike Linux, it's not about managing a single machine—it's about managing clusters of machines as if they were one.

The Core Abstraction

At its heart, Kubernetes is a declarative system. You tell it what you want (3 replicas of this container), and it figures out how to make it happen. You never tell it how—you tell it what.

This is the fundamental shift: from imperative ("start this container on this machine") to declarative ("maintain 3 replicas of this pod"). The system constantly reconciles reality with your desired state.

The Control Plane

Kubernetes runs a control plane—a set of components that make decisions about the cluster:

  • API Server — The front door. Every interaction with Kubernetes goes through this. It's a stateless REST API that stores everything in etcd.
  • etcd — The brain. A distributed key-value store that holds the entire cluster state. This is what gets backed up.
  • Controller Manager — The reconciliation engine. It watches the current state and constantly tries to move toward the desired state. Every resource has a controller.
  • Scheduler — The decision maker. When you create a new pod, this decides which node it runs on based on resources, constraints, and policies.

Key Concepts

Pod

The smallest deployable unit. Usually one container, but can be multiple (like a sidecar). Pods are ephemeral—they can die and be rescheduled at any time.

Service

A stable endpoint for a set of pods. Since pods are ephemeral (IP changes on restart), services provide a stable DNS name and load balancing. The magic: kube-proxy maintains iptables rules to route traffic to healthy pods.

Deployment

A declarative way to manage pods. You define "I want 3 replicas of this container" and Kubernetes handles the rest—including rolling updates, rollbacks, and scaling.

StatefulSet

Like Deployment, but for stateful applications. Guarantees stable network identities and ordered deployment. Essential for databases and systems that need stable hostnames.

Namespace

Logical isolation. Not security boundaries (that's RBAC), but a way to organize resources. Great for separating dev/staging/prod within one cluster.

How Scheduling Works

When you create a pod:

  1. API server stores it in etcd
  2. Scheduler notices the unscheduled pod
  3. Scheduler filters nodes (resource constraints, labels, taints)
  4. Scheduler ranks nodes (scoring based on resources, affinity rules)
  5. Scheduler binds pod to node
  6. Kubelet on that node sees the assignment and starts the container

The Networking Model

Kubernetes has a flat network model:

  • Every pod gets a unique IP
  • Pods can communicate with any other pod without NAT
  • Services provide internal DNS
  • Ingress handles external traffic

This is simpler than traditional infrastructure but requires CNI plugins (Calico, Cilium, Flannel) to implement.

Why This Matters

Kubernetes solves the problem of "how do I run distributed systems reliably?" It handles:

  • Self-healing (restarting failed containers)
  • Scaling (horizontal pod autoscaling)
  • Rolling updates (zero-downtime deployments)
  • Service discovery (internal DNS)
  • Load balancing (round-robin across pods)

But it's complex. The operational burden is real—monitoring, logging, security, upgrades. It's become its own ecosystem of tools (Helm, ArgoCD, Prometheus, Istio). The pitch is "write once, run anywhere" but the reality is "write once, debug Kubernetes everywhere."

For most teams, managed K8s (EKS, GKE, AKS) is the play. Running your own control plane is a massive time sink that rarely provides value.
