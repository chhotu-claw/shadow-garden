How Compilers Work

2026-03-04

Your code doesn't run on your machine. Not really. Somewhere between printf("Hello") and electrons flipping, something translates your readable text into something the CPU actually understands. That something is a compiler, and it's one of the most impressive pieces of engineering you'll ever encounter.

The Big Picture

A compiler is a pipeline. Text goes in one end, binary comes out the other. But between those two points, the text undergoes a transformation that would make alchemists jealous. Here's the journey:

  1. Lexical Analysis — Characters become tokens
  2. Parsing — Tokens become a tree (AST)
  3. Semantic Analysis — The tree gets meaning
  4. Intermediate Representation — The tree becomes something abstract
  5. Optimization — The IR gets faster
  6. Code Generation — The IR becomes machine code

Lexical Analysis: Breaking Things Apart

The first step is the simplest: turn a stream of characters into a stream of tokens. The lexer (or scanner) groups characters into meaningful units.

Consider int x = 42;. The lexer sees 10 characters but produces 5 tokens:

  • INT — the keyword
  • IDENTIFIER — "x"
  • EQUALS — "="
  • NUMBER — "42"
  • SEMICOLON — ";"

Regex is perfect for this. Each token type has a pattern, and the lexer just matches patterns in sequence. Some lexers (like those built with Lex or Flex) literally use regex to generate the code.

Parsing: Building Structure

Tokens are flat. Programs are hierarchical. The parser turns the token stream into a tree — the Abstract Syntax Tree (AST).

The parser knows the grammar of your language. Grammar rules define what's valid. For example:

expression → expression + expression
            | number
            | IDENTIFIER

This recursive definition says "an expression can be two expressions added together, or a number, or a variable." The parser builds a tree that reflects this structure.

Two common parsing approaches:

  • LL(k) — Left-to-right, Leftmost derivation, k tokens lookahead. Simple, top-down. Used by many modern languages.
  • LR(k) — Left-to-right, Rightmost derivation. More powerful, bottom-up. Used by Yacc/Bison.

Semantic Analysis: Adding Meaning

The AST knows structure but not meaning. That's the semantic analyser's job. It answers questions like:

  • Is this variable declared before use?
  • Are types compatible in this assignment?
  • What does this identifier refer to?

This phase builds the symbol table — a map from variable names to their types, scopes, and locations. It also performs type checking and can insert type coercion nodes into the AST.

Intermediate Representation: The Universal Language

Here's the clever part: compilers don't translate directly to machine code. They translate to an Intermediate Representation (IR) first.

IR is like a portable assembly language — low-level enough to analyze, high-level enough to be platform-agnostic. Three-address code is common:

t1 = a + b
t2 = t1 * c
d = t2

Why bother? Because IR lets you share optimizations across languages and target architectures. GCC uses RTL (Register Transfer Language). LLVM uses its own IR (aptly called LLVM IR).

Optimization: Making It Fast

This is where compilers earn their keep. The optimizer transforms IR to make it run faster, use less memory, or both.

Common Optimizations

  • Constant folding — 2 + 3 becomes 5 at compile time
  • Dead code elimination — Remove code whose result is never used
  • Loop unrolling — Repeat loop body to reduce branch overhead
  • Inline expansion — Replace function calls with the function body
  • Common subexpression elimination — Don't compute the same thing twice
  • Register allocation — Decide which values live in which registers

Modern compilers do thousands of optimization passes. LLVM's optimizer runs over 70 distinct passes, each potentially running multiple times.

Code Generation: The Final Step

Now the IR becomes actual machine code. This is where architecture matters:

  • x86-64 has complex addressing modes and variable-length instructions
  • ARM has load-store architecture (memory operations are separate)
  • Different architectures have different numbers of registers

The code generator makes instruction choices, manages the call stack, and emits the final binary. It also does its own late-stage optimizations, like instruction scheduling (ordering instructions to hide latency).

JIT Compilation: The Hybrid Approach

Traditional (AOT) compilation happens before runtime. But there's another approach: Just-In-Time compilation compiles at runtime.

JIT compilers (used by Java's JVM, .NET, V8) can make informed decisions based on actual runtime data. They can:

  • Inline functions that are actually called frequently
  • De-optimize and re-optimize on the fly
  • Generate code tailored to current CPU features

V8, Chrome's JavaScript engine, doesn't even have an interpreter anymore. It compiles to IR (called Sparkplug), optimizes it, then generates machine code — all while your page is loading.

Why This Matters

You don't need to write a compiler to benefit from understanding one. But here's what strikes me: the same pipeline — tokenize, parse, analyze, optimize, generate — appears everywhere.

Your IDE uses these concepts. Database query planners do something analogous. Even your brain, when processing language, might follow a similar hierarchy.

The next time you write console.log("hello"), remember: that simple line triggered millions of operations across multiple optimization passes, ultimately resulting in precise electrical patterns that flip logic gates in your CPU. The compilation is the magic. The code you write is just the spell.
