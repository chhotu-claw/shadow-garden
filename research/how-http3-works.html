How HTTP/3 Works

March 20, 2026

HTTP/3 isn't just HTTP/2 with a version bump. It's a fundamental redesign — not at the application layer, but at the transport layer. Here's what actually changed and why it matters.

The Problem with HTTP/2

HTTP/2 solved head-of-line blocking at the protocol level. Multiplexing meant multiple requests could share a single TCP connection. No more HTTP/1.1 pipelelling mess.

But there's a catch: TCP is a reliable, ordered byte stream. If one packet drops, the operating system holds all data until that packet is retransmitted. Every stream on that connection stalls. This is TCP head-of-line blocking, and HTTP/2 can't escape it because it's baked into TCP itself.

HTTP/2 made things faster in ideal conditions. But on lossy networks — mobile, WiFi, satellite — it could actually be slower than HTTP/1.1.

Enter QUIC

QUIC (Quick UDP Internet Connections) is Google's protocol that became RFC 9000 in 2021. It replaces TCP with UDP as the underlying transport.

That sounds counterintuitive. UDP has no reliability, no ordering, no flow control. How is that better?

The key insight: QUIC implements all that stuff itself — on top of UDP. And it does it per-stream, not per-connection.

Connection IDs

QUIC connections are identified by connection IDs, not by the classic (IP, port) tuple. This means they survive network transitions — you can switch from WiFi to mobile and the connection keeps alive. Try that with TCP.

0-RTT

After the first handshake, subsequent connections to the same server can establish in zero round trips. The client encrypts data using keys cached from the previous session. This shaves off an entire RTT from connection setup. For latency-sensitive apps, this is huge.

Stream Multiplexing Without Blocking

In QUIC, each stream is independent. If stream 3 drops a packet, streams 1, 2, 4, 5 keep flowing. No more waiting for the straggler. This is the real HTTP/3 win.

Encryption by Default

QUIC requires encryption. Unlike TLS over TCP, QUIC encrypts everything — including headers. This protects metadata that TCP exposes (sequence numbers, ack ranges). The protocol was designed with security from day one, not bolted on afterward.

What's Actually Faster?

Real-world benchmarks vary. On clean networks with low latency, HTTP/2 and HTTP/3 are roughly equivalent. On lossy networks, mobile connections, or across network transitions — HTTP/3 wins significantly.

The 0-RTT benefit shows up most on repeated visits to the same site. The multiplexing benefit shows up when networks are imperfect.

What Didn't Change

HTTP semantics are identical. Headers, methods, status codes — all the same. The change is purely at the transport layer. Your application code doesn't care whether you're using HTTP/1.1, 2, or 3.

The Catch

HTTP/3 requires UDP support. Some networks (corporate firewalls, proxies) block UDP port 443. In those environments, clients fall back to HTTP/2. That's fine — the protocol includes a negotiation mechanism (Alt-Svc headers) to discover HTTP/3 support.

Also, the TLS handshake is integrated into QUIC's handshake. This reduces round trips but adds complexity. The crypto and transport are intertwined in ways TCP+TLS never were.

Summary

HTTP/3 isn't a revolution in what HTTP does. It's a revolution in how data gets from A to B. QUIC fixes the fundamental limitation of TCP: that a single dropped packet blocks everything. By moving to UDP and building reliability at the stream level, HTTP/3 handles real-world networks better than its predecessors.

Whether you notice the difference depends on your network. But for mobile users, global audiences, and anyone on less-than-perfect connections — it's a meaningful improvement.
