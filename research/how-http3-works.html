How HTTP/3 Works

March 8, 2026

HTTP/3 isn't just HTTP/2 with a new version number. It's a complete rewrite built on QUIC — a new transport protocol that throws out TCP entirely. Here's why that matters.

The Problem with TCP

TCP has been the backbone of the internet for decades. It's reliable, ordered, and handles congestion well. But it has a fundamental weakness: head-of-line blocking.

When you load a webpage, your browser opens multiple TCP connections to fetch resources in parallel. But if one packet gets lost on any of those connections, TCP waits for retransmission before delivering any data. Even if the other connections have all their packets ready, they block. The browser sits idle, waiting.

HTTP/2 tried to solve this by multiplexing everything over a single TCP connection. But that made it worse — now one lost packet blocks all streams. HTTP/2's priorities helped but couldn't fix the underlying problem.

Enter QUIC

QUIC (Quick UDP Internet Connections) was developed by Google and later standardized by the IETF. Its core insight: build reliability on top of UDP instead of fixing TCP.

UDP is connectionless — no handshake, no reliability, just fire-and-forget packets. QUIC adds:

  • Connection establishment — 1-RTT handshake (like TCP + TLS combined)
  • Reliability — packet acknowledgment, retransmission
  • Ordering — streams deliver in order
  • Congestion control — fair bandwidth sharing

But here's the key: each stream is independent. Lost packets only block that specific stream, not others. No more head-of-line blocking.

How HTTP/3 Uses QUIC

HTTP/3 runs directly over QUIC (instead of TLS over TCP). When you connect to an h3 server:

  1. Client sends CRYPTO frame — starts TLS handshake over QUIC
  2. Server responds — completes handshake (or 0-RTT if resumming)
  3. Streams begin — HTTP requests/responses flow over QUIC streams

Each HTTP request/response pair gets its own QUIC stream. They're independent, cancelable, and don't block each other.

What Makes HTTP/3 Different

0-RTT Connection Resumption

QUIC supports 0-RTT — if you've connected before, you can send data immediately without waiting for the handshake. The server remembers the previous state. This makes repeat visits nearly instant.

Connection Migration

QUIC connections are identified by a 64-byte connection ID, not IP/port. If your device switches from WiFi to cellular, the connection survives. No reconnection. This is huge for mobile.

No TLS Layer Separation

With HTTP/2 over TLS, you get one TLS session for everything. QUIC integrates TLS 1.3 directly — encryption is part of the protocol, not a layer on top. Simpler, faster.

Smaller Headers

HTTP/3 uses QPACK instead of HPACK. It compresses headers like HPACK but handles out-of-order delivery properly (since streams don't block each other anymore).

Performance Reality

In ideal conditions (low latency, reliable network), HTTP/3 is marginally faster than HTTP/2. The real gains come from:

  • Lossy networks — mobile, satellite, WiFi interference. HTTP/3 recovers faster.
  • Slow start — 0-RTT makes repeat connections instant.
  • Mobile — connection migration prevents reconnection delays.

On a stable fiber connection, you might not notice much difference. On a phone switching networks or a shaky connection, HTTP/3 is noticeably better.

Current State

Browser support is solid — Chrome, Firefox, Safari, and Edge all support HTTP/3. Server support is growing — Cloudflare, nginx (experimental), Caddy, and most CDNs.

The adoption curve looks like HTTP/2's but faster. Many sites are already serving over HTTP/3 by default.

The Trade-offs

HTTP/3 isn't perfect:

  • UDP blocked — some corporate firewalls and networks block UDP, forcing fallback to HTTP/2
  • Higher CPU — QUIC's encryption is more CPU-intensive than TLS (though this is improving)
  • Complexity — implementing QUIC correctly is hard. Many CDNs handle it so you don't have to.

Why This Matters

HTTP/3 is the first major transport protocol shift since TCP. It solves problems that HTTP/2 couldn't — not by tweaking TCP, but by replacing it.

The lesson: sometimes the best fix isn't optimization, it's a different foundation.
