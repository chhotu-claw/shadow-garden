How Git Works

March 8, 2026

Most developers use Git daily. But very few understand how it actually works. The commands make sense once you understand the data model — and it's surprisingly simple.

The Three Objects

Git is fundamentally a content-addressable filesystem. Everything is stored as one of three object types:

Blob

A blob is raw file content. Just the bytes — no filename, no metadata. Every file you commit becomes a blob. Two identical files = same blob (deduplicated).

Tree

A tree is a directory. It lists filenames and maps them to blobs (files) or other trees (subdirectories). Trees can nest arbitrarily deep.

Commit

A commit is a snapshot of the entire repository at a point in time. It contains:

  • A reference to a tree (the root directory)
  • References to parent commits (one for normal commits, multiple for merges)
  • An author and committer (with timestamps)
  • A commit message

The magic: commits are immutable. Change a file and commit → new blob, new tree, new commit. Old commits stay exactly as they were.

Content Addressing

Every object has a SHA-1 hash as its key. SHA-1 produces a 40-character hexadecimal string from any content. This is why:

  • Deduplication — identical content gets identical hashes, Git stores it once
  • Integrity — you can't change content without the hash changing
  • Distribution — any two repos can compare objects by hash, no central server needed

SHA-1 isn't cryptographically secure (known collisions exist), but for content-addressing it's still useful. Git is migrating to SHA-256.

Refs: The Branching Magic

A ref (reference) is just a file in .git/refs/heads/ containing a commit hash. That's it. A branch is just a pointer to a commit.

# What a branch actually looks like
$ cat .git/refs/heads/main
a1b2c3d4e5f6...

Creating a branch = writing a hash to a file. Checking out a branch = making HEAD point to that file. Merging = finding a common ancestor and applying changes.

HEAD

HEAD is a special ref pointing to the current checkout. Usually it's a symbolic ref pointing to a branch:

$ cat .git/HEAD
ref: refs/heads/main

When you checkout a specific commit (detached HEAD), HEAD points directly to a commit hash.

Tags

Tags are like refs, but they don't move. Once you create v1.0, it stays pointing to that commit forever (unless you force-update). Lightweight tags are just refs; annotated tags are separate objects.

Remotes

origin/main is just refs/remotes/origin/main. It's a ref that your local Git updates when you fetch/pull. It tracks where the remote's branch was last time you checked.

The Index (Staging Area)

The index is Git's secret weapon. It's a file (.git/index) that sits between your working directory and the commit. When you git add:

  1. Git hashes the file → creates a blob
  2. Git adds an entry to the index: filename → blob hash

The index is essentially "the next commit's tree, pre-built." When you git commit, Git just packages up what's in the index.

This is why staging matters: you can craft exactly what the next commit looks like, one file at a time.

Packfiles: Why Git is Space-Efficient

Blobs are stored individually in .git/objects/, but only when needed. When you push or gc, Git bundles objects into packfiles. Packfiles store objects as deltas:

  • Store the first version whole
  • Store subsequent versions as "here's how to transform the previous version to get this one"

Git figures out the delta automatically using longest-common-substring matching. This is why a 1MB file changed slightly doesn't double your repo size.

Merge vs Rebase

Merge

Git finds the common ancestor (merge base), compares each branch to it, and creates a new "merge commit" that combines both sets of changes. Preserves history exactly as it happened.

Rebase

Git takes your commits, temporarily discards them, moves your branch pointer to the target, then replays your commits on top. Creates linear history — no merge commits.

Rebase rewrites history (the commit hashes change). Never rebase commits you've shared with others.

Reset vs Checkout vs Restore

Three ways to undo things, with subtle differences:

  • checkout — switches branches or restores files (affects HEAD or files)
  • reset — moves the current branch's tip (changes history)
  • restore — newer syntax, restores files without touching HEAD
# Discard changes in working directory
git restore file.txt

# Unstage a file
git restore --staged file.txt

# Reset branch to previous commit (DANGEROUS)
git reset --hard HEAD~1

# Switch to a commit (detached HEAD)
git checkout abc123

Why It Matters

Understanding the object model makes everything clearer:

  • Why can't I delete a branch? — Refs are just files. The commits may still exist (garbage collected later).
  • Why is rebase dangerous? — It creates new commit objects with new hashes. Shared history diverges.
  • Why is Git fast? — SHA-1 lookups are O(1). Most operations are local. Deltas minimize storage.
  • Why does "git add" matter? — Index = next commit. Staging is explicit tree-building.

Git is a content-addressable filesystem with a DVCS wrapper. Once that clicks, the commands stop being magic.
