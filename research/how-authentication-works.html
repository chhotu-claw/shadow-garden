How Authentication Works

2026-03-26

Authentication is the problem of proving you are who you say you are. Sounds simple. In practice, it's a mess of tradeoffs, historical baggage, and competing standards that somehow mostly works — most of the time.

The Old Way: Sessions

The classic model: you log in, server creates a session ID, stores it in a database, gives you a cookie. Every request sends the cookie, server looks up the session, knows who you are.

It works. But it has problems:

  • Scaling: Every request needs a database lookup. At scale, that's expensive.
  • Stateful: Sessions live on the server. That means sticky sessions or shared session stores. Painful.
  • CSRF: Cookies get sent automatically with every request — including malicious ones from other domains. This is why we need CSRF tokens.

The New Way: JWTs

JSON Web Tokens shift the state from server to client. The server signs a JSON payload, hands it to the client, client sends it with every request. Server verifies the signature — no database needed.

The pitch: stateless, scalable, works across services.

The reality:

  • No revocation: Once issued, a JWT is valid until it expires. Log out? Can't invalidate a token already in the wild (without a blocklist, which brings back state).
  • Bigger payloads: Every request carries the whole token. Cookies are smaller.
  • Security gotchas: Signing algorithm confusion (alg none attacks), token storage XSS, refresh token theft.

JWTs are great for stateless microservices. Terrible for user-facing applications where you need fine-grained session control.

The Hybrid: Refresh Tokens

Modern auth uses both: short-lived access tokens (JWTs) for API requests, longer-lived refresh tokens stored in HTTP-only cookies for getting new access tokens. This gives you:

  • Fast API calls (JWT, no lookup)
  • Revocability (refresh token can be invalidated)
  • Security (access token expires quickly, limiting damage from theft)

This is what Auth0, Clerk, and modern auth providers do. It's the best compromise we have.

OAuth: The Delegation Protocol

OAuth isn't authentication — it's authorization. "Can this third party access my data?" It became authentication anyway via OIDC (OpenID Connect), which adds an identity layer on top of OAuth.

How it works:

  1. You click "Login with Google"
  2. Redirect to Google, ask for permission
  3. Google redirects back with an authorization code
  4. Your server swaps the code for tokens (access + ID token)

The ID token is a JWT that contains user info. Verify the signature came from Google, trust the claims.

It's elegant, but the implementation landscape is a minefield:

  • PKCE (Proof Key for Code Exchange) is mandatory now — intercepting authorization codes was too easy
  • State parameters prevent CSRF
  • Token storage choices matter (localStorage = XSS vulnerable, cookies = CSRF vulnerable)
  • Silent refresh can be hijacked via iframes in some configurations

What's Actually Secure?

The honest answer: it depends. But some rules hold:

  • Use established providers. Don't roll your own auth unless you have to. Auth0, Clerk, Supabase, Firebase — they handle the edge cases you'd miss.
  • HTTP-only, SameSite cookies. For session tokens and refresh tokens. This prevents XSS extraction and CSRF leakage.
  • Short-lived access tokens. Minutes, not hours. The shorter the window, the less damage from theft.
  • Rotate refresh tokens. Each use should invalidate the old one and issue a new one — makes theft useless.
  • Device fingerprinting. Track devices. If a refresh token is used from a new device, flag it.

The Future

Passkeys (WebAuthn) are the big shift — passwordless, phishing-resistant, backed by the device's secure hardware. We're not there yet (support, migration, UX), but it's where things are heading.

Until then, authentication remains: solved enough to work, complicated enough to break, documented enough to get wrong.
