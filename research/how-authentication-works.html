How Authentication Works

March 16, 2026

Authentication is the gatekeeper of the internet. It's how systems know who you are. But "knowing who you are" is surprisingly complex — and the solutions have evolved dramatically over time.

The Three Factors

There are three ways to prove your identity:

  • Something you know — password, PIN, security question
  • Something you have — phone, hardware token, smart card
  • Something you are — fingerprint, face, retina

Most systems use just one (password). Better systems use two or three. The trend is moving toward "something you have" via authenticator apps or hardware keys (YubiKey, Google Titan).

Session-Based Authentication

The classic approach: you log in, server creates a session ID, stores it in a database, sends it to your browser as a cookie. Every subsequent request includes the cookie. Server looks up the session, finds your user ID.

Browser → POST /login (email, password)
Server → validates → creates session #abc123 → stores {user: 42}
Server → Set-Cookie: session_id=abc123

Browser → GET /dashboard (Cookie: session_id=abc123)
Server → looks up session #abc123 → finds user 42 → returns dashboard

Pros: Simple, server has full control (can revoke anytime), stateless after login.

Cons: Server must store sessions (database or Redis), doesn't scale horizontally without shared storage, CSRF attacks possible.

Token-Based Authentication (JWT)

Instead of server storing sessions, the server signs a token containing your identity. The token is self-contained — the server can verify it without looking anything up.

Browser → POST /login (email, password)
Server → validates → creates JWT with payload {user: 42, exp: ...}
Server → returns JWT (not stored)

Browser → GET /dashboard (Authorization: Bearer <jwt>)
Server → verifies signature → extracts user 42 → returns dashboard

Pros: No server-side storage needed, scales horizontally, works across services.

Cons: Can't revoke tokens (until they expire), token lives in local storage (XSS risk), larger request size.

JWT Structure

A JWT has three parts separated by dots: header.payload.signature

// Header
{"alg": "HS256", "typ": "JWT"}

// Payload (this is what you see when you "decode" a JWT)
{"sub": "1234567890", "name": "John", "admin": true, "exp": 1516239022}

// Signature
HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)

Important: Anyone can decode the payload (base64 decode). The signature proves the server created it. Don't put secrets in the payload.

OAuth 2.0 — Delegated Authorization

OAuth isn't authentication — it's authorization. But it's often used for login (via "OAuth providers" like Google, GitHub). The flow:

  1. You click "Login with Google"
  2. Redirected to Google, asked to approve
  3. Google redirects back with authorization code
  4. Your server exchanges code for access token
  5. Your server uses access token to get user info from Google
User → clicks "Login with Google"
→ Redirects to: https://accounts.google.com/o/oauth2/v2/auth?
  client_id=YOUR_ID&redirect_uri=CALLBACK&response_type=code&
  scope=openid%20email%20profile

Google → shows consent screen → User approves
→ Redirects to CALLBACK?code=AUTH_CODE

Server → POST https://oauth2.googleapis.com/token
  code=AUTH_CODE&client_id=...&client_secret=...
→ Returns: {access_token: "...", id_token: "..."}

Server → GET https://www.googleapis.com/oauth2/v2/userinfo
  Header: Authorization: Bearer ACCESS_TOKEN
→ Returns: {id: "...", email: "..."}

Key insight: OAuth gives you an access token, not a session. The user can revoke access at any time (via their Google account), and your token stops working.

OpenID Connect (OIDC)

OIDC builds on OAuth 2.0 to provide authentication. It's basically OAuth with a standardized identity layer. When you "Login with Google" on a site, that's OIDC.

Key difference: OAuth is "what can this app do?" (authorization). OIDC is "who is this user?" (authentication).

Refresh Tokens

Access tokens expire quickly (15 minutes to a few hours). Refresh tokens let your app get new access tokens without re-logging in.

// When access token expires:
App → POST /refresh_token (refresh_token)
Server → validates refresh token → returns new access_token + new refresh_token

// Old refresh token is invalidated (rotation) or stays valid (static)

Best practice: Use refresh token rotation — each refresh gives you a new refresh token, old one is invalidated. This limits damage if a refresh token is stolen.

Password Storage — Hashing

Never store passwords in plain text. Always hash them. But not all hashes are equal:

  • MD5, SHA-1, SHA-256 — Too fast. Attackers can try billions per second. Don't use.
  • bcrypt — Slow, includes salt. Good. Work factor controls speed.
  • Argon2 — Winner of Password Hashing Competition. Memory-hard (resistant to GPU attacks). Recommended.
  • scrypt — Memory-hard like Argon2. Also good.
// Wrong (plain text — fireable offense)
{"email": "bob@example.com", "password": "hunter2"}

// Better (but still bad — too fast)
{"email": "bob@example.com", "password": "5e884898da28047151d0e56f8dc6292773603d0d6aabbdd62a11ef721d1542d8b"}

// Right (bcrypt with salt)
{"email": "bob@example.com", "password": "$2b$12$KIXxL...HASH..."}

Common Vulnerabilities

1. Password Reuse

Users reuse passwords. If your DB leaks and you used bcrypt with a good work factor, you're okay. If you used MD5... everyone's passwords are now public.

2. Token Storage

Storing JWT in localStorage = XSS theft risk. HttpOnly cookies = protected from JS, but vulnerable to CSRF. Best: use both (access token in memory, refresh token in HttpOnly cookie).

3. OAuth Redirect URIs

If you allow any redirect URI, attackers can steal auth codes. Strictly whitelist your URLs.

4. Session Fixation

Attack sets a session ID before you log in, then hijacks after. Fix: regenerate session ID after login.

The Future: Passkeys (WebAuthn)

Passwords are fundamentally broken. Passkeys replace them with cryptographic key pairs. Your device stores the private key, the server has the public key. No password to type, no password to leak.

WebAuthn is the standard. It supports:

  • Platform authenticators (Touch ID, Face ID, Windows Hello)
  • Cross-device credentials (sync via iCloud Keychain, Google Password Manager)
  • Attestation (prove the device is legitimate)

We're still early in the passkey transition, but it's the direction the industry is moving.

Summary

  • Authentication answers: "who are you?"
  • Sessions = server stores state; Tokens = client carries state
  • JWT = self-contained tokens, useful for APIs and SPAs
  • OAuth = delegated authorization; OIDC = authentication on top of OAuth
  • Never store plain passwords; use bcrypt or Argon2
  • The future is passkeys (WebAuthn) — no passwords to leak
