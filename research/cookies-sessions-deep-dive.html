How Cookies and Sessions Actually Work

April 4th, 2026

Every web developer uses cookies. Every web developer gets them wrong. Here's what actually happens when you log into a website.

The basic problem

HTTP is stateless. Each request is independent. The server has no idea who you are from one request to the next. That's the core problem cookies solve.

But "solving" it opened a can of worms that we're still dealing with today.

What a cookie actually is

A cookie is just a key-value pair the server sends in a response header:

Set-Cookie: session_id=abc123; Path=/; HttpOnly; Secure; SameSite=Strict

The browser stores this and sends it back with every subsequent request:

Cookie: session_id=abc123

That's it. That's the whole mechanism. Everything else is built on top of this.

Types of cookies

Session cookies — No expiration. Deleted when the browser closes. Used for temporary login states.

Persistent cookies — Have an Expires or Max-Age directive. Can survive browser restarts. Used for "remember me" features.

Third-party cookies — Set by domains other than the one you're visiting. The basis of ad tracking. Safari and Firefox are blocking these by default now.

The session cookie problem

Here's where it gets dangerous. A session cookie is just a magic token. If someone steals it, they become you. There's no way to tell the difference between the real user and the cookie thief.

This is called session hijacking. And it's easier than you'd think:

  • Network sniffing on public WiFi
  • XSS attacks that read document.cookie
  • Log files that accidentally leak session IDs
  • SQL injection that dumps the entire session table

What the flags actually do

HttpOnly — JavaScript can't read this cookie. Prevents XSS-based theft. Always use this for session tokens.

Secure — Only sent over HTTPS. Prevents network-level sniffing.

SameSite — Controls when the cookie is sent:

  • Strict — Only sent to the exact origin. Best for security.
  • Lax — Sent on top-level navigations and GET requests. The default.
  • None — Sent on all requests. Required for cross-site APIs but enables CSRF.

The better approach: JWTs (and their problems)

JSON Web Tokens moved session data from the server to the client. The server verifies a signature instead of looking up a session in a database.

But JWTs have their own problems:

  • No built-in expiration (you have to check exp claim)
  • No built-in revocation (once issued, it's valid until it expires)
  • Stored in localStorage (vulnerable to XSS)
  • Can be stolen just like session cookies

What actually works

The most secure approach combines multiple mechanisms:

  • Short session expiry (15-30 minutes)
  • Refresh tokens for longer sessions
  • IP address and user-agent validation
  • Device fingerprinting
  • Rate limiting on authentication endpoints
  • HTTP-only, Secure, SameSite=Strict flags

There's no perfect solution. There's only layers of defense.

The take

Cookies are a 1997 solution to a 2026 problem. They work, but they're not designed for the security landscape we live in. Every session token is a potential attack vector. The best you can do is make exploitation hard enough that attackers move on to easier targets.
