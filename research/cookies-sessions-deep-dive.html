How Cookies and Sessions Actually Work

2026-04-05

Every web developer uses cookies. Every web developer gets them wrong. This is a deep dive into what actually happens when you log into a website.

The cookie header

When you log in, the server sends this in the response header:

Set-Cookie: session_id=abc123; Path=/; HttpOnly; Secure; SameSite=Lax

Your browser stores this and sends it back with every request:

Cookie: session_id=abc123

That's the basics. Here's where it gets complicated.

The flags matter

  • HttpOnly — JavaScript can't read this cookie. Prevents XSS attacks stealing the session.
  • Secure — Only sent over HTTPS. Prevents eavesdropping on HTTP.
  • SameSite — Controls when cookies are sent with cross-site requests. Lax (default) allows normal navigation. Strict prevents all cross-site. None disables cross-site restriction (requires Secure).

Session hijacking

If an attacker gets your session cookie, they're logged in as you. Methods:

  • XSS — Inject JavaScript to read document.cookie
  • Network sniffing — Unencrypted HTTP traffic
  • Social engineering — Convincing you to click a link that sends the cookie

JWTs aren't the answer

Many switch to JWTs thinking they're more secure. They're not. A JWT is just a signed token — if it's stored in localStorage and vulnerable to XSS, it's no more secure than a cookie. The HttpOnly protection is the key, and JWTs in localStorage don't have it.

What actually works

  • Always use HttpOnly + Secure + SameSite=Strict for session cookies
  • Implement session expiration
  • Rotate session IDs on login
  • Use secure, random session IDs
  • Consider fingerprinting beyond just the cookie

Cookies are simple to use, hard to get right. Understanding the mechanics is the only way to secure them properly.
