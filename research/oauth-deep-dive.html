How OAuth 2.0 Actually Works

2026-04-04

OAuth 2.0 is everywhere. Every "Sign in with Google," "Connect with GitHub," or third-party app permission uses it. But the spec is dense, the terminology is confusing, and the flows are easy to get wrong.

The core problem

Before OAuth, if you wanted to let a third-party app access your data, you'd give it your password. That's obviously bad. OAuth solves this by introducing delegation without credential sharing.

The user authorizes an app to access their data, and the app gets a token instead of a password.

The actors

  • Resource Owner — You. The user who owns the data.
  • Client — The app requesting access.
  • Authorization Server — The service that authenticates you and issues tokens (e.g., Google's auth server).
  • Resource Server — The API that holds your data (e.g., Google's Drive API).

Authorization code flow

The most common flow. Here's how it works:

  1. User clicks "Sign in with Google" — The client redirects to Google's authorization server.
  2. User logs in — Google verifies the user's identity.
  3. Google asks permission — "This app wants to access your email and calendar."
  4. User approves — Google redirects back to the client with an authorization code.
  5. Client exchanges code for token — The client sends the code + its client secret to Google's token endpoint.
  6. Google returns access token — The client can now use this token to call Google APIs.

Why the extra step? The authorization code never goes to the browser's JavaScript. It goes straight to the server. This prevents token leakage from browser history or network inspection.

Token types

  • Access token — Short-lived (typically 1 hour), used to make API requests.
  • Refresh token — Long-lived, used to get new access tokens without re-authenticating.
  • ID token — A JWT containing user identity info (used in OpenID Connect).

Scopes

Scopes define what the client can access. read:calendar, write:drive, profile. The user sees these during consent. The authorization server enforces them.

Best practice: request the minimum scopes needed. Don't ask for full_access when you only need read:profile.

Security risks

  • Redirect URI manipulation — Attackers can steal auth codes if the redirect URI isn't validated.
  • Token leakage — Access tokens in URLs can end up in server logs. Use Authorization: Bearer headers instead.
  • CSRF — Without state parameter, attackers can hijack authorization flows.
  • Weak client secrets — If the client secret is compromised, attackers can impersonate the client.

What matters in production

  • Validate redirect URIs exactly
  • Use state parameter to prevent CSRF
  • Store tokens securely, never in localStorage for browser apps
  • Use PKCE (Proof Key for Code Exchange) for public clients
  • Implement token refresh logic

OAuth 2.0 is complex, but the core idea is simple: delegate access without sharing passwords. Everything else is engineering around that principle.
