How HTTP/2 Works

2026-03-14

HTTP/2 arrived in 2015 after nearly two decades of HTTP/1.1. It wasn't a rewrite — it was a retrofit. The goal: fix the performance problems that had accumulated as web pages grew from kilobytes to megabytes, without breaking the web.

The Problems with HTTP/1.1

HTTP/1.1 was designed for a web of simple HTML documents with inline images. It had two major bottlenecks:

Head-of-line blocking. A single TCP connection can only handle one request at a time. If you request 50 assets (HTML, CSS, JS, images, fonts), they're queued. The browser opens multiple connections (typically 6 per domain), but that's not enough for modern pages.

Inefficient headers. Every request repeats headers that barely change: User-Agent, Accept, Accept-Language. Cookies grow with every request. This overhead adds up.

Binary Framing

HTTP/1.1 is text-based — you can telnet to port 80 and type requests. HTTP/2 is binary. Under the hood, all data is broken into frames.

There are 10 frame types, but two matter most:

  • HEADERS — carries header data
  • DATA — carries the message body

Each frame has a stream identifier. This is the key to everything else.

Multiplexing

Multiple requests and responses can flow over a single TCP connection simultaneously. Each request/response pair is a "stream" identified by a number. Frames from different streams are interleaved — no waiting.

This eliminates head-of-line blocking at the application layer. (TCP-level blocking still exists, but TLS + HTTP/2 usually handles this well.)

Header Compression

HTTP/2 uses HPACK compression. Two tricks:

  • Static table. Common headers (method, path, status) are pre-defined and referenced by index.
  • Dynamic table. Headers seen in this connection are added to a table and referenced later.

HPACK also uses Huffman encoding for string values. The result: headers go from hundreds of bytes to a few bytes per request.

Server Push

When the client requests index.html, the server can proactively push style.css and app.js before the client parses the HTML and discovers it needs them.

How it works: server sends a PUSH_PROMISE frame announcing "I'm going to send you this." The client can refuse (send a RST_STREAM) if it already has it cached.

Stream Prioritization

Multiplexing creates a new problem: what loads first? HTTP/2 lets clients assign weights (1-256) and dependencies to streams.

Example: CSS should load before images. JavaScript is critical — block rendering. You express this as stream dependencies, and the server decides the order.

Other Features

  • Flow control. Like TCP window sizing, but per-stream. Client can say "stop, I can't handle more right now."
  • Stream reset. Send RST_STREAM to cancel a request mid-flight.
  • No pipelining. HTTP/1.1 had pipelining (send requests without waiting), but it was buggy and rarely used. Multiplexing replaces it.

What HTTP/2 Didn't Change

  • HTTP methods (GET, POST, etc.) — same
  • Status codes (200, 404, etc.) — same
  • URIs and semantics — same
  • The upgrade dance — optional, usually TLS ALPN is used instead

This was intentional. Proxies, servers, and clients could upgrade without breaking the web.

HTTP/3: The Next Generation

HTTP/3 replaces TCP with QUIC (UDP-based). Why? TCP has a fundamental problem: if one packet drops, the whole connection stalls until retransmission. QUIC handles packet loss at the stream level — one lost packet doesn't block other streams.

HTTP/3 also has 0-RTT connection establishment (like TLS 1.3), making it even faster on repeat visits.

What's Actually Different

For developers, HTTP/2 is mostly invisible. You don't write HTTP/2 code — your server enables it, your CDN terminates it, your browser handles it. The visible effects: faster page loads, especially over high-latency connections, and less need for domain sharding or sprite sheets.

The invisible win: fewer TCP connections means less CPU and memory on servers. This is why CDNs love HTTP/2.
