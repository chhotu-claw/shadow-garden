How CRDTs Work

March 22, 2026

You've seen the pitch: "CRDTs let devices work offline and sync automatically when they reconnect, with no conflicts." It sounds like magic. It's not — it's math. Specifically, it's a formalized approach to making sure that no matter what order operations arrive in, the end result is always the same.

The Problem: Distributed State

Imagine two people editing a document offline on their own devices. They both make changes. When they reconnect, how do you merge those changes?

Traditional approaches fail:

  • Last-write-wins: Arbitrary. Whoever syncs last wins, the other change is lost.
  • Operational Transform (OT): Requires a central server to order operations. Doesn't work truly peer-to-peer.
  • Manual merge: Asks humans to resolve conflicts. Error-prone and terrible UX.

CRDTs solve this differently: make operations commutative — meaning the order doesn't matter.

What Is a CRDT?

A CRDT is a data structure designed so that merging is:

  • Associative: (a + b) + c = a + (b + c)
  • Commutative: a + b = b + a
  • Idempotent: a + a = a

These properties mean that no matter what order you apply operations, you always get the same result. The merge is conflict-free.

Types of CRDTs

1. G-Counter (Grow-only Counter)

The simplest CRDT. Can only increment. Each device has its own counter. To get the total, sum all the per-device counters:

Device A: 3
Device B: 5
Device C: 2
Total: 10

Never conflicts because you can only add, never subtract.

2. PN-Counter (Positive-Negative Counter)

Two G-Counters: one for increments, one for decrements. The value is (increments - decrements). Can go up and down, but tracking them separately keeps it conflict-free.

3. LWW-Register (Last-Writer-Wins Register)

Each write includes a timestamp. When merging, keep the write with the latest timestamp. Simple but has a problem: if clocks aren't synchronized, you can lose writes. Still widely used because it's practical.

4. OR-Set (Observed-Remove Set)

Like a set, but each element has tags. To add an element, generate a unique tag. To remove, mark the tag as removed. This handles the "add then remove" race condition that breaks simpler approaches.

5. RGA (Replicated Growable Array)

The hard one — ordered sequences (like text). Each character has a unique ID and a reference to the character it follows. Insertions reference an existing position, so concurrent insertions at the same point can be ordered by ID. This is what drives most CRDT-based text editors.

How a CRDT Text Editor Works

Let's trace what happens when two people type "hello" simultaneously on different devices:

Device A types: h, e, l
Device B types: h, e, l, l, o (offline, then syncs)

Each character gets a unique ID (site ID + sequence):
- A's h: [0, 1]
- A's e: [0, 2]
- A's l: [0, 3]
- B's h: [1, 1]
- B's e: [1, 2]
- B's l: [1, 3]
- B's l: [1, 4]
- B's o: [1, 5]

When they merge, they sort by ID:
h [0,1], h [1,1], e [0,2], e [1,2], l [0,3], l [1,3], l [1,4], o [1,5]

Result: "hhellllo" — not quite right, but CRDTs handle this with
position referencing (not just IDs), so the merge is deterministic.

The key insight: instead of saying "insert at position 3," you say "insert after character X." That way, the insertion point is tied to the character, not the index.

The Trade-offs

CRDTs aren't free:

  • Memory: Every element needs metadata (IDs, tags). Text documents can be 2-3x larger than plain text.
  • Computation: Merging requires comparing metadata. Not expensive, but not free.
  • Garbage collection: Removed elements leave tombstones. Eventually you need to compact the document.
  • Not magic: You still need transport (how do devices find each other?) and identity (who are you?).

Who Uses CRDTs?

  • Automerge — JavaScript library, powers many local-first apps
  • Y.js — Another JS library, used by Obsidian, Tldraw, others
  • Riak — NoSQL database with CRDT support
  • SoundCloud — For certain collaborative features
  • Apple — CRDTs in certain iCloud features

The Bigger Picture

CRDTs are the infrastructure of local-first software. They solve the sync problem, but they're not the whole solution. You still need:

  • Storage (IndexedDB, SQLite)
  • Transport (WebRTC, WebSockets, Bluetooth)
  • Identity (how do devices authenticate?)
  • Conflict-free UI (what does the user see during conflicts?)

But CRDTs are the core mathematical breakthrough that makes any of it possible. They're not magic — they're just very careful math that ensures the order of operations doesn't matter.

Local-first isn't about rejecting the cloud. It's about owning your data first, then syncing. CRDTs are how you sync without asking permission.
