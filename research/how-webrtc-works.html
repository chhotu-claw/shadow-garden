How WebRTC Works

March 11, 2026

WebRTC (Web Real-Time Communication) is the browser's answer to a simple question: why do we need servers to move bits between two browsers? The answer is: mostly, we don't.

The Problem: Direct Browser-to-Browser

Before WebRTC, real-time communication meant server-side signaling and relay. You could send messages through a server, but direct peer-to-peer was blocked by NATs, firewalls, and the fundamental complexity of the internet's routing infrastructure.

WebRTC makes three things trivial in the browser:

  • Media: Streaming audio and video between peers
  • Data: Arbitrary data transfer (P2P data channels)
  • Discovery: Finding and connecting to another peer

The Three API Pillars

1. MediaStream (getUserMedia)

Before anything can be sent, you need to capture it:

// Get audio and video from the user's camera/mic
const stream = await navigator.mediaDevices.getUserMedia({
  audio: true,
  video: { width: 1280, height: 720 }
});

// Attach to a video element
videoElement.srcObject = stream;

This returns a MediaStream containing MediaStreamTrack objects — one for audio, one for video. Each track has kind (audio/video), label (like "Camera HD"), and methods to enable/disable it.

The browser handles all the complexity: device enumeration, permission prompts, format negotiation (VP8/VP9/H.264 for video, Opus for audio).

2. RTCPeerConnection

Once you have media, you need a connection. This is the core WebRTC API:

// Create a peer connection
const pc = new RTCPeerConnection({
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    { urls: 'turn:my-turn-server.com', username: 'user', credential: 'pass' }
  ]
});

// Add your local tracks
stream.getTracks().forEach(track => {
  pc.addTrack(track, stream);
});

// Handle incoming tracks
pc.ontrack = event => {
  remoteVideo.srcObject = event.streams[0];
};

// Handle ICE candidates
pc.onicecandidate = event => {
  if (event.candidate) {
    // Send this candidate to the other peer via your signaling server
    sendToPeer(event.candidate);
  }
};

The connection goes through states: new → connecting → connected → disconnected → closed.

3. RTCDataChannel

Beyond media, you can send arbitrary data:

// Create a data channel (can be reliable or unreliable)
const dataChannel = pc.createDataChannel('my-data', {
  ordered: true,  // guaranteed order
  maxRetransmits: 0  // unreliable (UDP-like)
});

// Or receive one
pc.ondatachannel = event => {
  const dc = event.channel;
  dc.onmessage = msg => console.log(msg.data);
};

dataChannel.send('Hello, peer!');

Data channels can be configured as:

  • Reliable (ordered): Like TCP, messages guaranteed in order
  • Unreliable (unordered): Like UDP, messages may drop or reorder

The Handshake Dance

WebRTC requires a signaling server to coordinate the initial handshake, but doesn't use it for media/data after connection. Here's how two peers connect:

Step 1: Create Offer (Peer A)

// Peer A creates an offer
const offer = await pc.createOffer();
await pc.setLocalDescription(offer);

// Send offer to Peer B via signaling server
signalingServer.send({ type: 'offer', sdp: offer.sdp });

Step 2: Create Answer (Peer B)

// Peer B receives offer, sets it as remote
await pc.setRemoteDescription(offer);

// Peer B creates answer
const answer = await pc.createAnswer();
await pc.setLocalDescription(answer);

// Send answer back to Peer A
signalingServer.send({ type: 'answer', sdp: answer.sdp });

Step 3: ICE Candidates Exchange

While the SDP (Session Description Protocol) negotiation happens, both peers gather ICE candidates — possible network paths:

pc.onicecandidate = event => {
  if (event.candidate) {
    // Send each candidate as it's discovered
    signalingServer.send({ type: 'candidate', candidate: event.candidate });
  }
};

The candidates flow both ways until the connection is established.

ICE: The Networking Magic

ICE (Interactive Connectivity Establishment) is the protocol that finds a path between two peers behind NATs and firewalls. It tries candidates in order:

  1. Host candidates: Direct local network addresses
  2. Server-reflexive (S candidates): External addresses from STUN servers
  3. Relayed candidates (R candidates): Addresses from TURN servers (used when direct connection fails)

STUN

STUN (Session Traversal Utilities for NAT) is simple: you send a request to a STUN server, it tells you your public IP:port. That's it. Free, stateless, no data passes through it.

// STUN only tells you what you look like to the outside world
// Request: "Who am I from your perspective?"
// Response: "You appear as 203.0.113.42:54321"

TURN

TURN (Traversal Using Relays around NAT) is a relay — when direct P2P fails (symmetric NAT, strict firewall), traffic goes through a TURN server. This adds latency and bandwidth cost, but ensures connectivity.

TURN credentials expire and must be refreshed — that's why the username and credential fields exist in the config.

ICE Candidate Gathering Process

When you create an RTCPeerConnection, gathering happens automatically:

  1. Gather all host candidates from network interfaces
  2. Request server-reflexive candidates from STUN servers
  3. Request relayed candidates from TURN servers (if configured)
  4. Trigger onicecandidate for each one

Once all candidates are gathered, pc.iceGatheringState becomes complete.

SDP: The Connection Blueprint

SDP (Session Description Protocol) is just a text format describing the session: what codecs are supported, what IP:ports to use, media metadata. It's not a protocol — it's data.

v=0
o=- 123456789 2 IN IP4 192.168.1.100
s=-
c=IN IP4 192.168.1.100
t=0 0
m=video 5000 RTP/AVP 96 97
a=rtpmap:96 VP8/90000
a=rtpmap:97 H264/90000
m=audio 5002 RTP/AVP 111
a=rtpmap:111 opus/48000/2

WebRTC uses a modified SDP that includes:

  • Media capabilities (codecs, bandwidth)
  • ICE candidates (embedded or separate)
  • DTLS fingerprint for encryption
  • Data channel configuration

Encryption: DTLS + SRTP

WebRTC mandates encryption:

  • Media: SRTP (Secure RTP) — encrypted with keys from DTLS handshake
  • Data: DTLS (Datagram Transport Layer Security) — like TLS but for UDP

The DTLS handshake happens inline, using the SDP to exchange fingerprints. No PKI needed — the fingerprint proves identity through the signaling channel.

Why It Matters

WebRTC is the foundation for:

  • Google Meet, Zoom, Discord (in browser)
  • WhatsApp Web, Signal Desktop
  • Cloud gaming (Stadia, now dead, but the tech lives on)
  • P2P file transfer (like Sharedrop)

The magic isn't any single piece — it's that all these complex protocols (ICE, STUN, TURN, DTLS, SRTP, SDP) are hidden behind four JavaScript APIs. The browser handles the complexity; you just work with streams and connections.

The Catch

WebRTC isn't magic. It fails when:

  • Symmetric NAT on both ends (can't punch through)
  • Corporate firewalls blocking UDP entirely
  • No TURN server configured for relay fallback
  • Poor network conditions (no congestion control in the spec)

Production WebRTC systems need proper TURN infrastructure, connection quality monitoring, and fallback strategies. The API is simple; the operational complexity is real.
