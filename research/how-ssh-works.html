How SSH Works

March 19, 2026

SSH is the backbone of secure server access. Yet most developers treat it as magic — run ssh user@host and hope for the best. Let's crack it open.

The Protocol Layers

SSH isn't a single protocol — it's a stack:

  • Transport Layer — handles encryption, key exchange, compression
  • User Authentication Layer — validates who you are (password, key, etc.)
  • Connection Layer — multiplexes channels (shell, port forwarding, file transfer)

When you connect, the client and server negotiate all of this in about 4 round trips.

Key Exchange: How You Agree on a Secret

The hardest problem in cryptography: how do two parties agree on a shared secret without anyone else learning it?

SSH uses Diffie-Hellman (or elliptic-curve Diffie-Hellman in SSHv2). Here's the magic:

  1. Client and server agree on a large prime number p and a generator g
  2. Client generates a private key a, computes ga mod p, sends this to server
  3. Server generates private key b, computes gb mod p, sends to client
  4. Client computes (gb)a = gab mod p
  5. Server computes (ga)b = gab mod p
  6. Both now have the same shared secret — but an eavesdropper only sees ga and gb, which are useless without a or b

This is the same math that powers TLS. The elegance: you never transmit the actual secret.

Symmetric Encryption: What Actually Protects Your Data

Once key exchange gives both parties a shared session key, symmetric encryption takes over. SSHv2 supports several ciphers:

  • AES — the standard. AES-256-GCM is preferred (authenticates encryption, catches tampering)
  • ChaCha20-Poly1305 — fast, modern, preferred by OpenSSH
  • 3DES — deprecated, slow, don't use

The session key is derived from the Diffie-Hellman output using a key derivation function (KDF) — typically HKDF or something similar. This splits the key into separate encryption and MAC keys for each direction.

Server Authentication: How You Know It's The Right Server

Client authenticates server via its host key. On first connect, you see:

The authenticity of host 'server.com (1.2.3.4)' can't be established.
ECDSA key fingerprint is SHA256:abcd1234...

The client checks this key against ~/.ssh/known_hosts. If you've connected before and trusted it, you're good. If not — you're taking a leap of faith (or should verify the fingerprint out-of-band).

Host keys live in /etc/ssh/ on servers: ssh_host_ecdsa_key, ssh_host_ed25519_key, etc.

User Authentication: How The Server Knows It's You

Once the transport is encrypted, the server needs to verify you're allowed in:

Password Authentication

Simple — client encrypts the password with the session key and sends it. Server compares against /etc/shadow. Works, but:

  • Passwords can be brute-forced server-side
  • You're trusting the server with your password
  • No protection against server compromise

Public Key Authentication (The Right Way)

You have a keypair: ~/.ssh/id_ed25519 (private) and ~/.ssh/id_ed25519.pub (public). Here's how it works:

  1. Client sends the public key to server
  2. Server checks if that key is authorized in ~/.ssh/authorized_keys
  3. Server generates a random challenge, encrypts it with the user's public key
  4. Client decrypts it with their private key, sends it back
  5. Server verifies — if it matches, you're in

The critical insight: your private key never leaves your machine. The server can't impersonate you because it doesn't have your key. This is asymmetric cryptography's power.

Forward Secrecy: Why Session Keys Die

SSH provides perfect forward secrecy (PFS) by default with Diffie-Hellman key exchange. Each session gets a unique key — if someone steals a server's private key years later, they can't decrypt old sessions.

TLS learned this the hard way. SSH got it right from the start.

Port Forwarding: SSH's Secret Superpower

Once authenticated, SSH can tunnel other traffic:

  • -L (local forward): ssh -L 8080:localhost:80 user@host — forward local 8080 to host's port 80
  • -R (remote forward): ssh -R 8080:localhost:80 user@host — let host forward to your local port 80
  • -D (SOCKS proxy): ssh -D 1080 user@host — create a SOCKS proxy through the tunnel

This is how developers access cloud databases, bypass firewalls, and tunnel traffic through untrusted networks. The encryption protects everything inside.

What SSH Doesn't Fix

  • Server compromise — if the server is hacked, attacker gets your session
  • Client compromise — malware on your machine steals keys
  • Social engineering — you can be tricked into connecting to the wrong server
  • Key management — SSH keys accumulate, expire, get revoked poorly

The Modern Best Practices

  • Use Ed25519 keys — fast, secure, small
  • Add keys to ssh-agent — so you type passphrase once per session
  • Use Jump hosts (-J) instead of exposing servers to the internet
  • Use HashiCorp Vault or similar for key rotation in organizations
  • Audit authorized_keys regularly — orphaned keys accumulate

Why SSH Still Matters

In an era of web consoles, REST APIs, and managed platforms, SSH remains the escape hatch. When everything else fails, SSH is there — direct access to the metal, encrypted, authenticated, reliable.

It's been around since 1995 (SSH-1), evolved to SSHv2 in 2006, and remains fundamentally unchanged in its architecture. That's not stagnation — that's solid engineering.
