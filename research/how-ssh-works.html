How SSH Works

The protocol behind every secure server connection

SSH (Secure Shell) is thirty years old and still the standard for secure server access. It replaced rlogin, telnet, and FTP — protocols that sent everything in plaintext. Every time you connect to a server, a remarkable dance of cryptography happens in milliseconds. Here's how it actually works.

The Three Layers

SSH is actually three protocols stacked together:

  • SSH-TRANS — The transport layer. Handles encryption, compression, server authentication, and key exchange.
  • SSH-AUTH — The authentication layer. Decides who can log in (password, public key, etc.).
  • SSH-CONN — The connection layer. Multiplexes channels (shell, port forwarding, file transfer) over a single encrypted tunnel.

When people say "SSH," they usually mean the whole stack. But understanding each layer makes debugging and security much easier.

Key Exchange (Diffie-Hellman)

Before any data is sent, client and server must agree on encryption keys — without sending the keys themselves. SSH uses Diffie-Hellman (or its elliptic curve variant, ECDH) for this.

Diffie-Hellman Key Exchange

1. Client and server agree on a large prime p and generator g

2. Client picks secret a, sends ga mod p

3. Server picks secret b, sends gb mod p

4. Both compute gab mod p — the shared secret

5. Derive session keys from this shared secret

The brilliance: an eavesdropper sees ga and gb but cannot compute gab without solving the discrete logarithm problem — computationally infeasible for large primes.

Server Authentication

How do you know you're connecting to the real server and not an attacker's machine?

Each server has a host key — a permanent public/private key pair. On first connection, the client doesn't know the server's key:

$ ssh server.example.com
The authenticity of host 'server.example.com (203.0.113.1)' can't be established.
ECDSA key fingerprint is SHA256:abc123...
Are you sure you want to continue connecting (yes/no)?

Once you type yes, the key is stored in ~/.ssh/known_hosts. Future connections are verified against this stored key. If it changes, SSH warns you — possible attack.

For larger deployments, DNS SSHFP records or a local CA can bootstrap trust, but the core mechanism remains: you verify the server's host key on first connect.

User Authentication

Once the transport is encrypted, the server needs to verify you are who you claim to be. SSH supports multiple methods:

Password Authentication

The oldest method. Server requests password, client sends it (encrypted over the SSH tunnel). Simple but requires typing a password each time — and passwords are the weakest link.

Public Key Authentication (Preferred)

You generate a key pair, put the public key on the server:

# On server - add client's public key to authorized_keys
$ cat ~/.ssh/authorized_keys
ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABgQC... user@client

When you connect, the server sends a challenge (random data). Your private key signs it. The server verifies the signature with your public key. If it matches, you're in — no password needed.

1. Client sends public key
→
2. Server encrypts random challenge
→
3. Client decrypts with private key, signs
→
4. Server verifies signature

The private key never leaves your machine. Even if the server is compromised, your password isn't exposed.

Keyboard-Interactive

A generalization of passwords. Server can prompt for multiple challenges (e.g., password + OTP). Used by some MFA setups.

Agent Forwarding

If you need to SSH from the server to another machine, your keys can be forwarded through the connection:

$ ssh -A user@jump-server
$ ssh user@internal-server  # Uses your local keys via agent



Useful for bastion hosts, but be careful — the intermediate server gets access to your agent (can ask for signatures, not keys).



Encryption



Once authenticated, all traffic is encrypted. SSH uses symmetric ciphers like ChaCha20-Poly1305 (SSHv2 default), AES-GCM, or AES-CTR.



Keys are derived from the Diffie-Hellman shared secret using a Key Derivation Function (KDF) — typically HKDF or just repeated hashing. This produces:



    

  • Encryption key — encrypts data
    • 

  • MAC key — authenticates data integrity
    • 

  • IV — initialization vector for block ciphers
    • 




Different keys for each direction (client→server, server→client). This is called separate encryption contexts.

The whole handshake typically completes in 2–4 round trips (~50–200ms on typical networks). After that, every keystroke is encrypted.



Channels and Port Forwarding



Once connected, SSH multiplexes multiple logical channels over a single connection:



    

  • Interactive shell — your terminal session
    • 

  • exec channel — running a single command
    • 

  • pty allocation — pseudo-terminal for interactive programs
    • 

  • TCP port forwarding — tunnel traffic through SSH
    • 

  • Agent forwarding — forward key agent
    • 




Local Port Forwarding (-L)


$ ssh -L 8080:internal-service:80 user@bastion



Connects to bastion, then forwards local port 8080 to internal-service:80 through that tunnel. Useful for accessing internal services through a jump host.



Remote Port Forwarding (-R)


$ ssh -R 8080:localhost:3000 user@public-server



Server's port 8080 forwards to your localhost:3000. Lets you expose a local service to the internet through a remote server.



Dynamic Port Forwarding (SOCKS Proxy, -D)


$ ssh -D 1080 user@bastion



Creates a local SOCKS proxy on port 1080. Configure your browser to use it, and traffic routes through the bastion. Effectively turns SSH into a poor man's VPN.

Security Pitfalls



Never share your private key. Never put a private key on a server you don't control.



    

  • Weak keys — DSA keys are broken; use RSA-4096 or Ed25519
    • 

  • Agent forwarding to untrusted servers — they can use your agent to authenticate elsewhere
    • 

  • Password auth — vulnerable to brute force; use keys
    • 

  • PermitRootLogin — never allow root login directly; use sudo from a regular user
    • 

  • Outdated SSH versions — SSHv1 is broken; force SSHv2 (default since OpenSSH 7.0)
    • 

  • known_hosts poisoning — an attacker with DNS/CA compromise can impersonate servers
    • 




Modern Best Practices



    

  • Use Ed25519 keys — modern, fast, secure. RSA as fallback.
    • 

  • Protect private keys with passphrases — ssh-keygen -o -a 100 for stronger KDF
    • 

  • Use SSH certificates — avoids deploying keys to many servers
    • 

  • Disable password auth — PasswordAuthentication no
    • 

  • Use Match blocks in sshd_config for conditional rules
    • 

  • Consider ssh-agent to avoid typing passphrases repeatedly
    • 




The Bigger Picture



SSH is elegant because it layers simple primitives into something powerful: Diffie-Hellman for key exchange, asymmetric crypto for authentication, symmetric ciphers for encryption, and channel multiplexing for flexibility. It's a reminder that protocols age — but good architecture lasts.



Three decades later, SSH remains the workhorse of server administration. Not bad for a protocol designed on a train.
