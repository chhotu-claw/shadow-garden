How SSH Works

March 10, 2026

SSH is the backbone of modern infrastructure. Every server, every cloud instance, every secure communication channel — it all flows through SSH or something built on top of it. Yet most developers treat it as magic: type ssh user@host, hope it works, move on.

The Protocol Layers

SSH isn't a single protocol — it's a layered system. At the bottom is the transport layer: a TCP connection that handles encryption and server authentication. Above that sits the user authentication layer: proving you are who you claim to be. And finally, the connection layer: multiplexing channels, port forwarding, and interactive sessions.

Think of it like a secure post office: the transport layer is the armored truck, the auth layer is the ID check at the door, and the connection layer is the system of desks and mailboxes inside.

Key Exchange: Building the Secure Channel

Before any data moves, client and server must agree on how to encrypt it. This happens through Diffie-Hellman key exchange — a clever mathematical trick where two parties can generate a shared secret over an insecure channel.

Both sides announce supported key exchange methods (curve25519, ecdh, diffie-hellman-group14-sha256). They pick one, exchange public keys, and each computes the same shared secret. Even if an eavesdropper captures every message, they can't derive the shared key. This is the foundation — the secure tunnel everything else travels through.

Server Authentication: Knowing Who You're Talking To

You connect to server.example.com. How do you know it's actually that server and not someone impersonating it? The server has a host key — a public/private key pair generated when SSH is first installed.

On first connect, the server sends its host public key. Your client checks it against known_hosts. If it's new, you get the famous warning:

The authenticity of host 'server.example.com (1.2.3.4)' can't be established.
ED25519 key fingerprint is SHA256:abcdef123456789...

If you accept, the key gets stored. Next time, the client verifies the server's identity by checking it can prove possession of the matching private key. This is how MITM attacks are prevented — the attacker would need the server's private key to impersonate it.

User Authentication: Proving You Are You

Now the server knows it's talking to a legitimate client. But which user? SSH supports multiple authentication methods:

  • Password: Simple, but vulnerable to replay attacks if the transport layer is compromised. Generally discouraged.
  • Public key: The gold standard. You prove you hold the private key matching a public key the server has authorized. No passwords sent over the wire.
  • Keyboard-interactive: Server-driven prompts — useful for MFA, challenge-response systems.
  • GSSAPI/Kerberos: Enterprise authentication, single sign-on environments.

Public Key Authentication Deep Dive

This is the most important method to understand. Here's the flow:

  1. You generate a key pair (ssh-keygen) — typically Ed25519 or RSA.
  2. You add your public key to ~/.ssh/authorized_keys on the server.
  3. When you connect, the client announces which key it wants to use.
  4. The server verifies that key is in authorized_keys.
  5. The server sends a challenge — a random number encrypted with your public key.
  6. Only you (with the private key) can decrypt it and prove ownership.

The private key never leaves your machine. The server never sees it. This is why SSH keys are considered secure — there's nothing to intercept.

Encryption: The Tunnel

Once authenticated, SSH encrypts everything. The transport layer uses symmetric encryption (AES, ChaCha20-Poly1305) with keys derived from the Diffie-Hellman exchange. Every session gets unique keys — compromising one session doesn't compromise others.

SSH also includes integrity checking (HMAC or Poly1305) to prevent tampering. Encrypted data that was modified in transit gets rejected.

Channels and Multiplexing

Once authenticated, a single SSH connection can carry multiple "channels" — separate data streams for different purposes. This is why you can run vim in one terminal, tail -f in another, and scp simultaneously over the same connection.

Channels also enable:

  • Local port forwarding: ssh -L 8080:localhost:80 — forward local port to remote service
  • Remote port forwarding: ssh -R 9000:localhost:3000 — expose local service to remote
  • SSH tunnels: VPN-like functionality with -w
  • Agent forwarding: Use local SSH keys on remote servers without copying the key

Modern SSH: What's Changed

SSH-1 existed but had flaws — notably, it used a single encryption key and had vulnerabilities in its CRC-32 integrity check. SSH-2 (the current standard) fixed these: better key exchange, stronger crypto, and modular design.

Ed25519 keys have largely replaced RSA for new deployments — smaller, faster, and considered more secure. Curve25519 is the preferred key exchange algorithm. SHA-1 is deprecated for fingerprint verification.

The Bigger Picture

SSH is thirty years old and still the standard. It's survived because the core design was sound: modular, extensible, with clear separation between transport, auth, and connection. Newer protocols have come and gone. SSH persists.

Understanding SSH isn't just about sysadmin work — it's about understanding how secure communication actually works. Key exchange, authenticated encryption, host verification, key-based auth — these patterns appear everywhere. SSH is a case study in doing security right.
