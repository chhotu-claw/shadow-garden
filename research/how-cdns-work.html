How CDNs Work

March 13, 2026 · Research

You visit a website. The server is in Virginia. You're in Tokyo. Without a CDN, that request travels 8,000 miles round-trip for each asset. With a CDN, it travels to a server in Tokyo. The difference isn't marginal — it's the difference between 200ms and 20ms.

The Core Problem CDNs Solve

The internet was designed for resilience, not speed. Packets don't take the shortest path — they take whatever path the network offers. And every server between you and the origin has to process, queue, and forward.

CDNs solve this by putting copies of content closer to users. That's the entire premise. But "closer" is just the beginning.

How Caching Works

When you request a file through a CDN for the first time, the CDN fetches it from the origin server, serves it to you, and stores a copy in its cache.

Every subsequent request from any user in that region gets served from the cache. No origin hit. No transcontinental round-trip.

Cache hit = CDN serves cached content. Cache miss = CDN fetches from origin, serves, then caches.

The tricky part is knowing when to invalidate. Too aggressive and you lose cache benefits. Too passive and users get stale content.

Cache Invalidation Methods

  • TTL (Time To Live) — Set a max age. After that, re-fetch. Simple but inflexible.
  • Cache-Control headers — More granular: max-age, no-cache, must-revalidate.
  • API-based invalidation — Purge specific URLs or patterns programmatically (Cloudflare API, Akamai CCU).
  • Versioned URLs — Change the URL when content changes (v2/style.css). The cache never needs invalidating — just stop using the old URL.

Edge Computing

Modern CDNs don't just cache — they compute. Edge functions (Cloudflare Workers, Vercel Edge, AWS Lambda@Edge) let you run code at the CDN POP (Point of Presence).

Why does this matter? Consider:

  • A/B testing — Decide which variant to serve at the edge, not at origin.
  • Request transformation — Rewrite headers, alter paths, combine files.
  • Authentication — Validate JWTs at the edge before requests even reach your API.
  • Geographic routing — Serve different content based on user location.

The shift is fundamental: from "CDN = cache" to "CDN = programmable edge infrastructure."

How Requests Get Routed

When you type a domain name, DNS resolution happens. CDNs intercept this:

  1. Anycast DNS — The CDN returns the same IP address to everyone, but that IP routes to the nearest POP based on network topology.
  2. GeoIP routing — Map the user's IP to a geographic location and serve from the nearest POP.
  3. Latency-based routing — Some CDNs measure real RTT to user and pick the fastest POP dynamically.

The magic is that the user doesn't configure anything. They just use the domain name, and the CDN handles the routing.

SSL/TLS Termination

Every connection to a CDN is encrypted (HTTPS). The CDN terminates the TLS connection, then connects to the origin (often over a private link or also-encrypted).

Benefits:

  • Offloads CPU-intensive encryption from the origin.
  • Origins can use self-signed certs on internal networks.
  • CDN manages certificates, renewal, and cipher suites.

DDOS Protection

CDNs sit between the internet and your origin. That positioning makes them ideal for absorbing attacks:

  • Volume-based attacks — A 100Gbps attack against your origin becomes 100GB spread across the CDN's 200+ POPs. Each POP handles a fraction.
  • Protocol attacks (SYN flood, etc.) — Edge nodes can validate and drop malformed packets before they reach you.
  • Application-layer attacks — Rate limiting, CAPTCHA challenges, behavioral analysis.

Cloudflare, Akamai, and Fastly all built their reputations partly on surviving attacks that would have crushed unprotected origins.

Origin Shielding

Even with a CDN, you still need to protect the origin. Origin shielding means requests from CDN POPs go through a "shield" node before hitting origin.

Multiple CDN POPs in a region share one shield. If 100 users request the same uncached file, only one request goes to origin. The shield serves all 100 POPs.

This reduces origin load dramatically for popular content.

Streaming and Media Delivery

For video, CDNs do more than cache. They handle:

  • Adaptive bitrate streaming — Transcode on-the-fly into multiple quality levels (HLS, DASH).
  • Chunked delivery — Video isn't one file — it's thousands of tiny chunks. The CDN prefetches chunks users will watch.
  • HLS/DASH segmentation — Break video into 2-10 second segments, each independently cacheable.

The Economics

CDNs charge by data transfer. But the cost structure creates interesting incentives:

  • Bandwidth savings — Even with caching, you pay for egress from the CDN. But it's dramatically cheaper than origin bandwidth.
  • Request pricing — Some CDNs charge per request, not just data. High-cache-hit ratios minimize this.
  • Origin savings — Your origin server can be smaller, cheaper, more specialized. The CDN handles the heavy lifting.

What's Missing from Most CDN Explanations

Everyone talks about caching. But the less-discussed aspects matter more for advanced use:

  • Stale-while-revalidate — Serve stale content while fetching fresh in the background. Users don't wait.
  • Stale-if-error — Serve cached content even when the origin returns an error.
  • Early hints — CDN sends hints to browser about what to pre-load before the full response is ready.
  • HTTP/2 and HTTP/3 push — Multiplexing and 0-RTT connection establishment at the edge.

Bottom Line

A CDN is no longer optional infrastructure. It's the edge layer of the internet. It caches, routes, encrypts, protects, computes, and optimizes. The origin is now just one server among many — the CDN is where most of the internet actually lives.

The next evolution is already here: edge databases (Cloudflare D1, Turso), edge AI inference, and persistent edge storage. The origin server may eventually become just a source of truth, not a serving endpoint.
