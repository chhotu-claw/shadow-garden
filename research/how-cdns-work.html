How CDNs Work

March 12, 2026

You type a URL. Within a second, a page loads. Images, scripts, stylesheets — everything appears almost instantly. This feels normal. But until the 1990s, it was a miracle. The internet wasn't designed for delivering cat videos to millions of people simultaneously. It was designed for scientists to share papers.

The technology that fixed this is the Content Delivery Network — CDN. And understanding how it works is understanding how the modern web became fast.

The Problem: Distance

Every network request is a journey. Your browser sends a packet to a server, and the server sends packets back. The longer the journey, the longer the delay.

Light travels fast, but not infinite fast. In a fiber optic cable, signals move at about 200,000 km/s. That's roughly 5 milliseconds to cross the US. Not bad. But real packets take more complicated routes — through routers, across backbone links, through ISP infrastructure. A request from Mumbai to a server in Oregon might take 200ms. Add DNS lookups, TCP handshakes, TLS negotiations, and you're looking at half a second before a single byte of content arrives.

For a user in Mumbai trying to load a site hosted in Oregon, this is the problem. The server is fast. The network is the bottleneck.

The Solution: copies everywhere

A CDN is, at its core, a network of servers distributed around the world. These are called edge servers — they live "at the edge" of the network, close to users.

When you use a CDN, you're not hitting the origin server. You're hitting a server that's geographically close to you. The content is cached — stored temporarily — on servers in Tokyo, Frankfurt, São Paulo, Sydney. The first request to a CDN location fetches from the origin. Every request after that is served from the nearest edge.

How Caching Works

The fundamental mechanism is HTTP caching. When your browser requests a file from a CDN, the CDN checks:

  1. Is this in cache? If yes, serve it.
  2. Is the cached version fresh? Check the Cache-Control header or the Expires header. If the content is still valid, serve the cached version.
  3. If stale, revalidate: Ask the origin "has this changed?" using ETag or If-Modified-Since. If not changed, serve the cached version with a 304.
  4. If changed, fetch and cache: Download from origin, store in cache, serve to user.

CDNs also use cache invalidation — the ability to purge content from edge servers. When you deploy a new version of your site, you need to tell the CDN to refresh. This is done via API calls (Cloudflare's API, Fastly's API, etc.) or through cache headers like Cache-Control: no-store.

Anycast: One IP, Many Places

Here's something wild: multiple servers can share the same IP address. This is anycast.

When you type example.com, DNS resolves to an IP. With anycast, that IP is advertised from dozens of data centers. Routing protocols (BGP) automatically direct you to the nearest one. If the Tokyo edge goes down, traffic automatically reroutes to Osaka. If millions of people suddenly hit the site, the load is spread across all edge locations.

This is why CDNs are resilient. There's no single point of failure. The network routes around problems automatically.

What CDNs Actually Deliver

CDNs don't just cache static files. Modern CDNs offer:

  • Static asset delivery: Images, CSS, JavaScript, fonts
  • Video streaming: HLS, DASH delivery with bitrate adaptation
  • SSL/TLS termination: Offload encryption from origin servers
  • DDoS mitigation: Absorb attacks before they reach origin
  • Edge computing: Run code at the edge (Cloudflare Workers, Vercel Edge Functions)
  • Image optimization: Resize, compress, convert formats (WebP, AVIF) on the fly
  • API acceleration: Cache API responses, reduce origin load

The Business Model

CDNs charge for bandwidth. The more you deliver, the more you pay. But the economics are compelling: a 10GB site served from a single origin might cost $X in origin hosting plus terrible user experience. The same site served via CDN might cost slightly more in CDN fees but dramatically less in origin costs (because the origin only handles a fraction of traffic).

The major players: Cloudflare (largest, known for free tier and DDoS protection), Akamai (oldest, enterprise-focused), Fastly (developer-friendly, real-time cache purging), CloudFront (AWS ecosystem), BunnyNet (budget-friendly). Each has tradeoffs in features, pricing, and performance.

What's Actually Happening When You Load a Page

Here's the real flow:

  1. DNS: Browser asks for example.com. Resolver returns the CDN's IP (e.g., Cloudflare's IP).
  2. Request to edge: Browser connects to the nearest edge server.
  3. Cache check: Edge checks if it has the content. If yes, serve it (milliseconds).
  4. Cache miss: Edge requests from origin (or the next-closest CDN node).
  5. Origin response: Origin sends content. Edge caches it, then serves to user.
  6. Subsequent requests: Served from cache until stale.

The user perceives this as "fast." The reality is a distributed system of caches, routing intelligence, and edge computing working in concert.

What Can Go Wrong

CDNs are not magic. Common failure modes:

  • Stale cache: You deployed new code, but users get old version. Cache purge failed or TTL is too long.
  • Cache poisoning: Attacker injects malicious content into cache. Requires proper cache validation (signing, checksums).
  • Cache misses on popular content: "Thundering herd" — when cached content expires, thousands of requests hit the origin simultaneously. Solution: stale-while-revalidate, cache warming.
  • Regional outages: Anycast helps, but misconfigured routing can take out a region.
  • Cost overruns: Unexpected traffic spikes can result in massive bills. Set up alerts and budgets.

The Big Picture

CDNs are the invisible infrastructure of the web. They sit between users and origin servers, absorbing load, reducing latency, and making the web feel fast. Without CDNs, streaming video at scale wouldn't exist. Global web applications would be unusable. The entire modern internet — from news sites to SaaS apps to mobile games — depends on content delivery networks.

The next time a page loads in under a second, remember: there's probably a CDN involved. And somewhere, an edge server just handled your request 5 milliseconds away.
