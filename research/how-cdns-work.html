How CDNs Work

March 20, 2026

CDNs are everywhere. You've used one. Your site probably uses one. But "using a CDN" is vague — it could mean edge caching, TLS termination, DDoS mitigation, or all of the above. Let's untangle what they actually do.

The Basic Problem

Origin servers are usually in one location. Users are everywhere. Distance = latency. A user in Tokyo fetching from a server in Virginia is doing round-trips of 200+ms just to establish connection, before any data moves.

CDNs solve this by putting servers closer to users. But "putting servers everywhere" is just the start — the interesting part is what happens at each edge.

Edge Caching

The most obvious CDN feature: caching. When a user requests a file, the CDN checks its edge cache. Hit? Serve from the nearest edge node (fast). Miss? Fetch from origin, serve to user, and also store in cache for next time.

Cache invalidation is the hard part. CDNs solve this with:

  • TTL (Time To Live) — how long to cache. Set by the origin via Cache-Control headers.
  • Cache purging — API to manually evict specific URLs or patterns.
  • Versioned URLs — the hack: change the URL (e.g., /v2/app.js) to force a fresh fetch.

The trade-off: cache hit rate vs. stale content. Higher TTL = better performance, worse freshness.

TLS Termination

When you use a CDN, the TLS handshake happens at the edge, not your origin. User → edge (TLS) → origin (plain HTTP or TLS).

This means:

  • Your origin never sees a raw TCP connection — it receives proxied requests.
  • CDNs manage certificates for you (Let's Encrypt, automated renewal).
  • Latency drops because the expensive crypto handshake ends closer to the user.

But there's a trust implication: the CDN can see all your traffic. That's not nothing.

DNS Resolution

CDNs use Anycast: multiple servers share the same IP address. DNS returns that IP, and the network routes the user to the closest node based on BGP paths.

This is why changing DNS to use Cloudflare makes "the internet feel faster" — you're not just using Cloudflare's cache, you're getting routed to a nearby edge node for everything, not just cached content.

DDoS Mitigation

This is where Cloudflare and similar shine. They absorb volumetric attacks (DNS amplification, SYN floods) by:

  • Capacity — Tbps of edge bandwidth that would crush a single origin.
  • Rate limiting — blocking IPs or patterns that exceed thresholds.
  • Challenge pages — CAPTCHA or JS challenges for suspicious traffic.
  • Bot detection — fingerprinting clients, behavioral analysis.

The dirty secret: DDoS protection is also a data collection mechanism. To filter traffic, they need to inspect it. If you want maximum privacy, self-hosting might look better — but then you're exposed.

What CDNs Don't Do

CDNs are not magic. They don't fix:

  • Slow origin — if your API behind the CDN is slow, caching helps only for cacheable responses.
  • Dynamic content — personalized responses can't be cached the same way.
  • Bad architecture — a monolithic app behind a CDN is still a monolith.

The Consolidation Problem

Most of the internet now routes through a handful of CDNs. Cloudflare alone handles something like 20%+ of global web traffic. That's incredible concentration — and it means a CDN outage takes down a huge chunk of the web.

There's no easy answer. The efficiency gains are real. But we've traded origin reliability for edge reliability, and the edge is shared infrastructure.

The takeaway: CDNs are proxy layers that handle caching, TLS, DNS, and attack mitigation at the edge. They're not optional for any serious site — but understanding what they do (and don't do) matters. The privacy trade-off is real, and the consolidation is a systemic risk.
