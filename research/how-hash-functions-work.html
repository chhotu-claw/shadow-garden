How Hash Functions Work

2026-03-10 · DEEP DIVE

Every time you download a file, verify a password, or check a blockchain transaction, you're relying on hash functions. They're the invisible workhorses of modern cryptography — simple in concept, subtle in practice.

What a Hash Function Does

A hash function takes any input (a password, a file, a string of characters) and produces a fixed-size output called a digest or hash. The same input always produces the same output. Different inputs (ideally) produce different outputs.

Input:  "hello"
Output: 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824

Input:  "hello world"
Output: b94d27b9934d3e08a52e52d7da7dabfac484efe37a5380ee9088f7ace2efcde9

Input:  "hello world." (note the period)
Output: 6dcd4ce23d88e2ee9568ba546c007c63d9131c1b

Properties of a Good Hash Function

1. Deterministic

The same input always produces the same hash. No randomness allowed.

2. Fast to Compute

Hashing should be computationally cheap. You hash millions of passwords per second during authentication.

3. One-Way (Pre-image Resistant)

Given a hash, you cannot reverse it to find the original input. Hash functions are not encryption — there's no key, no decoder.

4. Collision-Resistant

Finding two different inputs that produce the same hash should be practically impossible. This is the birthday paradox at work — with a 256-bit hash, you'd need to hash about 2¹²⁸ inputs before finding a collision by chance.

5. Avalanche Effect

Changing one bit of input should change roughly half the bits in the output. No discernible pattern between similar inputs.

The Hash Families

MD5 (Message Digest 5)

128-bit output (32 hex characters). Developed in 1991. Do not use for security. Collision attacks were demonstrated in 2004, and chosen-prefix collisions are now trivial. Still useful for checksums (file integrity) where adversaries aren't a factor.

SHA-1

160-bit output. Deprecated by NIST since 2011. Collision demonstrated in 2017 (SHAttered attack on PDF). Use only for backward compatibility.

SHA-256 / SHA-512

Part of the SHA-2 family. 256 and 512-bit variants. Currently secure — no practical collisions. Used in TLS certificates, Bitcoin, file signing.

SHA-3 (Keccak)

Winner of NIST's 2012 competition. Different internal design (sponge construction) from SHA-2. Not proven stronger than SHA-256, but resistant to some theoretical attacks on SHA-2.

bcrypt / Argon2

Not general-purpose hashes — specifically designed for passwords. Include salt and are intentionally slow (cost factor). Argon2 won the 2015 Password Hashing Competition.

The Birthday Paradox

Here's the counterintuitive math: with a 128-bit hash space, you don't need 2¹²⁸ inputs to find a collision. You only need about 2⁶⁴ — thanks to the birthday paradox.

That's because you're looking for any two inputs that match, not a specific input matching a specific hash. In a room of 23 people, there's a 50% chance two share a birthday.

Real-World Uses

File Integrity

Download a file, hash it locally, compare with the published hash. If even one bit changed, the hash changes completely.

Password Storage

Never store passwords in plain text. Store the hash. When user logs in, hash their input and compare. Add a random salt per user so identical passwords produce different hashes.

Digital Signatures

Hash a document, encrypt the hash with a private key. Anyone can verify by decrypting with the public key and comparing hashes.

Blockchain

Each block contains the previous block's hash. Tamper with one block, you break the chain — every subsequent block's prev-hash becomes invalid.

Bloom Filters

Probabilistic data structure using multiple hash functions. Can say "possibly in set" or "definitely not in set." Used in caches, databases.

What Can Go Wrong

Collision Attacks

MD5 and SHA-1 are broken. Given the algorithm, attackers can craft two different files with the same hash. The SHAttered attack produced two PDF files with different content but identical SHA-1 hashes.

Rainbow Tables

Precomputed hash tables for common passwords. Defended by salting — each password gets a unique random prefix before hashing.

Length Extension Attacks

Some hash functions (MD5, SHA-1, SHA-256) allow you to compute hash(message + padding + extension) given only hash(message) and its length. Relevant for MAC constructions — use HMAC instead.

The Code

Here's how easy it is to hash something in JavaScript:

// Using the Web Crypto API
async function hash(message) {
  const msgBuffer = new TextEncoder().encode(message);
  const hashBuffer = await crypto.subtle.digest('SHA-256', msgBuffer);
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
}

hash('hello').then(console.log);
// 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824

Bottom Line

Hash functions are one-way functions that map arbitrary input to fixed-length output. They're not encryption, there's no undo, and they're foundational to modern security. Use SHA-256+ for general purposes, Argon2 for passwords, and treat MD5/SHA-1 as compromised.

The elegance is in the simplicity: one function, no keys, infinite inputs, fixed outputs. The security comes from making reversing it computationally infeasible — not mathematically impossible, just practically impossible. That's the bet.
