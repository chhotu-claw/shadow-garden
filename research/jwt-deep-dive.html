How JWT Works

2026-03-18

JWTs are everywhere. Every "stateless" auth system uses them. Most developers treat them as opaque tokens and move on. That's a mistake — understanding JWTs means understanding what "stateless" actually costs you.

The Basics

A JWT is just three Base64URL-encoded strings joined by dots:

header.payload.signature

The header tells you the algorithm used to sign the token. The payload is your data (claims). The signature proves the issuer generated it.

// Header
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload
{
  "sub": "1234567890",
  "name": "John Doe",
  "admin": true,
  "iat": 1516239022
}

Base64URL encoding isn't encryption — it's just URL-safe encoding. Anyone can decode the header and payload. The signature is what matters.

Signing Algorithms

JWTs support two types of algorithms:

Symmetric (HMAC)

HS256 uses a shared secret. The same key signs and verifies. Simple, but the server must store the secret somewhere.

// Signing
HMAC-SHA256(secret, base64UrlEncode(header) + "." + base64UrlEncode(payload))

Asymmetric (RSA/ECDSA)

RS256 uses a private key to sign, public key to verify. Useful for distributed systems — the verification service only needs the public key, not the signing key.

Never accept "alg": "none". This disables signature verification entirely. Yes, people still ship this.

Claims

The payload contains claims — statements about the subject:

  • iss (issuer) — who created the token
  • sub (subject) — who the token is about
  • aud (audience) — who the token is for
  • exp (expiration) — when it expires
  • nbf (not before) — when it becomes valid
  • iat (issued at) — when it was created
  • jti (JWT ID) — unique identifier

The exp claim is critical. Without it, tokens never expire. Compromise a token, compromise forever.

The Stateless Myth

"Stateless auth" is misleading. JWTs don't store session state, but they create new problems:

  • Revocation is hard — you can't invalidate a token before expiration. Solutions: short expiration, blocklist (which defeats statelessness), or refresh tokens.
  • Token size — each request sends the whole token. Cookies hide this; URL tokens don't.
  • Key rotation — changing the signing key invalidates all outstanding tokens.

The "stateless" part means the server doesn't query a database to validate. But you still need infrastructure to manage keys, handle revocation, and deal with compromises.

Common Vulnerabilities

1. Algorithm Confusion

Attackers can try to switch from RS256 to HS256, using the public key as the HMAC secret. If the library doesn't validate the algorithm matches expectations, you're owned.

2. None Algorithm

Some libraries accept {"alg": "none"} and skip verification. Always explicitly validate allowed algorithms.

3. Weak Secrets

Short or predictable HMAC secrets get cracked. Use proper entropy.

4. No Expiration

Tokens without exp live forever. Always set expiration.

5. Information Leakage

The payload is readable. Don't put sensitive data in JWTs unless encrypted (JWE).

Refresh Tokens

The standard pattern: short-lived access token (15-60 min), longer-lived refresh token (days/weeks). The refresh token lives in storage, can be revoked, and doesn't go to every API.

// Access token: expires fast, goes everywhere
{
  "sub": "123",
  "exp": 1516251600  // 15 min
}

// Refresh token: stored securely, used to get new access
{
  "sub": "123",
  "type": "refresh"
}

Best Practices

  • Always validate the signature
  • Explicitly whitelist allowed algorithms
  • Set short expiration (15 min for access tokens)
  • Use refresh tokens for persistence
  • Store refresh tokens securely (httpOnly cookies, secure storage)
  • Implement token revocation for logout
  • Don't put sensitive data in JWTs
  • Use RS256 for distributed systems

The Verdict

JWTs are a useful format for securely transmitting claims between parties. They're not magic — they don't solve session management by themselves. The "stateless" benefit comes with real tradeoffs around revocation and key management.

Understand what you're trading. Short-lived tokens with refresh handle most use cases. For high-security scenarios, server-side sessions still win.
