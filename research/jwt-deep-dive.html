JWT Deep Dive

2026-04-02

JWTs (JSON Web Tokens) are everywhere. Every auth system, every API, every "stateless" service uses them. But how do they actually work? And more importantly — where do they break?

The Anatomy

A JWT looks like three base64url strings separated by dots:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

That's three parts:

  • Header — declares algorithm and token type
  • Payload — the actual data (claims)
  • Signature — proves it wasn't tampered with

The Header

{"alg": "HS256", "typ": "JWT"}

HS256 means HMAC-SHA256. There's also RS256 (RSA), ES256 (ECDSA), and none (unsigned, which is dangerous).

The Payload

This is where you put your claims. Three types:

  • Registered — iss (issuer), exp (expiration), sub (subject), aud (audience), iat (issued at)
  • Public — namespaced claims like https://shadow.garden/roles
  • Private — whatever you want

The Signature

This is where it gets interesting. The signature is computed as:

HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)

Here's the kicker: the signature only verifies that YOU created it. It doesn't verify that the data is trustworthy. If I steal your secret key, I can forge any token. And if you use algorithm "none" (yes, some libraries allow this), there's no signature at all.

Security Gotchas

1. Algorithm Confusion

In 2017, Auth0 disclosed a vulnerability where an attacker could switch the algorithm from RS256 (asymmetric) to HS256 (symmetric) and sign tokens using the public key as the secret. If your verification code trusts the "alg" header, you're pwned.

2. None Algorithm

Some JWT libraries accept "alg: none" as valid. Attackers exploit this by stripping the signature entirely. Always explicitly list allowed algorithms.

3. Weak Secrets

Brute-forcing "secret", "password", or "123456" on HS256 is trivial. Use strong, random secrets. Better yet, use asymmetric algorithms (RS256, ES256).

4. No Expiration

If your payload doesn't include an exp claim, the token lives forever. Always set expiration.

5. Information Leakage

The payload is just base64 — anyone can decode it. Never put sensitive data (passwords, PII) in a JWT. It's not encryption, it's just encoding.

Why They Suck

JWTs are popular because they're stateless — the server doesn't need to store session data. But this is also their weakness:

  • You can't revoke them instantly (need a blocklist/denylist)
  • They're opaque to the client — can't see what's inside without decoding
  • They grow with every claim you add

The Verdict

JWTs are fine for short-lived, stateless authentication between services you control. They're terrible for user sessions where you need instant revocation. Use them wisely, verify carefully, and never trust the algorithm header.
