How OAuth Works

March 9, 2026

Every time you click "Login with Google" or "Connect with GitHub," OAuth is working behind the scenes. It's the standard that lets you prove who you are to a service without handing over your password. But OAuth is famously confusing — the flow involves redirects, tokens, and actors with confusing names. Let's untangle it.

The Problem OAuth Solves

Before OAuth, the only way to authorize a third-party app was to give it your password. This was terrible: the app got full access to your account, you couldn't revoke just that app, and if the app was compromised, your whole account was compromised.

OAuth introduces a middleman: the authorization server. Instead of giving your password to the app, you give the app a token. The token is scoped (can only do specific things), expiring (must be refreshed), and revocable (kill the token, kill the access).

The Actors

OAuth has four roles:

  • Resource Owner — That's you. The user.
  • Client — The app wanting access (the "third-party app").
  • Authorization Server — The trusted party that issues tokens (Google, GitHub, your own auth server).
  • Resource Server — The API that holds your data and accepts tokens.

The Flow (Authorization Code Grant)

This is the most common and most secure OAuth flow. Here's what happens:

1. User clicks "Login with Google"

The client redirects the browser to Google's authorization server with:

  • client_id — identifies the app
  • redirect_uri — where Google sends the user back
  • scope — what permissions the app wants
  • state — a random string to prevent CSRF

2. User logs in and consents

The user sees a Google login screen (if not already logged in) and a consent screen listing what the app is asking for. If they approve, Google redirects back to the redirect_uri with a temporary authorization code.

3. App exchanges code for tokens

The app takes that authorization code and makes a back-channel request (server-to-server) to Google to exchange it for an access token and optionally a refresh token. This happens in code the user never sees.

4. App uses the access token

Now the app can make API requests to Google (the resource server) by including the access token in the Authorization: Bearer header. The resource server validates the token and returns the data.

5. Refresh (optional)

Access tokens expire (usually in an hour). The app uses the refresh token to get a new access token without bothering the user.

Why This Is Secure

  • Password never leaves the user — the app never sees your Google password.
  • Temporary code — the authorization code can only be used once and expires quickly.
  • Back-channel exchange — the token exchange happens server-to-server, not in the browser where it could be intercepted.
  • Scoping — you can limit what the token can do (read-only vs full access).

OAuth 2.0 Grants

Authorization Code is the gold standard for web apps. But there are others:

  • Implicit Flow — (Deprecated) Tokens returned directly to browser. Too vulnerable to token leakage.
  • Client Credentials — Machine-to-machine, no user involved (e.g., a background job accessing its own data).
  • Device Code — For smart TVs and CLI apps — shows a code on the device, user enters it on another device.

OpenID Connect (OIDC)

OAuth is for authorization (what can you do?). If you also need authentication (who are you?), you layer OpenID Connect on top. OIDC adds an id_token that contains user identity info (name, email, profile picture). "Login with Google" typically uses OIDC.

What Could Go Wrong

  • Redirect URI manipulation — attacker registers a malicious redirect URI. Mitigated by requiring exact URI matches.
  • CSRF attacks — attacker tries to bind your account to their OAuth session. Mitigated by the state parameter.
  • Token leakage — access tokens in URLs get logged in server logs. Use POST body or headers instead.
  • Weak client secrets — if the client secret leaks, attacker can impersonate the app. Use PKCE (Proof Key for Code Exchange) for public clients.

PKCE: The Modern Standard

PKCE (pronounced "pixie") adds cryptographic proof that the app that started the flow is the same app finishing it. It replaces the client secret for mobile and SPA apps where there's no server to keep a secret. If you're building a new OAuth app, use Authorization Code + PKCE.

OAuth 2.0 is authorization. OpenID Connect is authentication built on OAuth 2.0. Most "Login with X" uses both.

Summary

OAuth 2.0 solved the password-sharing problem by introducing a token-based middleman. The user authorizes an app once, gets a temporary code, the app exchanges it for a token, and the token is used for subsequent requests. It's not perfect — the spec is complex, security pitfalls abound — but it's the backbone of modern API security.
