How OAuth Works

The authorization framework behind "Login with Google" — explained

You've clicked "Login with Google" a thousand times. But what actually happens? OAuth 2.0 is the answer — a framework that lets your app ask another service "can I get this user's info?" without ever seeing their password.

The Core Problem

Before OAuth, the ugly truth was: if you wanted to let users log in with their Google account, you'd ask them for their Google password. That's insane. You're giving your app access to everything — email, contacts, photos, the works — and if your app gets hacked, the attacker gets their Google account too.

OAuth solves this by introducing a middleman: the authorization server. Instead of you handling credentials, the user authenticates with Google, Google issues a token to your app, and that token is scoped to exactly what you asked for.

Actors in the Drama

  • Resource Owner — That's the user. They "own" their data (photos, emails, whatever).
  • Client — Your app. The thing that wants access.
  • Authorization Server — Google's login page, GitHub's login page. The thing that issues tokens.
  • Resource Server — The API that actually serves the protected data (Google Photos API, GitHub API).

The Flows

1. Authorization Code Flow (The Gold Standard)

This is what "Login with Google" actually uses. It's the most secure flow — the access token never lives in the browser.

User clicks "Login with Google"
→
Redirect to Google with client_id & redirect_uri
→
User enters password on Google's page
→
Google redirects back with code
→
Your server swaps code for access_token

The key insight: the code is not the token. It's a one-time-use coupon. Your server exchanges it for the real token, which Google issues directly to your server (not the browser). This prevents token leakage via browser history or logs.

2. PKCE (Proof Key for Code Exchange)

PKCE adds a layer of security for public clients — mobile apps and SPAs that don't have a server to keep a client secret.

Client generates random code_verifier
→
Send code_challenge (hash of verifier) to auth server
→
Auth server stores code_challenge
→
Client proves it has code_verifier when exchanging code

Even if someone intercepts the code, they can't use it without the code_verifier. PKCE is now mandatory for public clients in OAuth 2.1.

3. Implicit Flow (Don't Use This)

Historically, SPAs used the implicit flow: the token came back in the URL fragment directly. This is dead now — too many ways to leak tokens. If you see it in the wild, it's legacy code that needs updating.

The Tokens

Access Token

The real key. Short-lived (usually an hour), gets sent to the API with every request:

Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

Refresh Token

The spare key. Long-lived, used to get new access tokens without bothering the user. If your refresh token leaks, an attacker can impersonate your user indefinitely — which is why good implementations rotate refresh tokens (issue a new one each time you use it).

ID Token

This is the JWT that proves who the user is. It contains claims about the user — their ID, name, email, profile picture. It's meant for your app to read, not to use as an API credential.

Token Storage

Access tokens go in memory (never localStorage — XSS can grab them). Refresh tokens go in secure, httpOnly cookies if you have a backend, or encrypted storage on mobile. The browser's sessionStorage is acceptable for access tokens as long as they're short-lived.

Scopes: What You're Asking For

Scopes are the permissions you request. Google's look like profile, email, https://www.googleapis.com/auth/calendar.readonly.

The principle of least privilege: only ask for what you need. If you just need their email, don't ask for calendar access. Users see the permissions requested and increasingly deny apps that ask for too much.

Security Pitfalls
  • Never trust the alg field in JWTs. Set it explicitly when verifying. The "alg:none" attack is real.
  • Redirect URI validation is critical. If you accept arbitrary redirect URIs, attackers canphish for tokens.
  • State parameter is mandatory. Without it, CSRF attacks can trick users into authorizing attackers' accounts.
  • Short-lived access tokens. One hour is standard. If you need longer, use refresh tokens — they can be revoked.

OpenID Connect (OIDC)

OAuth 2.0 is for authorization (what can you do?). OpenID Connect adds an identity layer on top — it's for authentication (who are you?).

When you "Login with Google", you're actually using OIDC. The id_token is the OIDC addition — it's a signed JWT that contains verified user claims. Your app uses it to know the user's email, name, and picture.

Why This Matters

OAuth is everywhere. Every "Login with X" button, every "Connect your Spotify" feature, every app that reads your GitHub repos — all of them use OAuth. Understanding the flows helps you:

  • Debug auth issues when they break
  • Build more secure apps
  • Understand what you're actually agreeing to when you click "Allow"

The framework is fifteen years old and still the standard. That's not because it's perfect — it's because the tradeoffs are right. It separates concerns cleanly, lets users revoke access without changing passwords, and keeps your app far away from storing credentials.
