How OAuth 2.0 Works

March 10, 2026

OAuth 2.0 is the protocol behind "Sign in with Google," "Connect your Spotify account," and countless other integrations. It's not an authentication protocol — it's an authorization protocol. Understanding the difference is key.

The Core Problem

Before OAuth, apps asked for your password. You'd give your Google password to a third-party app, and it'd have full access to your account. That's terrifying. OAuth solves this by introducing a middleman: the access token.

Four Actors

  • Resource Owner — You. The person who owns the data.
  • Client — The app wanting access (e.g., a calendar app).
  • Authorization Server — The service that grants permission (e.g., Google).
  • Resource Server — The API holding your data (e.g., Google Calendar API).

The Authorization Code Flow

The most common flow. Here's how it works:

  1. User clicks "Sign in with Google" — The app redirects to Google's authorization server with a client_id, redirect_uri, and requested scopes.
  2. User logs in and consents — Google asks "Allow this app to read your calendar?" The user says yes.
  3. Authorization server redirects back — Google sends an authorization code to the app's callback URL (not a token!).
  4. App exchanges code for tokens — The app POSTs the code + its client_secret to Google's token endpoint.
  5. Authorization server returns tokens — Access token + refresh token.
  6. App uses access token — Requests data from the resource server using the access token.

Why the Code Exchange Matters

The access token never travels through the user's browser. Only the authorization code does — and that code is useless without the client_secret (which stays on the server). This prevents token leakage via browser history, logs, or inspect element.

Access Tokens vs Refresh Tokens

  • Access token — Short-lived (usually minutes to hours). Used to make API calls. If leaked, damage is limited by expiration.
  • Refresh token — Long-lived (weeks/months). Used to get new access tokens. Stored securely server-side.

PKCE: The Mobile App Fix

Native apps (mobile, SPAs) can't safely store client_secret — the app binary is public. PKCE (Proof Key for Code Exchange) solves this:

  1. App generates a random code_verifier (secret).
  2. App sends code_challenge (hash of verifier) to auth server.
  3. Auth server stores challenge, returns code.
  4. App sends verifier in token exchange.
  5. Auth server verifies and issues tokens.

Even if an attacker intercepts the code, they can't use it without the verifier.

Common Security Pitfalls

  • Implicit flow — Returning tokens directly in URL fragment. Deprecated. Use authorization code + PKCE instead.
  • Weak redirect_uri validation — Attackers register similar domains. Validate strictly.
  • Not validating tokens — Always verify token signature, audience, issuer, and expiration.
  • Scope explosion — Requesting more permissions than needed. Stick to minimum necessary.

OpenID Connect (OIDC)

OAuth 2.0 is for authorization. OIDC builds on it builds add authentication, adding an id_token that contains user identity info (claims). "Sign in with Google" typically uses OIDC, not raw OAuth.

The Bottom Line

OAuth 2.0 is about delegation without password sharing. The complexity exists because security is hard. The key insight: never let tokens touch the client-side unless absolutely necessary, always use PKCE for public clients, and treat access tokens as ephemeral.
