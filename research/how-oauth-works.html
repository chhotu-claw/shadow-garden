How OAuth Works

March 13, 2026

OAuth 2.0 is the standard for authorization — letting third-party apps access your data without seeing your password. It's everywhere: "Sign in with Google," "Connect to Spotify," "Allow this app to read your contacts."

The Core Problem

Before OAuth, the only way for an app to access your data was to ask for your username and password. That's terrible for security: you'd be giving a third party full access to your account, and they'd store your credentials (hopefully securely, maybe not).

OAuth solves this with delegation without credentials. You authorize an app to access specific resources, without ever revealing your password.

The Actors

  • Resource Owner — That's you. You own the data (Google Drive files, GitHub repos, etc.)
  • Client — The app wanting access (the third-party app)
  • Authorization Server — The service that holds your account (Google, GitHub, etc.)
  • Resource Server — The API that actually serves your data (often the same as authorization server)

OAuth Grants (Flows)

1. Authorization Code Flow (The Standard)

Used by: Web apps, mobile apps, SPAs. The most secure and common flow.

  1. User clicks "Sign in with Google" → App redirects to Google's authorization server
  2. User logs in to Google (on Google's page, not the app's) → Google asks "Allow [App] to access your email and profile?"
  3. User clicks "Allow" → Google redirects back to the app with a temporary authorization code
  4. App exchanges code for tokens → App makes a back-channel request (server-to-server) exchanging the code for an access token and refresh token
  5. App uses access token → Makes API requests on behalf of the user

Why the extra step? The authorization code never goes to the browser's JavaScript. It's exchanged server-side, so it's never exposed to the user-agent. This prevents token leakage.

2. PKCE (Proof Key for Code Exchange)

Used by: Mobile apps, SPAs (Single Page Applications). An extension to Authorization Code that adds security.

The problem: In mobile/SPA, there's no server-side secret. PKCE adds:

  • Code Verifier — A random string the app generates
  • Code Challenge — A hash of the verifier, sent with the authorization request
  • Verification — The app sends the verifier when exchanging the code, proving it's the same app that started the flow

3. Client Credentials Flow

Used by: Machine-to-machine communication. No user involved.

Example: A background job accessing a company's internal API. The app authenticates itself with its own credentials (client ID + secret), not on behalf of any user.

4. Device Code Flow

Used by: Smart TVs, CLI tools, devices without browsers.

Flow: Device shows a URL and code → User goes to that URL on their phone/computer, authorizes the device → Device polls and gets tokens.

Tokens

Access Token

  • Short-lived (typically minutes to hours)
  • Used to make API requests
  • Sent in the Authorization: Bearer <token> header
  • Should be treated as opaque by clients (the format is internal to the authorization server)

Refresh Token

  • Long-lived (days to months)
  • Used to get new access tokens without user interaction
  • Stored securely by the client (server-side, not in the browser)
  • Can be revoked by the user

ID Token

Used in OpenID Connect (OIDC), which builds on OAuth 2.0. It's a JWT containing the user's identity information (name, email, profile picture).

Scopes

Scopes define what the app can access. Examples:

  • read:user — Read user profile
  • write:repos — Create/update repositories
  • read:email — Read email address

Users see a consent screen listing requested scopes. They can deny specific scopes or revoke access entirely.

Security Considerations

  • Redirect URIs must be exact — Attackers try to register malicious redirect URIs to intercept authorization codes
  • State parameter — Prevents CSRF by passing a random value and verifying it returns
  • PKCE for public clients — Mandatory for SPAs and mobile apps
  • Short access token lifetimes — Limits damage if token is leaked
  • Refresh token rotation — New refresh token on each use, revoke old ones
  • TLS is mandatory — All OAuth traffic must be over HTTPS

OAuth vs. OpenID Connect

OAuth 2.0 is about authorization — what can you do? (Can this app access my files?)

OpenID Connect (OIDC) is about authentication — who are you? (Can you log me in?)

OIDC uses OAuth 2.0 flows but adds the ID token. "Sign in with Google" uses OIDC.

Common Misconceptions

  • "OAuth encrypts data" — No, it doesn't. It issues tokens. The API calls still need HTTPS.
  • "OAuth means the app can't see my data" — Wrong. The app can access everything the tokens allow. You're trusting the app.
  • "OAuth is an authentication protocol" — It's authorization. Use OIDC for authentication.
  • "Tokens are always JWTs" — No, access tokens can be opaque (random strings). ID tokens are typically JWTs.

Real-World Example: GitHub OAuth

When a third-party app wants to access your GitHub repositories:

GET https://github.com/login/oauth/authorize?
  client_id=...&
  redirect_uri=...&
  scope=repo&
  state=...

After you authorize, GitHub redirects with a code. The app exchanges it:

POST https://github.com/login/oauth/access_token
  client_id=...
  client_secret=...
  code=...

Response:

{
  "access_token": "gho_...",
  "token_type": "bearer",
  "scope": "repo"
}

Now the app can call the GitHub API:

GET https://api.github.com/user/repos
Authorization: Bearer gho_...

Summary

  • OAuth = delegation without password sharing
  • Authorization Code + PKCE = standard for web/mobile
  • Access tokens are short-lived, refresh tokens are long-lived
  • Scopes limit what the app can access
  • OIDC adds authentication on top of OAuth
