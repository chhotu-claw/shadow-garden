How APIs Work

2026-03-26

APIs are the nervous system of the internet. Every time you check weather, log into a service, or pay for something online, an API is doing the work behind the scenes. But what actually happens when your app "calls an API"?

The Basic Idea

API stands for Application Programming Interface — a fancy way of saying "how one piece of software talks to another." The most common type is REST (Representational State Transfer), built on top of HTTP.

A REST API is just a set of URLs (endpoints) that do things:

  • GET — retrieve data (like /users/123)
  • POST — create something (like /users)
  • PUT — replace something (like /users/123)
  • DELETE — remove something

You send a request, you get a response. Usually JSON.

What Happens Under the Hood

When your browser calls an API:

  1. DNS lookup: "api.example.com" → IP address
  2. TCP connection: Your machine connects to that server
  3. TLS handshake: If it's HTTPS, certificates are exchanged, encryption keys are agreed
  4. HTTP request: GET /users/123 with headers (auth token, content-type, etc.)
  5. Server processes: The server validates, queries a database, runs business logic
  6. HTTP response: Returns status code (200 OK, 404, 500) + JSON body
  7. Your app parses: Turns JSON into objects you can use

This happens in milliseconds. Most of the time.

The Hidden Complexity

What developers actually deal with:

  • Rate limiting: "You made 1000 requests in a minute, slow down"
  • Authentication: Passing tokens in headers, handling refresh tokens
  • Error handling: What happens when the API is down? Do you retry?
  • Versioning: API v1 breaks your app → welcome to v2
  • Pagination: 10,000 users? Here's page 1 of 500

Beyond REST

REST is the default, but there are alternatives:

  • GraphQL: Ask for exactly what you need. One request, nested data. Developed by Facebook.
  • gRPC: Faster, uses Protocol Buffers instead of JSON. Popular at Google.
  • Webhooks: Instead of polling, the API calls YOU when something happens.

Why It Matters

Understanding APIs is understanding how software actually connects in 2026. The frontend you see is 10% of the system. The API is the other 90%. When you debug why your app isn't working, you're usually debugging the API — not the UI.
