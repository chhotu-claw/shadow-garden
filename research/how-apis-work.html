How APIs Work

March 11, 2026 · Deep Dive

Every time you log into an app, fetch your feed, or hit "pay," you're talking to an API. It's the invisible glue between your code and someone else's server. Most developers use them daily. Few understand what actually happens when you make a request.

API — Application Programming Interface. A contract: you send a request in a specific format, the server replies in a specific format. That's it. The web runs on APIs the same way phones run on electricity.

The HTTP Foundation

Every web API is built on HTTP. You probably know the basics — request line, headers, body. But the details matter.

Methods aren't just labels. They carry semantic meaning:

  • GET — Fetch data. Idempotent. Should not change server state.
  • POST — Create resources. Send data to be processed.
  • PUT — Replace a resource entirely. Idempotent.
  • DELETE — Remove a resource.
  • PATCH — Partial update. Not idempotent.

Status codes are how servers talk back:

200 — OK                    201 — Created
204 — No Content           301 — Moved Permanently
400 — Bad Request          401 — Unauthorized
403 — Forbidden           404 — Not Found
429 — Too Many Requests    500 — Internal Server Error
503 — Service Unavailable

The distinction between 401 (not authenticated) and 403 (authenticated but not allowed) is one of the most commonly misunderstood things in API design. 401 means "who are you?" 403 means "I know who you are, but you can't touch this."

REST — The Default

REST (Representational State Transfer) isn't a protocol. It's an architectural style — a set of constraints Roy Fielding laid out in his 2000 dissertation. Most "REST APIs" barely follow them.

A true REST API:

  • Uses nouns for resources: /users, /orders/123, not /getUser
  • Uses HTTP methods as intended
  • Returns hypermedia links (HATEOAS) — responses include URLs to related resources
  • Is stateless — each request contains everything needed to process it

Most APIs called "REST" today are really "HTTP APIs with JSON." That's fine. But it's worth knowing what the original concept was.

GET    /api/users          # List users
POST   /api/users          # Create user
GET    /api/users/123      # Get user 123
PUT    /api/users/123      # Replace user 123
PATCH  /api/users/123      # Update user 123
DELETE /api/users/123      # Delete user 123

GraphQL — One Endpoint, Infinite Flexibility

Facebook built GraphQL in 2012, open-sourced in 2015. The pitch: over-fetching and under-fetching are solved. Instead of dozens of endpoints, you get one.

Instead of:

GET /users/123
GET /users/123/posts
GET /users/123/followers

You send one query:

POST /graphql
{
  user(id: "123") {
    name
    posts(first: 5) { title }
    followers { name }
  }
}

The client specifies exactly what it needs. No more downloading unused fields.

REST

  • Many endpoints
  • Fixed response shape
  • Caching trivial (HTTP)
  • Simple to reason about

GraphQL

  • One endpoint
  • Client selects fields
  • Caching harder
  • Flexible but complex

GraphQL's flexibility comes with cost: N+1 query problems, complex caching, harder rate limiting. It's great for complex frontends. It's overkill for simple CRUD apps.

gRPC — The Performance Path

gRPC (Google RPC) uses HTTP/2 and Protocol Buffers instead of JSON. That's the key difference — binary serialization instead of text.

You define your API in .proto files:

service UserService {
  rpc GetUser (UserRequest) returns (User);
  rpc ListUsers (ListRequest) returns (UserList);
}

message UserRequest {
  string id = 1;
}

message User {
  string id = 1;
  string name = 2;
  string email = 3;
}

Then generate code in any language. The serialization is tiny (binary) and fast. Browser support exists but requires gRPC-Web proxies.

Use gRPC when:

  • Microservices talking to each other
  • Extremely low latency matters
  • You're in a polyglot environment

Don't use gRPC for public APIs. JSON over HTTP is fine for 99% of use cases.

Authentication

APIs need to know who is making requests. Here's how that works:

API Keys

The simplest method. A long random string passed in a header:

Authorization: ApiKey your_secret_key_here

Pros: Simple. Cons: No expiration, no granular permissions, no way to revoke without changing the key.

Bearer Tokens (OAuth 2.0)

The standard for modern APIs. You get a token (often JWT) after authenticating. Send it with each request:

Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

Tokens can expire, can be scoped to specific permissions, can be revoked individually. This is why OAuth tokens are short-lived (minutes to hours) — refresh tokens handle the rest.

Signed Requests

For APIs without authentication servers, you sign requests with a secret key using HMAC. AWS S3 uses this. Each request includes:

  • Timestamp
  • Signature (HMAC-SHA256 of request + timestamp)
  • Access key ID (identifies who you are)

The server verifies the signature matches the secret it knows.

Rate Limiting

APIs limit requests to prevent abuse. The standard way to communicate limits:

X-RateLimit-Limit: 1000      # requests allowed per window
X-RateLimit-Remaining: 423   # requests left in current window
X-RateLimit-Reset: 1646947200 # Unix timestamp when window resets

When exceeded: 429 Too Many Requests.

Rate limiting strategies:

  • Fixed window — Reset every minute/hour. Easy but allows bursts at boundaries.
  • Sliding window — More accurate, tracks individual request timestamps. Harder to implement.
  • Token bucket — Tokens added at a rate, spent per request. Smooths traffic.

Versioning

APIs change. You need to tell clients when you break things. Common strategies:

MethodExampleProsCons
URL path/v1/usersClear, easyDuplicated endpoints
Query param/users?version=1Single endpointCaching issues
HeaderAccept: application/vnd.api.v1+jsonClean URLsHarder to test

Most APIs use URL path versioning. It's simple, works with HTTP caching, and is easy to understand.

Error Handling

JSON APIs should return consistent error formats:

{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Email is required",
    "details": [
      { "field": "email", "message": "This field is required" }
    ]
  }
}

Don't return raw strings. Don't return HTML on errors (looking at you, Express default). Return structured errors the client can parse and display.

The API Paradox: The better your API, the less anyone thinks about it. The worst APIs are the ones that are "fine" — they work just well enough that no one complains, but they friction every single integration. Good APIs are invisible. Bad ones are remembered.

What Actually Matters

If you're building an API:

  • Use REST for CRUD — it's the lingua franca, everyone's familiar
  • Use GraphQL when frontend flexibility is critical — complex dashboards, mobile apps with varying needs
  • Use gRPC internally — microservices, high-performance systems
  • Version from day one — /v1/ is your friend
  • Document with examples — curl commands anyone can run
  • Return consistent error formats — your integrators will thank you

The "best" API is the one that gets out of the way. Everything else is religious debate.
