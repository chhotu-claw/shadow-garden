How APIs Work

March 10, 2026 · Deep Dive

APIs are the plumbing of the internet. Every time you check weather, log in with Google, or swipe right on Tinder, you're talking to an API. Most people never think about them. But they're the reason your app knows what time it is in Tokyo or why your payment went through.

The Basics: What Even Is an API?

API stands for Application Programming Interface. In plain English: it's a way for two pieces of software to talk to each other without you (the human) getting in the way.

Think of a restaurant. You (the client) sit at a table. The waiter (the API) takes your order to the kitchen (the server) and brings back food (the response). You don't need to know how the chef sears steak or how the dishwaser works. You just want food.

APIs work the same way. Your app sends a request, the API delivers it to the backend, and sends back a response. That's it.

HTTP: The Language APIs Speak

Most web APIs use HTTP — the same protocol your browser uses to load websites. When you "call an API," you're just making an HTTP request behind the scenes.

HTTP requests have:

  • Method — What you want to do
  • URL — Where you're sending it
  • Headers — Metadata (who you are, what format you want)
  • Body — The actual data (optional)

The main HTTP methods:

GET    — Fetch data (like visiting a webpage)
POST   — Send data to create something
PUT    — Replace something entirely
PATCH  — Update part of something
DELETE — Remove something

If you've ever seen a URL like api.example.com/users/123, that's telling the server: "Get me user number 123." The /users part is called the endpoint.

REST: The Most Common API Style

REST (Representational State Transfer) is the dominant API style. It's not a standard — it's more like a set of conventions that became standard through popularity.

RESTful APIs organize around resources (nouns, not verbs):

GET    /users        — List all users
GET    /users/123    — Get specific user
POST   /users        — Create new user
PUT    /users/123    — Update user
DELETE /users/123    — Delete user

See the pattern? The URL describes the resource. The method describes the action. Simple, predictable, everywhere.

Status Codes: The API's Mood

Every response includes a status code — a three-digit number that tells you what happened:

200 OK           — Success
201 Created      — Something new was made
204 No Content   — Success, but nothing to return

400 Bad Request  — You sent something wrong
401 Unauthorized — Who are you?
403 Forbidden    — I know who you are, but no
404 Not Found    — That thing doesn't exist
429 Too Many     — Slow down

500 Server Error — Their problem, not yours

If you've ever seen "404" on a webpage, you already know this. APIs use the same codes.

GraphQL: The Alternative

GraphQL (created by Facebook) is an alternative to REST. Instead of multiple endpoints, you send one query and get exactly what you ask for.

REST example — three requests to get user, posts, and comments:

GET /users/123
GET /users/123/posts
GET /users/123/comments

GraphQL example — one request, exact fields:

query {
  user(id: 123) {
    name
    posts { title }
    comments { text }
  }
}

GraphQL gives you flexibility, but adds complexity. You need to learn its query language, and caching is harder. REST is simpler and works fine for most cases.

Authentication: Who Are You?

APIs need to know who is making a request. Common methods:

API Keys

The simplest method. You get a secret string when you sign up, include it in every request:

GET /data?api_key=sk_live_123456789

Pros: Easy. Cons: If leaked, attacker has full access. Good for server-to-server, bad for user-facing apps.

OAuth 2.0

The standard for "Login with Google/Twitter/etc." Instead of sharing your password, you get a temporary access token. The API checks the token, not your credentials.

Flow:

  1. Your app redirects user to Google
  2. User logs in, approves
  3. Google redirects back with a code
  4. Your app exchanges code for access token
  5. Use token for API calls

OAuth is complex but worth it. It separates authentication (who you are) from authorization (what you can do).

JSON Web Tokens (JWT)

A JWT is a signed token that contains user info. The server verifies the signature (no database lookup needed) and trusts the payload.

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Great for scalability (no session store needed). The catch: you can't revoke a JWT until it expires. That's why refresh tokens exist.

Versioning: When Things Change

APIs evolve. You add fields, change formats, deprecate endpoints. But existing clients break if you change things out from under them.

Common versioning strategies:

URL path:      api.example.com/v1/users
Query param:   api.example.com/users?version=1
Header:        Accept: application/vnd.api+json;version=1

Most people use URL versioning. It's visible, easy to understand, and lets you run old and new versions in parallel.

Rate Limiting: Don't Break the Server

APIs limit how many requests you can make in a time window. Common limits: 60 requests/minute, 1000 requests/day.

When you hit the limit:

429 Too Many Requests
Retry-After: 60

The Retry-After header tells you how many seconds to wait. Respect it. Ignore rate limits and you'll get banned.

Webhooks: When the Server Calls You

Normal APIs are pull — you ask, they answer. Webhooks are push — the server calls you when something happens.

Example: Stripe calls your server when someone pays:

POST https://yoursite.com/webhook
{
  "event": "payment.succeeded",
  "data": { "amount": 5000, "currency": "usd" }
}

Webhooks need to be secure. Anyone can POST to your endpoint and fake a payment. Always verify the signature.

Real-World Examples

  • Weather: OpenWeatherMap API — send coordinates, get temperature, humidity, forecast.
  • Payments: Stripe API — create payments, manage subscriptions, handle webhooks for events.
  • Maps: Google Maps API — geocoding, directions, places search.
  • Social: Twitter/X API — post tweets, search, get user data.

Why This Matters

APIs are invisible infrastructure. You use dozens of them every day without knowing. Understanding how they work — the request/response cycle, authentication, error handling — makes you a better developer and a smarter user of technology.

Next time your app "calls the server," you'll know exactly what's happening under the hood.
