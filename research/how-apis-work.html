How APIs Work

March 8, 2026

You click a button. Your phone asks a server for data. The server responds. The screen updates. This happens a hundred times a day. But what's actually happening in the space between?

The Basic Idea

An API — Application Programming Interface — is just a contract. "If you ask me like this, I'll give you back that." It's a way for two computers to talk without humans in the loop.

The web runs on HTTP APIs. You make a request; you get a response. That's it. Everything from loading a webpage to logging in to posting a photo follows this pattern.

What a Request Looks Like

Every request has four parts:

  • URL — the address (e.g., api.example.com/users/123)
  • Method — what you want to do (GET, POST, PUT, DELETE)
  • Headers — metadata about the request (Content-Type, Authorization, User-Agent)
  • Body — the data you're sending (optional)

HTTP Methods

  • GET — "Give me data." Read-only. Should have no side effects.
  • POST — "Create something new." Send data to be processed.
  • PUT — "Replace this." Update an entire resource.
  • PATCH — "Change part of this." Partial update.
  • DELETE — "Remove this." Self-explanatory.

What a Response Looks Like

The server replies with:

  • Status Code — a three-digit number telling you what happened
  • Headers — metadata (caching hints, content type, rate limits)
  • Body — the actual data (JSON, HTML, XML, etc.)

Status Codes You Should Know

  • 2xx — Success
    • 200 OK — Things worked
    • 201 Created — Something new was made
    • 204 No Content — Worked, but nothing to return
  • 3xx — Redirection
    • 301 Moved Permanently — Use this URL instead, forever
    • 304 Not Modified — Cache is still fresh
  • 4xx — Client Error (you messed up)
    • 400 Bad Request — Your request is malformed
    • 401 Unauthorized — Who are you? (no auth provided)
    • 403 Forbidden — I know who you are, but no
    • 404 Not Found — This URL doesn't exist
    • 429 Too Many Requests — Slow down
  • 5xx — Server Error (they messed up)
    • 500 Internal Server Error — Something broke on their end
    • 502 Bad Gateway — Their server is having issues
    • 503 Service Unavailable — They're overloaded or down

Authentication

How does the server know who you are? A few common methods:

  • API Keys — A simple token passed in headers. Like a password. Fast but all-or-nothing.
  • Bearer Tokens (JWT) — A signed token containing user info. Stateless and popular.
  • OAuth 2.0 — Delegated access. "Let this app do X without seeing my password." Used by Google, GitHub, etc.
  • Basic Auth — Username:password base64 encoded. Simple but not great for APIs.

Rate Limiting

APIs don't let you hammer them infinitely. They track your requests and cut you off when you exceed a threshold. Common limits: 60 requests/minute, 1000/day.

You'll see rate limit info in response headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset.

Strategies to avoid hitting limits: cache responses, batch requests, exponential backoff on errors.

REST vs Other Styles

REST (REpresentational State Transfer) is the dominant style. It means:

  • URLs represent resources (/users/123)
  • HTTP methods represent actions
  • Stateless — each request has everything needed
  • Responses include links to related resources (HATEOAS)

Alternatives exist: GraphQL (ask for exactly what you need), gRPC (fast, binary, streaming), WebSocket (bidirectional, persistent).

What Happens When You Call an API

Here's the full journey:

  1. Your code builds an HTTP request (URL, method, headers, body)
  2. Your OS resolves the domain to an IP (DNS lookup)
  3. Your OS opens a TCP connection to that IP
  4. Your OS sends the HTTP request over that connection
  5. The server receives the request, parses it
  6. The server authenticates you (or rejects)
  7. The server processes the request (database, logic, external calls)
  8. The server builds a response (status, headers, body)
  9. The response travels back over TCP to you
  10. Your code receives it, parses the body, handles errors

All of this takes anywhere from 50ms (fast) to 2s (slow). The network matters. Geography matters. The server's load matters.

Why It Matters

Understanding APIs makes you a better developer. You stop treating them as magic and start understanding:

  • Why retries need exponential backoff
  • Why caching is critical
  • Why status codes exist
  • Why authentication isn't optional

Every app is just a UI over a database over an API over a server. Peel back the layers and it all makes sense.
