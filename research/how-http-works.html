How HTTP Works

2026-03-19

HTTP is the protocol that powers the web. It's deceptively simple — text messages going back and forth — but the ecosystem around it is anything but. Let's trace a request from your browser to a server and back.

The Basics: Request and Response

Every HTTP communication has two parts: the request (what the client wants) and the response (what the server sends back).

A typical request looks like:

GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html,application/xhtml+xml
Accept-Language: en-US,en;q=0.5
Connection: keep-alive

And the response:

HTTP/1.1 200 OK
Server: nginx/1.18.0
Content-Type: text/html; charset=UTF-8
Content-Length: 1256
Connection: keep-alive

<html>
<head>...</head>
<body>...</body>
</html>

HTTP Methods

The first line of any request has a method — the verb that tells the server what kind of action the client wants:

  • GET — Retrieve data. Should be idempotent (calling it multiple times has the same effect).
  • POST — Send data to create a resource. Not idempotent.
  • PUT — Replace a resource entirely. Idempotent.
  • PATCH — Partially update a resource. Not idempotent.
  • DELETE — Remove a resource. Idempotent.
  • HEAD — Like GET but only headers, no body.
  • OPTIONS — What methods does this URL support?

People often confuse PUT and POST. The rule: POST = create new, PUT = replace existing. If you're sending the same PUT request twice, it should have the same result. POSTing twice typically creates two resources.

Status Codes

The response's first line has a three-digit status code:

  • 1xx — Informational. Rarely seen (100 Continue, 101 Switching Protocols).
  • 2xx — Success. 200 OK, 201 Created, 204 No Content.
  • 3xx — Redirection. 301 Moved Permanently, 302 Found, 304 Not Modified.
  • 4xx — Client error. 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found.
  • 5xx — Server error. 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable.

404 is the famous one, but 403 is often more interesting — it means the server understood the request and knows who you are, but you're not allowed. 401 means you're not authenticated (but might be allowed if you were).

Headers

Headers are the metadata of HTTP. They control caching, cookies, content types, compression, authentication, and more.

Request headers come from the client:

  • Host — Required. The domain being requested.
  • User-Agent — What client (browser, curl, etc.).
  • Accept — What content types can I handle?
  • Authorization — Credentials for authentication.
  • Cookie — Sessions and tracking.
  • Referer — Where did I come from? (Yes, it's misspelled in the spec.)

Response headers come from the server:

  • Content-Type — What is this? (text/html, application/json, etc.)
  • Content-Length — How big is the body?
  • Set-Cookie — Here's a cookie for you.
  • Cache-Control — How should this be cached?
  • Server — What software is running?

State and Cookies

HTTP itself is stateless — each request is independent. But websites need state (you're logged in, your cart has items, etc.). The hack: cookies.

The server sends Set-Cookie with a unique ID. The browser stores it and sends it back with every request. The server looks up that ID and knows who you are.

Cookies have flags that matter:

  • Secure — Only send over HTTPS.
  • HttpOnly — JavaScript can't read it (prevents XSS theft).
  • SameSite — CSRF protection. "Strict" = only same site, "Lax" = navigations okay, "None" = all (requires Secure).

HTTP/1.1 vs HTTP/2 vs HTTP/3

HTTP/1.1 (1997) — The long-running standard. One request at a time per connection (pipelining was tried but failed). Browsers work around this by opening multiple connections (6-8 typically).

HTTP/2 (2015) — Major update:

  • Multiplexing — Multiple requests over one TCP connection, interleaved.
  • Header compression — HPACK reduces redundant headers.
  • Server Push — Server can send resources before asked (controversial, often disabled).
  • Binary — Not text anymore. More efficient to parse.

HTTP/3 (2022) — The new kid:

  • QUIC — Built on UDP instead of TCP. No head-of-line blocking.
  • 0-RTT — Can start sending data immediately on reconnect.
  • Connection migration — Switch networks without reconnecting.

The practical difference most people notice: HTTP/3 is faster on poor connections (mobile, WiFi congestion) because QUIC handles packet loss better than TCP.

HTTPS: HTTP Over TLS

HTTP sends everything in plaintext. HTTPS wraps it in TLS (Transport Layer Security). The connection is encrypted and authenticated (the server proves it owns the private key for the domain).

The handshake:

  1. Client says "hello" and lists supported cipher suites.
  2. Server picks a suite and sends its certificate.
  3. Client verifies the certificate (checks signature, expiration, hostname).
  4. They do key exchange (usually ECDHE) and establish a session key.
  5. Encrypted HTTP begins.

This happens before any HTTP data. With HTTP/1.1, it's 2 extra round trips. With TLS 1.3 + HTTP/3, it's effectively 0-RTT.

What HTTP Doesn't Solve

HTTP is just the transport. It doesn't care what you're sending — that's the application's problem. Common issues that HTTP alone doesn't fix:

  • Authentication — HTTP has Basic (username:password base64) and Digest (hash challenge-response), but real apps use cookies, tokens (JWT), or OAuth.
  • Authorization — HTTP doesn't say what you're allowed to do. That's your app.
  • Integrity — TLS encrypts but doesn't sign at the application layer. For that, you need JWT or similar.
  • Privacy from the server — The server sees everything. For that, you need end-to-end encryption at the application layer.

What Actually Happens When You Type a URL

  1. Browser extracts the scheme (https://). If missing, defaults to HTTP.
  2. Browser does DNS lookup for the hostname → gets IP.
  3. Browser opens TCP connection (or QUIC for HTTP/3) to that IP:port (443 for HTTPS).
  4. TLS handshake: verify certificate, exchange keys.
  5. Browser sends HTTP request: GET / HTTP/1.1 (or HTTP/2 binary frames, or HTTP/3 datagrams).
  6. Server processes, sends response.
  7. Browser renders, parses HTML, finds CSS/JS/images → more requests.

This happens dozens of times for a typical page. That's why HTTP/2 and HTTP/3 matter — they make all those requests faster.

The Takeaway

HTTP is simple text. But it's evolved massively from HTTP/1.1's line-by-line parsing to HTTP/3's UDP-based multiplexing. The important parts aren't the status codes or header names — it's understanding what problems each version solved (and what they didn't).

Your browser does dozens of HTTP requests per page load. The difference between HTTP/1.1 and HTTP/3 on a latency-heavy page can be seconds. That's why this protocol still matters.
