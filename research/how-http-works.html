How HTTP Works

March 12, 2026

HTTP is the lingua franca of the web. Every time you type a URL, load a page, or fetch an API, HTTP is doing the heavy lifting. It's been around since 1991, went through versions 0.9, 1.0, 1.1, 2, and now 3 — and it's still the foundation of everything.

The Request-Response Model

HTTP is simple: a client sends a request, the server sends a response. That's it. No state, no sessions, no magic. Just messages.

A request looks like:

GET /index.html HTTP/1.1
Host: example.com
Accept: text/html
User-Agent: Mozilla/5.0

A response looks like:

HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1256

<html>...</html>

HTTP Methods

The method tells the server what you want to do:

  • GET — retrieve data. Safe, idempotent.
  • POST — send data to create something. Not idempotent.
  • PUT — replace something entirely. Idempotent.
  • PATCH — partial update. Not idempotent.
  • DELETE — remove something. Idempotent.
  • HEAD — like GET but only headers. Used for checking if something exists.
  • OPTIONS — what methods does this resource support?

Idempotent means you can call it twice and get the same result. DELETE is idempotent (deleting something twice is still deleted). POST is not (creating something twice creates two things).

Status Codes

The three-digit code tells you how it went:

  • 1xx — Informational. "I'm working on it." (101 Switching Protocols is the useful one — used for WebSocket upgrade)
  • 2xx — Success. 200 OK, 201 Created, 204 No Content
  • 3xx — Redirection. 301 Moved Permanently, 302 Found (temporary), 304 Not Modified (caching)
  • 4xx — Client error. 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests
  • 5xx — Server error. 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout

Headers

Headers are metadata. They tell you about the message, not the content. Some important ones:

  • Content-Type — what format is the body? text/html, application/json, image/png
  • Content-Length — how big is the body?
  • Authorization — credentials (Bearer token, Basic auth)
  • Cookie — send cookies back to server
  • Set-Cookie — server sets a cookie
  • Cache-Control — caching directives (max-age, no-cache, no-store)
  • Accept — what format can the client handle?
  • User-Agent — what client is making the request?
  • Referer — where did this request come from? (yes, it's misspelled in the spec)
  • X-Forwarded-For — proxy chain, original IP

HTTP/1.1 vs HTTP/2 vs HTTP/3

HTTP/1.1

The workhorse. Persistent connections (keep-alive) by default. Chunked transfer encoding. But: each request/response is sequential. If you need three images, you wait for each one to finish before starting the next. Browsers work around this with connection pooling (6 connections per domain) and domain sharding.

HTTP/2

Multiplexing — multiple requests over one TCP connection. Header compression (HPACK). Server push — server sends resources before client asks. Stream prioritization. But: still TCP (head-of-line blocking problem).

HTTP/3

QUIC instead of TCP. No head-of-line blocking — each stream is independent. Built-in TLS 1.3. Connection migration — same connection works across network changes. Still rolling out.

Caching

Caching is how the web stays fast:

  • Cache-Control: max-age=3600 — cache for one hour
  • Cache-Control: no-store — don't cache at all
  • Cache-Control: no-cache — revalidate every time
  • ETag — server gives a token, client sends If-None-Match: token, server returns 304 if unchanged
  • Last-Modified / If-Modified-Since — time-based version checking

Connection Management

Every TCP connection has a cost (three-way handshake). HTTP/1.1 keeps connections open (keep-alive). HTTP/2 multiplexes over one connection. HTTP/3 goes further with QUIC.

The default limit is 6 connections per origin in most browsers. That's why CDNs matter — they give you more origins to parallelize across.

Security

HTTP itself has no encryption. That's what HTTPS adds (HTTP over TLS). TLS wraps the HTTP stream, encrypts it, verifies the server's identity.

HSTS (Strict-Transport-Security) header tells browsers: "only ever connect to this domain over HTTPS."

The Big Picture

HTTP is elegantly simple: text-based requests and responses, no state, just messages. Everything else — authentication, caching, compression, encryption — is built on top. That simplicity is why it's lasted 35 years. The protocol doesn't care what you're sending — JSON, HTML, images, video. It's just bytes with metadata.

HTTP/2 and HTTP/3 added performance tricks underneath, but the request/response model stayed the same. That's the genius of it.
