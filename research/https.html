How HTTPS Works

March 2, 2026

When you visit a secure website, a lot happens in the background: key exchange, certificate verification, symmetric encryption setup. Here's how your browser and the server establish a secure connection.

The Problem HTTPS Solves

HTTP sends data in plain text. Anyone between you and the server can see what you're sending: passwords, credit cards, private messages.

HTTPS adds encryption. Even if someone intercepts the traffic, they can't read it. But here's the catch: how do you agree on encryption keys without an attacker intercepting them?

TLS: Transport Layer Security

HTTPS uses TLS (formerly SSL) to secure connections. TLS has two main phases:

  1. Handshake — Verify the server, exchange keys, agree on encryption
  2. Data Transfer — Encrypted communication using the agreed keys

The Handshake

When you connect to an HTTPS site:

  1. Browser sends "Hello" — lists supported cipher suites
  2. Server responds with its certificate and chosen cipher
  3. Browser verifies the certificate (more on this below)
  4. Browser generates a pre-master secret, encrypts it with the server's public key
  5. Both derive the master secret, then session keys
  6. Handshake complete — encrypted data transfer begins

Certificates: Who Are You?

The certificate proves the server owns the domain. It's signed by a Certificate Authority (CA) — companies like DigiCert, Let's Encrypt, Comodo.

Your browser has a list of trusted CAs. When it receives a certificate:

  • Checks if the certificate is expired
  • Verifies the CA's signature
  • Checks if the domain matches

If any check fails, your browser shows a scary warning.

Public Key Cryptography

The magic trick: the server has a key pair — public and private. The public key encrypts data; only the private key decrypts it.

RSA (most common):

encrypted = plaintext ^ public_key mod n
plaintext = encrypted ^ private_key mod n

Elliptic Curve (EC) is now preferred — smaller keys, same security.

Symmetric Encryption

After handshake, both sides use symmetric encryption (AES, ChaCha20) with the session keys. This is much faster than public key crypto.

AES-256-GCM is common — 256-bit key, authenticated encryption (detects tampering).

Let's Encrypt and Certificate Automation

Traditionally, certificates cost money and needed manual renewal. Let's Encrypt (launched 2016) changed this:

  • Free certificates
  • Automated renewal via ACME protocol
  • HTTPS is now the default

What Remains

HTTPS is now everywhere — over 80% of web traffic is encrypted. But it's not perfect:

  • Certificate authorities can be compromised
  • TLS 1.2 has known weaknesses; TLS 1.3 is better
  • HTTPS doesn't hide metadata (who you're talking to, for how long)

The next time you see that lock icon, remember: it's the result of a complex dance of cryptography, certificates, and trust — all happening in milliseconds.
