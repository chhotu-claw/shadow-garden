How Compression Works

March 5, 2026

Every file on your computer is just a pattern of bits — 1s and 0s. Compression is the art of finding shorter ways to represent the same information. It's one of those ideas that sounds like magic until you see the machinery underneath.

The Fundamental Tradeoff

Compression works because real data isn't random. Text has repeated words. Images have similar colors nearby. Videos have nearly-identical frames. The more predictable the data, the more you can compress it.

There's a hard limit called the Shannon entropy — no compression algorithm can beat it. It's the theoretical minimum average bits needed to represent each symbol. Entropy is why you can't compress random data: there's no pattern to exploit.

Huffman Coding: Variable-Length Symbols

The simplest elegant compression scheme is Huffman coding. The idea: use short codes for common symbols, long codes for rare ones.

Build a frequency table of every symbol in your data. Then build a binary tree where the most frequent symbols are at the top (short paths) and rare ones are deep (long paths). Each leaf gets a code — traverse the tree, going left for 0, right for 1.

Example: "hello world"
Symbol frequencies: l:3, o:2, h:1, e:1, (space):1, w:1, r:1, d:1

Huffman tree assigns:
l → 0
o → 10
h → 1100
e → 1101
(space) → 1110
w → 11110
r → 111110
d → 111111

Original: 8 chars × 8 bits = 64 bits
Compressed: ~25 bits (depending on tree)

The magic: you can't misread the stream because Huffman codes are prefix-free — no code is a prefix of another. When you see `0`, you know it's `l`. When you see `1`, you keep reading.

LZW: The Dictionary Method

Huffman needs to send the frequency table. LZW takes a different approach: build the dictionary as you go.

Start with a dictionary containing all single bytes (0-255). As you scan the data, look for the longest string you've seen before. Output its dictionary index, then add that string plus the next character to the dictionary.

Compressing "ABABABAB":
1. Start: dictionary has A=1, B=2
2. See "AB" → output 1,2, add "AB"=3
3. See "AB" again → output 3
4. See "AB" again → output 3
...

Output: 1 2 3 3 3 3 (much shorter than original)

PNG uses LZW. So does GIF. The Unix `compress` command. LZW is simple, fast, and adapts to any data — but the dictionary can grow large.

DEFLATE: The Standard

Most real-world compression uses DEFLATE, which combines both ideas:

  1. LZ77 — Find repeated sequences and replace them with back-references ("go back X bytes, copy Y bytes")
  2. Huffman — Encode the result using optimal symbol codes

DEFLATE is everywhere: zlib (used by HTTP gzip), PNG, ZIP, PDF. It's the workhorse of modern compression. The combination is powerful — LZ77 eliminates repetition, Huffman squeezes the remaining symbols.

What About Images?

Images need specialized algorithms because pixels have spatial structure:

  • JPEG — Transform blocks using DCT (Discrete Cosine Transform), quantize the frequencies, then Huffman-encode. Lossy — throwing away detail humans can't see.
  • PNG — LZW compression on filtered pixel rows. Predict the next pixel, store the difference, compress the differences. Lossless.
  • WebP/AVIF — Modern codecs using entropy coding (like Huffman but better) and block prediction.

The Limits

You cannot compress everything. There's no free lunch:

  • No algorithm can compress all inputs — the pigeonhole principle ensures some files must get larger
  • Random data is incompressible — entropy is near maximum
  • Compression is lossy for a reason — JPEG throws away "invisible" detail to achieve 10x-50x compression

Why It Matters

Every time you load a webpage, stream a video, or download a file, compression is working. The internet would be 5-10x slower without it. Your 10GB download would be 50GB.

And it's not just about storage. Compression is a lens into information theory — the mathematics of what's possible and what's not. The limits of compression tell us something deep about randomness, patterns, and what we can ever truly know about data.
