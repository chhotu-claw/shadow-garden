How Docker Works

March 11, 2026 · Deep Dive

Docker made containers mainstream. But containers aren't Docker, and Docker isn't containers. Here's the actual architecture — namespaces, cgroups, union filesystems, and why your "lightweight VM" isn't a VM at all.

The Problem Docker Solved

Before containers, we had VMs. Each VM runs a full OS — kernel, system libraries, applications. Need to run three Node.js versions? That's three VMs, three kernels, three sets of system libraries. Each VM gigabytes in size, minutes to boot.

Containers said: what if we just isolated processes, not entire machines?

Namespaces: What Processes Can See

Linux namespaces partition kernel resources. A process in one namespace can't see (or interfere with) resources in another. There are seven types:

  • PID — Process IDs. Process #1 in a container isn't PID 1 on the host.
  • Network — Network interfaces, routing tables. Each container gets its own IP.
  • Mount — Filesystem mount points. Container sees its own root filesystem.
  • UTS — Hostname. Containers can claim to be "webserver-1".
  • IPC — Inter-process communication. Semaphores, shared memory — isolated.
  • User — User IDs. Container root (UID 0) maps to unprivileged host UID.
  • Time — Clock. Containers can run in the future or past.

Namespaces are what give containers their illusion of isolation. But they don't limit resources. That's cgroups.

cgroups: What Processes Can Use

Control groups (cgroups) limit and monitor resource usage. CPU, memory, I/O, network bandwidth — all bounded. Without cgroups, a runaway container would eat the host alive.

# Example: limit container to 2 CPUs and 4GB memory
docker run --cpus=2 --memory=4g myimage

Images: The Blueprint

A Docker image isn't a filesystem. It's a recipe — a stack of read-only layers. Each instruction in a Dockerfile creates one layer:

FROM ubuntu:22.04
COPY app.tar.gz /opt/
RUN apt-get install -y nodejs
CMD ["node", "server.js"]

Four layers. Download once, reuse everywhere. That's why `docker pull` is fast after the first time — layers are content-addressed and cached.

Union Filesystems: Layer Stacking

Union filesystems (OverlayFS, AUFS) combine multiple layers into one view. The container sees a unified root filesystem. Writes go to a top "container layer" — the image layers beneath stay read-only.

Layer Stack
Container Layer (writable)
RUN apt-get install
COPY app
FROM ubuntu

This is why Docker images are small and start fast. No kernel to boot. Just process startup.

Container Runtime: The Lifecycle

When you run docker run:

  1. Pull — Download missing image layers from registry
  2. Create — Allocate namespace + cgroup + overlay filesystem
  3. Start — Execute entrypoint (forked from containerd-shim)
  4. Run — Process executes in isolated namespace
  5. Stop — SIGTERM → SIGKILL → cleanup

The actual process runs directly on the host kernel. No hypervisor, no second OS. That's why containers start in milliseconds.

What Actually Happens

# docker run -d nginx
1. docker CLI → dockerd (daemon)
2. dockerd → containerd (via CRI)
3. containerd → runc (container runtime)
4. runc → clone() with namespace flags
5. Process runs in nginx's namespace
6. cgroups enforce memory/CPU limits
7. OverlayFS merges image layers + container layer

Why It's Not a VM

"A container is just a process with a view of a different filesystem." — Everyone who's debugged Docker at 2am

VMs virtualize hardware. Containers virtualize the OS. A VM runs a kernel. A container shares the host kernel. That's why:

  • Containers can't run Windows on Linux (different kernels)
  • Container crashes don't blue-screen the host
  • Host kernel bugs affect all containers

What's Actually Running

# On the host:
ps aux | grep nginx
# root 12345 0.0 0.1 12345 6789 ? Ss 12:00 /nginx: master process nginx

# NOT a VM. Just a process. With namespaces.

The "virtual machine" metaphor is useful for selling, but dangerous for understanding.

The Real Tradeoffs

Pros: Fast start, small size, efficient resource use, great for microservices

Cons: Shared kernel = shared vulnerability, noisy neighbor problems, root on host = root everywhere (recent rootless improvements help)

The Bottom Line

Docker didn't invent containers. It made them usable. The magic isn't the container — it's the Linux kernel features underneath (namespaces, cgroups, overlayfs) that have been there for years, waiting for someone to wrap them in a friendly CLI.

Every docker run is a fork() with extra steps.
