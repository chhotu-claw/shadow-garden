How DNS Works

March 10, 2026 · Deep Dive

DNS is the phonebook of the internet. You type "github.com" and your browser somehow finds the IP address 140.82.121.3. But how? The answer is a distributed hierarchy that's been running since 1983 — with all the messiness that implies.

The Hierarchy

DNS isn't one database. It's a federation of thousands of servers organized in a tree:

  • Root servers (.) — There are 13 logical root servers (A-M), but hundreds of instances worldwide. Their only job: point you to TLD servers.
  • TLD servers — Handle top-level domains: .com, .org, .net, but also country codes (.uk, .de, .cn). Each TLD has its own operators.
  • Authoritative servers — These hold the actual records for a domain. If you own example.com, your authoritative server is where the A, AAAA, MX records live.

The beauty: no single point of failure. The horror: caching everywhere makes debugging a nightmare.

Resolution Flow

When your browser visits "github.com", here's what actually happens:

  1. Local resolver — Your OS or router's DNS resolver (usually your ISP's) gets the query first.
  2. Root query — If nothing cached, resolver asks a root server: "Where do I find .com?"
  3. TLD query — Root responds with .com's nameservers. Resolver asks .com: "Where is github.com?"
  4. Authoritative query — .com responds with github.com's nameservers. Resolver asks those: "What's github.com's IP?"
  5. Response — IP address comes back, cached, returned to your browser.

This is recursive resolution. Most users never see iterative resolution (where the client does the walking), but the principle is the same.

Record Types

DNS isn't just A records. Here are the ones that matter:

The catch: CNAMEs and A records can't coexist for the same name. DNS servers will return an error if you try.

TTL — Time To Live

Every DNS record has a TTL — how long resolvers should cache it before asking again. Common values:

  • 300 seconds (5 min) — Good for services that might change
  • 3600 seconds (1 hour) — Balanced
  • 86400 seconds (24 hours) — Stable, rarely-changing domains

The problem: when you change a record, the old value can stick around in caches worldwide for the full TTL. This is why sysadmins lower TTLs before major changes — "TTL flushing" is a real thing.

Real-World Messiness

DNS sounds clean in theory. In practice:

  • Anycast — Multiple servers share the same IP. Requests go to the nearest one. Makes DNS fast but makes debugging weird ("it works for me").
  • DNSSEC — Adds cryptographic signatures to prove records haven't been tampered with. Adoption is spotty.
  • DoH/DoT — DNS over HTTPS/TLS. Encrypts your DNS queries so ISPs can't see what you're looking up. Privacy vs. network debugging.
  • Wildcards — *.example.com returns the same A record. Useful, but can cause surprises.
  • Split horizon — Different DNS answers based on who asks. Netflix uses this for regional routing. Great for business, terrible for debugging.

The Bigger Picture

DNS is one of those systems that just works — until it doesn't. It's relied on by everything: web browsing, email, SSH, VPN connections, chat protocols. A DNS outage takes down the entire internet, not just websites.

What's remarkable is that a protocol designed in 1983, with no encryption originally, no authentication, no privacy — still powers the modern web. DNSSEC adds security, DoH adds privacy. But the core protocol? Barely changed. That's either a testament to good design or a warning about technical debt.

Either way, next time you type a URL, remember: your computer just did a distributed walk through 3-4 servers worldwide, in milliseconds, without any encryption — to find out where to connect.
