How DNS Works

March 13, 2026 · DEEP DIVE

The Domain Name System is arguably the most critical infrastructure on the internet, yet most people have no idea how it works. You use it every time you type a URL. Without DNS, you'd have to memorize 142.250.185.46 instead of google.com.

The Problem DNS Solves

Computers talk to each other using IP addresses — numeric identifiers like 192.168.1.1 or the newer 2001:0db8:85a3::8a2e:0370:7334. Humans, unsurprisingly, are terrible at remembering strings of numbers. We remember names.

DNS is the bridge. It translates google.com into 142.250.185.46 (and dozens of other IPs for Google). This translation is called resolution.

The Hierarchy

DNS isn't a single database. It's a distributed hierarchy — a tree of servers that cooperate to resolve names. Here's how it works:

1. Root Servers (.)

At the top of the tree are 13 root server clusters (labeled A through M). They don't know Google's IP, but they know where .com TLD servers live. When you query google.com, the root server says "ask the .com TLD servers."

2. TLD Servers

Top-Level Domain servers handle the last part of a domain: .com, .org, .io, .gov, country codes like .uk, or the newer .app, .dev. They don't know Google's IP either, but they know which authoritative nameservers handle google.com.

3. Authoritative Nameservers

These are the servers that actually hold the DNS records for a domain. If you own example.com, your domain registrar provides or lets you configure authoritative nameservers (often through providers like Cloudflare, Route 53, or DNSimple). These servers return the final answer: "google.com → 142.250.185.46".

The Resolution Process

When your browser visits google.com, here's exactly what happens:

  1. Browser check: Your browser checks its cache first.
  2. OS check: If not found, your OS checks its DNS cache.
  3. Recursive resolver: Your computer asks your ISP's (or network's) recursive resolver: "What's the IP for google.com?"
  4. Root query: The recursive resolver asks a root server: "Who handles .com?"
  5. TLD query: Root replies with .com TLD server IPs. The resolver asks the .com TLD: "Who handles google.com?"
  6. Authoritative query: The TLD replies with authoritative nameserver IPs. The resolver asks one: "What's google.com's IP?"
  7. Answer: The authoritative server returns 142.250.185.46.
  8. Caching: The recursive resolver caches this result and returns it to your computer.

This sounds slow, but it's actually fast — results are cached at every level, and the actual query chain only happens the first time or when caches expire.

DNS Record Types

DNS isn't just for IP addresses. It stores many record types:

A (Address)

Maps a domain to an IPv4 address.

google.com.    IN A    142.250.185.46

AAAA

Maps a domain to an IPv6 address.

google.com.    IN AAAA    2607:f8b0:4005:802::200e

CNAME (Canonical Name)

Creates an alias — points one domain to another. The DNS resolver follows the chain.

www.google.com.    IN CNAME    google.com.

MX (Mail Exchange)

Specifies mail servers that accept email for a domain. Includes priority (lower number = higher priority).

example.com.    IN MX    10 mail1.example.com.
example.com.    IN MX    20 mail2.example.com.

TXT

Arbitrary text. Used for verification, SPF (email security), DKIM, DMARC, and more.

example.com.    IN TXT    "v=spf1 include:_spf.google.com ~all"

NS (Name Server)

Delegates a zone to authoritative nameservers.

google.com.    IN NS    ns1.google.com.

SOA (Start of Authority)

Contains administrative info: primary nameserver, admin email, serial number, refresh intervals.

PTR (Pointer)

Reverse DNS — maps IP to domain (the opposite of A/AAAA). Used for verification and spam filtering.

Caching: Why DNS Is Fast

DNS caching happens at multiple levels:

  • Browser cache: Minutes to hours (configurable)
  • Operating system cache: Minutes (Windows caches for ~24 hours by default)
  • Recursive resolver: Minutes to hours (TTL from the record)
  • ISP resolvers: Same as above

Each DNS record has a TTL (Time To Live) — how long to cache before re-querying. Typical values range from 300 seconds (5 minutes) to 86400 seconds (24 hours). Lower TTLs = faster propagation when you change records, but more load on authoritative servers.

Security Issues

DNS Spoofing / Cache Poisoning

An attacker injects fake DNS records into a resolver's cache. Users get redirected to malicious sites. This is why DNSSEC matters.

DNSSEC

Domain Name System Security Extensions adds cryptographic signatures to DNS data. It verifies that records haven't been tampered with. Not universally deployed, but increasingly common.

DDoS Attacks

DNS servers are frequent targets. The 2016 Dyn attack used Mirai botnet to flood DNS providers, taking down Twitter, Spotify, and dozens of major sites.

DNS over HTTPS (DoH) / DNS over TLS (DoT)

Traditional DNS queries are unencrypted — your ISP sees every domain you visit. DoH and DoT encrypt DNS queries, improving privacy. Browsers and some OSes now support them.

Interesting Stats

  • There are 13 root server IP addresses (not 13 servers — hundreds of instances worldwide)
  • Average DNS lookup takes 20-100ms
  • Cloudflare's 1.1.1.1 resolver processes trillions of queries daily
  • The oldest DNS root server (A.root-servers.net) has been running since 1985

TL;DR

DNS translates human-readable domain names into computer-readable IP addresses. It's a hierarchical, cached system with root servers at the top, TLD servers below, and authoritative nameservers holding the actual records. The resolution process involves querying up the chain, but caching makes it fast in practice.

It's also insecure by design — unencrypted, unauthenticated queries that anyone can intercept. DNSSEC and DoH/DoT are fixing that, slowly.
