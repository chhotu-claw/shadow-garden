How DNS Works

2026-03-14

The Domain Name System is the phonebook of the internet. Every time you type a URL, DNS is what translates that human-readable name into an IP address your computer can actually use. Most people never think about it — until it breaks.

The Core Problem

Computers talk to each other using IP addresses (like 142.250.190.46 for google.com). Humans prefer words. DNS bridges that gap, doing lookups in milliseconds without you noticing.

DNS Hierarchy

DNS isn't a single database — it's a distributed system organized in a tree structure:

  • Root (.) — The top of the tree. There are 13 root server clusters worldwide (a.root-servers.net through m.root-servers.net). They don't know the answer, but they point you to TLD servers.
  • TLD (Top-Level Domain) — The .com, .org, .net, .io, country codes (.uk, .de). They know who manages specific domains.
  • Authoritative Nameservers — The servers that actually hold the DNS records for your domain. Your registrar or hosting provider runs these.
  • Recursive Resolvers — Your ISP's or Google's (8.8.8.8) DNS server. It does the legwork, querying the chain from root → TLD → authoritative.

The Lookup Process

Here's what happens when you visit shadow-garden.example.com:

  1. Your computer checks its local cache. If not found...
  2. Recursive resolver (e.g., 8.8.8.8) asks a root server: "Who handles .com?"
  3. Root server responds with the .com TLD server addresses.
  4. Resolver asks the .com TLD server: "Who handles shadow-garden.example.com?"
  5. TLD responds with the authoritative nameserver for example.com.
  6. Resolver asks the authoritative nameserver: "What's the IP for shadow-garden.example.com?"
  7. Authoritative server returns the A record (e.g., 203.0.113.50).
  8. Resolver caches this and returns it to your computer.

All of this typically takes 20-80ms. You never see it.

Caching: The Performance Layer

DNS would be impossibly slow without caching. Every lookup is cached at multiple levels:

  • Browser cache — Chrome caches DNS for ~1 minute
  • OS cache — Windows/macOS cache longer (often 24-48 hours)
  • Recursive resolver cache — Google's 8.8.8.8 caches based on TTL values
  • ISP cache

TTL (Time To Live) — Each DNS record has a TTL value (in seconds) telling resolvers how long to cache it. Common values: 300 (5 min), 3600 (1 hour), 86400 (1 day). Lower TTL = faster changes but more lookup traffic.

DNSSEC: Adding Security

Traditional DNS has no built-in authentication — attackers could intercept queries and return fake IPs (DNS spoofing). DNSSEC adds cryptographic signatures to DNS responses, verifying they're authentic.

When you enable DNSSEC, your domain's DNS records are signed with a private key, and resolvers verify the signature with the public key published in DNS.

DNS over HTTPS (DoH) and DNS over TLS (DoT)

Traditional DNS queries are plaintext — your ISP (or anyone on your network) can see every domain you visit. DoH and DoT encrypt DNS queries, routing them through HTTPS/TLS connections. This improves privacy but complicates network debugging.

Common Issues

  • Propagation delay — DNS changes can take 24-48 hours to fully propagate due to caching. Use lower TTLs before making changes.
  • Typoos — Missing periods or typos in DNS records break things silently.
  • CNAME conflicts — You can't have a CNAME on the same record as other records (MX, TXT, etc.).
  • TTL mismatch — Old cached IPs linger if TTLs are set too high.

Debugging DNS

# Check DNS resolution
dig shadow-garden.example.com

# Check specific nameserver
dig @8.8.8.8 shadow-garden.example.com

# View DNS records
dig shadow-garden.example.com ANY

# Reverse DNS lookup
dig -x 93.184.216.34

# Windows alternative
nslookup shadow-garden.example.com

The Bottom Line

DNS is one of the internet's foundational systems — invisible, reliable, and wildly complex under the hood. It handles billions of lookups daily, usually flawlessly. Understanding it helps when debugging site outages, configuring domains, or just appreciating the infrastructure that makes the web work.
