How DNS Works

March 9, 2026

You type "google.com" into your browser. Three hundred milliseconds later, you're looking at a search page. What happened in that gap? A lot. More than you'd think.

The Problem DNS Solves

Computers don't think in names. They think in numbers — IP addresses, specifically. Every server on the internet has one: a 32-bit number (IPv4) or 128-bit number (IPv6). Remembering that 142.250.185.78 is Google is not something humans can do.

DNS is the translation layer. It maps human-readable names to machine-readable addresses. It's the internet's phone book — but unlike a phone book, it changes constantly and is distributed across the world.

The DNS Hierarchy

DNS isn't a single database. It's a hierarchy — a tree of servers, each responsible for a piece of the namespace.

Root Nameservers (.)

The top of the tree. There are 13 logical root server IP addresses (actually hundreds of instances worldwide). They don't know what google.com is — they know which TLD nameservers to ask about .com, .org, .net.

TLD Nameservers

Top-Level Domain servers. Responsible for everything under a TLD — all .com domains, all .org domains, etc. When you ask about google.com, the root says "ask the .com servers."

Authoritative Nameservers

These hold the actual DNS records for specific domains. Google's authoritative nameservers know that google.com maps to specific IP addresses. This is where the buck stops.

The Resolution Flow

Here's what actually happens when you visit a site:

  1. Browser checks its cache. Already visited? Recent? Browser remembers.
  2. OS checks its cache. OS has its own DNS cache.
  3. Recursive resolver gets the query. Usually your ISP's resolver (8.8.8.8, 1.1.1.1). This is where the magic happens.
  4. Resolver asks root. "Where do I find .com?"
  5. Resolver asks TLD. "Where do I find google.com?"
  6. Resolver asks authoritative. "What's the IP for google.com?"
  7. Answer comes back. Cached at each step for next time.

This is called a recursive query. Your machine asks the resolver to do the legwork. There's also iterative querying — the resolver bounces between servers itself, which is what actually happens under the hood.

DNS Records

DNS isn't just A records (address). There are many record types:

  • A — IPv4 address (e.g., google.com → 142.250.185.78)
  • AAAA — IPv6 address
  • CNAME — Canonical name (alias). blog.example.com → example.com
  • MX — Mail exchange. Where to deliver email.
  • TXT — Arbitrary text. Used for SPF, DKIM, domain verification.
  • NS — Nameserver. Delegates a zone to another server.
  • SOA — Start of Authority. Zone metadata — serial, refresh, retry.

The TTL Problem

Every DNS record has a Time To Live (TTL) — how long resolvers can cache it before asking again. Low TTL = fast changes but more queries. High TTL = less load but slower propagation.

When you change DNS, you're not flipping a switch. You're waiting for caches to expire. This is why DNS changes can take 24-48 hours to fully propagate — not because DNS is slow, because caches are sticky.

What's Actually in the Wild

The textbook explanation assumes a clean hierarchy. Reality is messier:

  • CDNs. Cloudflare, Akamai, Fastly sit in front of authoritative servers. They return different IPs based on where you are geographically.
  • Anycast. Multiple servers share the same IP. A request to 1.1.1.1 goes to the nearest instance. DNS naturally does this.
  • DNSSEC. Cryptographic signing of DNS data. Verifies authenticity — prevents spoofing.
  • DNS over HTTPS (DoH). Encrypted DNS queries. Bypasses ISP observation.
  • Wildcards. *.example.com can resolve to everything.

The Security Angle

DNS was designed in 1983. Trust was assumed. This is a problem.

  • DNS spoofing/cache poisoning. Attacker injects false records into a resolver's cache. Fixed by DNSSEC.
  • DNS tunneling. Exfiltrating data through DNS queries — DNS wasn't designed for security monitoring.
  • DNS hijacking. Redirecting users through malicious resolvers.
  • Fast flux. Rapidly changing DNS records to evade detection. Botnets use this.
DNS is the internet's duct tape. It was never designed for this scale, but it bends rather than breaks.

What You Actually Control

When you buy a domain, you control the authoritative nameservers. You point them at your registrar, your hosting provider, Cloudflare, or your own bind server. From there, you create records.

The protocol is simple: query, answer. But the ecosystem around it — caching, CDNs, load balancers, anycast — makes it surprisingly complex.

The Takeaway

DNS works because it's dumb. It's a distributed hash table with no central authority, no complex consensus, just trust and caching. It tolerates failure, absorbs scale, and barely anyone notices. That's the engineering sweet spot — boring infrastructure that just works.
