How DNS Works

March 7, 2026

You type "google.com" into your browser. Less than a second later, you're looking at a search page. But your computer doesn't speak "google.com" — it speaks numbers, specifically IP addresses like 142.250.185.78. Somewhere in between, something translated that name into that number. That something is DNS, and it's the closest thing the internet has to magic.

The Problem DNS Solves

IP addresses are how computers find each other. They're numeric, hierarchical, and efficient — for machines. But humans are terrible at remembering strings of numbers. Imagine trying to remember that Google's IP is 142.250.185.78, Facebook is 157.240.1.35, and every website you want to visit is just another string of digits.

DNS solves this with a simple idea: a distributed, hierarchical phone book. Names point to numbers. Humans remember names; computers handle numbers. The system is decentralized by design — no single entity controls the entire internet's address book.

The Hierarchy

DNS is organized as a tree. At the top are root servers — there are 13 logical root server IP addresses (with hundreds of physical instances worldwide). These servers know where the top-level domain (TLD) servers are: .com, .org, .net, .gov, country codes like .uk, .de, .in.

Below TLD servers are authoritative name servers for specific domains. Google's authoritative name servers know all the IP addresses for google.com and its subdomains. Amazon's Route 53 servers manage AWS properties. Your domain registrar's name servers handle your personal domain.

This hierarchy means no single database holds all the world's domain names. Instead, information is distributed across thousands of servers that talk to each other.

The Lookup Process

Here's what happens when you type "example.com" in your browser:

  1. Your computer checks its cache. If it recently resolved "example.com", it already has the IP and skips the rest.
  2. Your computer asks a recursive resolver. Usually provided by your ISP or a service like Google (8.8.8.8) or Cloudflare (1.1.1.1). This resolver does the heavy lifting.
  3. The resolver asks a root server. "Where do I find .com?" The root server responds with the IP of a .com TLD server.
  4. The resolver asks the TLD server. "Where do I find example.com?" The TLD server responds with the authoritative name server's IP.
  5. The resolver asks the authoritative server. "What's the IP of example.com?" The server responds with the answer: 93.184.216.34.
  6. The resolver gives your computer the IP. Your browser now has what it needs to connect.

This entire exchange takes milliseconds and happens millions of times per second worldwide.

Record Types

DNS isn't just for mapping names to IP addresses. It's a flexible system that stores many types of records:

  • A (Address) — maps a domain to an IPv4 address (e.g., example.com → 93.184.216.34)
  • AAAA — maps a domain to an IPv6 address
  • CNAME (Canonical Name) — maps an alias to the canonical name. "blog.example.com" might point to "example.github.io."
  • MX (Mail Exchange) — specifies the mail server that accepts email for the domain
  • TXT — arbitrary text, often used for domain verification and SPF records
  • NS (Name Server) — delegates a zone to authoritative servers
  • SOA (Start of Authority) — contains administrative info about the zone (serial number, refresh interval, etc.)

Caching All The Way Down

Every step in the DNS lookup chain caches results. This is critical for performance — otherwise, every website visit would require four separate server queries.

Your computer caches DNS results. Your router caches them. Your ISP's recursive resolver caches them. Authoritative servers also cache their results with a TTL (Time To Live) — a value specifying how many seconds the record should be considered valid.

TTL trade-offs matter. Short TTLs (like 300 seconds = 5 minutes) allow fast changes but increase query load. Long TTLs (like 86400 seconds = 24 hours) reduce load but slow down recovery if you need to change an IP address.

DNSSEC — Adding Security

DNS was designed in 1983, before security was a major concern. The original protocol sent queries in plain text — anyone could intercept or modify them. This is called DNS spoofing or cache poisoning.

DNSSEC adds cryptographic signatures to DNS records. Each zone signs its records with a private key; recursive resolvers verify the signature with a public key. This ensures the answer you get is actually from the authoritative server — not from an attacker intercepting your query.

DNSSEC is increasingly deployed but not universal. The root zone is signed; many TLDs are signed; many major domains are signed. But there's still a long way to go for universal adoption.

DNS Over HTTPS (DoH) and DNS Over TLS (DoT)

Even with DNSSEC, DNS queries themselves are unencrypted. Your ISP can still see every domain you resolve. DoH and DoT encrypt DNS traffic, wrapping it in HTTPS or TLS. This prevents eavesdropping and makes DNS lookups indistinguishable from regular web traffic.

Major browsers now support DoH. Cloudflare (1.1.1.1), Google (8.8.8.8), and Quad9 all offer encrypted DNS services. This is a significant privacy improvement, though it raises questions about who you trust with your DNS queries.

What Could Go Wrong

DNS is surprisingly fragile:

  • DDoS attacks — attackers can overwhelm DNS servers, making websites unreachable even if the servers themselves are fine. The 2016 Dyn attack used compromised IoT devices to flood DNS providers, taking down Twitter, Netflix, and dozens of major sites.
  • Cache poisoning — an attacker injects fake DNS records into a resolver's cache, redirecting traffic to malicious servers.
  • Domain hijacking — attackers transfer your domain to themselves by compromising your registrar account or exploiting DNS flaws.
  • TTL propagation delays — changing a DNS record doesn't happen instantly. The old IP can be cached for hours, causing confusion and downtime.

The Invisible Infrastructure

Most people never think about DNS. It just works, reliably, billions of times a day. It's one of those invisible systems that makes the internet usable — like IP itself, or TCP, or BGP.

But DNS is also deeply human. It's where tech meets language, where organization meets chaos, where trust is delegated across organizational boundaries. When you buy a domain, you're participating in a system that spans governments, corporations, and volunteer operators. It has no CEO, no central control, and no backup — and somehow, it still works.

Next time you type a website name, pause for a millisecond. Somewhere, a hierarchy of servers is translating that name into a number, passing it back through caches, and handing your browser the address it needs. It happens so fast you never notice. That's DNS.
