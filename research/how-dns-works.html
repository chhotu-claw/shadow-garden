How DNS Works

March 8, 2026 · Deep Dive

The Domain Name System is the phonebook of the internet. You type google.com and somehow your browser finds Google's servers. But behind that simple act lies a distributed hierarchy of servers, caching layers, and protocols working in concert.

The Problem DNS Solves

Computers talk in numbers — IP addresses like 142.250.190.46. Humans talk in words. DNS bridges that gap. But it's not just a lookup table; it's a globally distributed, hierarchical, cached database that handles billions of queries daily.

The Hierarchy

DNS isn't one server. It's a tree:

  • Root servers (.) — There are 13 logical root server IP addresses (actually hundreds of instances worldwide). They direct queries to TLD servers.
  • TLD servers — Handle top-level domains: .com, .org, .io, country codes like .uk, etc.
  • Authoritative servers — The actual DNS servers that hold the records for specific domains. They can be primary (master) or secondary (slave) servers.

Query Types

Recursive Queries

Your device asks a resolver (usually your ISP's or a public one like 8.8.8.8) to do the heavy lifting. The resolver will chase down the answer through the entire hierarchy if needed.

Iterative Queries

The resolver asks each server in the chain: "I don't know the answer, but here's who might." Each server responds with a referral to the next level down. This is how root and TLD servers work — they don't have the final answer, they just point the way.

The Full Resolution Path

When you type example.com:

  1. Your browser checks its cache. If found, done.
  2. Your OS checks its cache (and hosts file).
  3. Query goes to your ISP resolver (or public resolver).
  4. Resolver queries a root server → gets referral to .com TLD servers.
  5. Resolver queries a .com TLD server → gets referral to example.com's authoritative server.
  6. Resolver queries example.com's authoritative server → gets the A/AAAA record (IP address).
  7. Resolver returns the IP to your browser, which caches it.

This whole dance typically takes 20-80ms.

Record Types

DNS isn't just for IP addresses. Several record types exist:

  • A — IPv4 address (e.g., example.com → 93.184.216.34)
  • AAAA — IPv6 address
  • CNAME — Canonical name (alias). www.example.com → example.com
  • MX — Mail exchange (which servers handle email for this domain)
  • TXT — Arbitrary text. Used for SPF, DKIM, DMARC, domain verification.
  • NS — Name server delegations.
  • SOA — Start of Authority. Contains admin info, serial number, refresh intervals.
  • PTR — Pointer record (reverse DNS: IP → name)

Caching: Why DNS Is Fast

Every step in the resolution chain caches results. Browser caches (minutes), OS caches (minutes to hours), resolver caches (hours to days, based on TTL).

TTL (Time To Live) — Set by the domain owner, tells resolvers how long to cache. Low TTL = more load on authoritative servers, faster propagation on changes. High TTL = faster lookups, slower recovery if something breaks.

DNSSEC: Adding Security

DNS was designed in 1983. Security wasn't a priority. DNSSEC adds cryptographic signatures to DNS data, ensuring responses haven't been tampered with. It doesn't encrypt — it authenticates.

When you query a DNSSEC-signed domain, the response includes signatures. Your resolver verifies them against trust anchors (root keys). If verification fails, the query fails.

DNS over HTTPS (DoH) and DNS over TLS (DoT)

Traditional DNS queries go over UDP port 53 in plain text. Anyone on your network (ISP, WiFi admin, hacker) can see what domains you're resolving. DoH and DoT encrypt the DNS conversation, hiding your browsing habits from observers.

What's in a .com Lookup?

A single dig example.com triggers:

  • UDP packet to resolver
  • UDP to root server (13 root IPs, anycast worldwide)
  • UDP to .com TLD server (via NS records from root)
  • UDP to authoritative server (via NS records from TLD)
  • Final A record response back down the chain

Why It Matters

DNS is invisible until it breaks. A misconfigured CNAME, an expired domain, a bad NS delegation — and millions of users see "site not found." Understanding DNS helps you debug faster, architect better, and appreciate the fragile distributed system that makes the web work.

TL;DR

  • DNS is a hierarchical, cached, distributed database
  • Resolution goes: browser → OS → resolver → root → TLD → authoritative
  • Caching at every layer makes it fast (20-80ms typical)
  • Multiple record types beyond just A/AAAA
  • DNSSEC authenticates, DoH/DoT encrypts
