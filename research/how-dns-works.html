How DNS Works

March 25, 2026 · DEEP DIVE

Every time you type google.com into your browser, a invisible machine kicks into gear. A dozen servers around the world exchange messages in milliseconds, translating your human-readable request into the IP address your computer actually needs. This is DNS — the internet's address book, and one of the most critical systems you've never thought about.

The Problem DNS Solves

Computers don't understand "google.com". They understand IP addresses like 142.250.185.46 (IPv4) or 2607:f8b0:400e:c09::8a (IPv6). Remembering IP addresses for every website is insane. DNS is the bridge — it maps memorable names to numeric addresses.

But here's the thing: DNS wasn't designed to be secure. It wasn't designed to be fast. It was designed in 1983 as a simple lookup system for a small academic network. The fact that it still works at all at internet scale is a testament to clever engineering layered on top of a fundamentally fragile foundation.

The DNS Hierarchy

DNS is structured as a tree. At the top is the root domain — that invisible dot at the end of every domain. Below that are top-level domains (TLDs): .com, .org, .io, country codes like .uk or .jp. Below TLDs are second-level domains like "google" or "github". And below those are subdomains like "www" or "api".

. → .com → google → www

Each level of this tree is managed by authoritative name servers. When you ask "where is google.com?", you're essentially asking: "Who knows the IP for google.com?" The answer comes from the chain of these servers.

The Lookup Process

When your browser needs to resolve example.com, here's what actually happens:

  1. Your browser checks its cache. If it recently looked up example.com, it might already know the answer. Browsers cache DNS results for a few seconds to minutes.
  2. Your OS checks its cache. The operating system maintains its own DNS cache, usually with a longer TTL (time-to-live).
  3. Query goes to a recursive resolver. Your computer asks your configured DNS server (usually your ISP's or something like 8.8.8.8 from Google) — "What's the IP for example.com?"
  4. The recursive resolver starts its own journey. If it doesn't have the answer cached, it begins the iterative query process:
    1. Asks a root server (there are 13 logical root servers, but hundreds of instances globally): "Who knows about .com?"
    2. Root server responds: "Ask these TLD servers for .com."
    3. Asks a .com TLD server: "Who knows about example.com?"
    4. TLD server responds: "Ask this authoritative server for example.com."
    5. Asks the authoritative server for example.com: "What's the IP?"
    6. Authoritative server responds with the IP address.
  5. The recursive resolver returns the answer to your browser.
  6. Your browser connects to the IP.

This sounds slow, but it's actually incredibly fast — typically 20-100ms end-to-end. The secret? Caching everywhere. Recursive resolvers cache results at every level, so most lookups never hit the authoritative servers at all.

Record Types

DNS isn't just for IP addresses. It stores different record types for different purposes:

  • A (Address) — Maps a domain to an IPv4 address. The basic one.
  • AAAA — Maps a domain to an IPv6 address.
  • CNAME (Canonical Name) — Maps a domain to another domain. Useful for aliases. www.example.com might be a CNAME pointing to example.com.
  • MX (Mail Exchange) — Specifies which server handles email for the domain.
  • TXT — Arbitrary text. Used for verification (SPF, DKIM), ownership proof, and increasingly for security (DMARC, etc.).
  • NS (Name Server) — Delegates a subdomain to another name server.
  • SOA (Start of Authority) — Contains administrative info about the zone: serial number, refresh intervals, admin email.

The Caching Problem

DNS caching is both the system saving grace and its biggest weakness. When you change your website's IP address, old cached values can persist for hours — this is why DNS changes often take 24-48 hours to propagate fully.

The TTL (Time-To-Live) on a DNS record tells resolvers how long to cache it. Low TTLs (300 seconds = 5 minutes) mean faster propagation but more load on authoritative servers. High TTLs (86400 seconds = 24 hours) mean less load but slower changes.

When you deploy a new website and nothing loads, it's usually not your code. It's DNS.

Security Issues

DNS was designed without security in mind. Some problems:

  • DNS Cache Poisoning — An attacker tricks a resolver into caching a fake record, redirecting traffic to a malicious server. This is why DNSSEC (DNS Security Extensions) was created.
  • DNS Hijacking — Attackers compromise a domain's name server or trick ISP resolvers to return wrong IPs.
  • DNS Tunneling — Using DNS queries to tunnel data through the DNS infrastructure, bypassing firewalls. Useful for exfiltration.
  • Fast Flux — Malware rotates DNS records rapidly to evade detection and takedown.

DNSSEC adds cryptographic signatures to DNS records, allowing clients to verify authenticity. But adoption is still incomplete — some TLDs don't support it, and many recursive resolvers don't validate.

Modern Improvements

Beyond DNSSEC, there's been work on making DNS faster and more private:

  • DNS over HTTPS (DoH) — Encodes DNS queries as HTTPS requests, hiding them from network observers. Browser-native.
  • DNS over TLS (DoT) — Encapsulates DNS in TLS tunnels. Used by some OS-level resolvers.
  • Encrypted Client Hello (ECH) — Hides the Server Name Indication (SNI) — the part of a TLS handshake that reveals which website you're visiting.

Why It Matters

DNS is the foundation of the internet, yet most developers treat it as magic. Understanding how lookups work, why DNS can be slow, what record types do, and how to debug resolution problems is foundational infrastructure knowledge.

Next time your website "doesn't load," remember: it's probably not your code. It's DNS.
