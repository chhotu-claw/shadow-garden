How Vector Databases Work

March 24, 2026

You've seen the hype: "semantic search," "AI-native databases," "find things by meaning, not keywords." But what's actually happening under the hood? Let's take apart vector databases and see what makes them tick.

The Core Problem

Traditional databases query by exact match. You ask "where color = 'blue'" and get rows where color is exactly blue. Vector databases answer a different question: "find things most similar to this." That's fundamentally harder because "similar" lives in continuous space, not discrete buckets.

The trick: convert anything—text, images, audio—into a list of numbers (a vector) that captures its meaning. This is what embedding models do. The magic isn't the database; it's the conversion from raw data to numbers that somehow preserve meaning.

What Are Embeddings?

An embedding is just an array of floating-point numbers. A sentence becomes a vector of 768 or 1536 dimensions. The key property: similar things cluster together in this high-dimensional space.

If "cat" maps to [0.1, -0.3, 0.7, ...] and "dog" maps to [0.12, -0.28, 0.72, ...], they're close. "Car" might be [0.9, 0.1, -0.2, ...]—far away. The embedding model decides what "close" means based on what it learned during training.

These vectors live in a space with hundreds of dimensions. Distance metrics matter:

  • Cosine similarity: angle between vectors, ignores magnitude—used when direction matters more than length
  • Euclidean distance: straight-line distance—used when absolute position matters
  • Dot product: combination of angle and magnitude—fast, widely used

The Naive Approach (and Why It Breaks)

Given a query vector, you could iterate through every stored vector, compute distance to all 1 million others, sort, and return the top-k. This is exact—"brute force" search.

Problem: 1M vectors × 768 dimensions × floating-point math = slow. With 10M vectors, this doesn't scale. A single query might take seconds. Users expect milliseconds.

Approximate Nearest Neighbor (ANN)

Vector databases don't find the exact nearest neighbors—they find probably close enough ones, much faster. This is the ANN problem, and there are several tricks:

HNSW: Hierarchical Navigable Small World

The most popular algorithm. Think of it as a subway map:

  • Start at a central station (top layer)
  • Take express lines to get near your destination (higher layers = longer jumps)
  • Switch to local trains for the final approach (lower layers = shorter jumps)

Each vector gets linked to nearby neighbors. Search navigates this graph. You don't check every point—you hop through connections, like finding a friend in a city by asking strangers "do you know someone closer to where I'm going?"

HNSW builds this graph with random entry points, creating "small world" properties where paths are short despite the high dimensions. It's fast (logarithmic complexity) and accurate (95%+ recall is achievable).

IVF: Inverted File Index

Cluster vectors into groups (" Voronoi cells" in geometry). When searching, only check clusters near your query point—not all 1M vectors, just the few hundred in nearby clusters.

Builds clusters using k-means. Accuracy trade-off: more clusters = more accurate but slower.

Product Quantization (PQ)

Compress vectors by splitting them into chunks and representing each chunk with a limited codebook. Instead of storing 768 floats, store 4 bytes. Massive memory savings—critical when you're storing billions of vectors.

Search becomes "compare compressed codes" rather than full vectors. Some accuracy loss, but often worth it.

How Real Systems Combine These

Production vector databases layer multiple techniques:

  • Pinecone: HNSW on SSD for large indexes, in-memory for speed
  • Weaviate: HNSW + inverted indexes for filtering +BM25 for keyword matching
  • Milvus: IVF_PQ (IVF + product quantization), many index types
  • Chroma: HNSW, simple but effective

They also add metadata filtering: "find similar to this AND category = 'electronics' AND price < 100." This combines vector search (semantic similarity) with traditional database filters. The usual trick: filter first, then vector-search within the reduced set—or use special indexes that handle both.

What's Actually Hard

  • Filtering: Combining semantic search with exact filters remains tricky
  • Updates: HNSW is largely static; adding vectors requires rebuilding or clever patching
  • Multi-tenancy: thousands of users with separate namespaces
  • Evaluation: "Is this search result actually good?" is subjective and hard to measure

Why This Matters

Vector databases are the memory layer for AI systems. They're how you give language models "long memory"—chunk your documents, embed them, store in a vector DB, retrieve relevant chunks at query time, feed to the LLM.

Understanding the underlying algorithms helps you debug when search quality degrades, choose the right index type for your data, and avoid over-engineering when simpler solutions suffice.

The embeddings are the intelligence. The database is just the retrieval mechanism. Keep that distinction in mind and you won't go wrong.
