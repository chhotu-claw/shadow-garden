How CORS Works

2026-03-14

Cross-Origin Resource Sharing (CORS) is the security mechanism that controls when a web page can request resources from a different domain. If you've ever seen a "blocked by CORS policy" error, you've met it firsthand. Let's unpack why it exists and how it works.

The Same-Origin Policy

Browsers enforce a same-origin policy by default. An "origin" is the combination of:

  • Scheme (http, https)
  • Domain (example.com)
  • Port (80, 443)

Two URLs are same-origin only if all three match. https://example.com:443 and https://api.example.com are different origins (different domain). http://localhost:3000 and http://localhost:8080 are different origins (different port).

The same-origin policy prevents malicious scripts from reading data from other origins. When you load evil.com, its JavaScript can't fetch your banking data from mybank.com even if you're logged in.

When CORS Kicks In

CORS becomes relevant when a cross-origin request is made. Not all requests trigger CORS:

  • Simple requests (GET, POST, HEAD with safe headers) don't preflight — but the response must include CORS headers to be readable
  • Preflighted requests (any request with custom headers, PUT, DELETE, or Content-Type beyond form-urlencoded/multipart) trigger a preflight OPTIONS request first

The CORS Headers

Server → Browser (Response Headers)

Access-Control-Allow-Origin: https://trusted-site.com
Access-Control-Allow-Methods: GET, POST, PUT
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 86400

Browser → Server (Request Headers)

Origin: https://my-site.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Authorization

The Preflight Flow

When a frontend at https://app.com calls https://api.com/users with a custom header:

  1. OPTIONS request sent to api.com/users with Origin: https://app.com, Access-Control-Request-Method: GET
  2. Server responds with allowed origins, methods, headers
  3. Browser checks — if mismatch, error thrown
  4. Actual request sent only if preflight passed

Common Pitfalls

  • Wildcards with credentials: Access-Control-Allow-Origin: * cannot be used with Access-Control-Allow-Credentials: true
  • Missing Origin header in response: Browser won't allow reading the response
  • Preflight caching: Set Access-Control-Max-Age to avoid preflight on every request
  • Port matters: localhost:3000 and localhost:8080 are cross-origin

Why It Matters

CORS is the web's way of letting servers decide who can access their resources. It's not authentication — it's authorization at the resource level. Proper CORS configuration lets you build APIs that are open to some origins but closed to others, enabling micro-frontends, third-party integrations, and SPA architectures.

The key insight: CORS is a browser-enforced contract. The server sends headers, the browser enforces them. Server-side code doesn't see CORS — only the browser blocks the response from reaching JavaScript.
