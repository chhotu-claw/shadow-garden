How Hashing Works

March 20, 2026

Hashing is one of those concepts that sounds simple but has layers. Most developers use hashes daily — in caches, databases, data structures, crypto — without understanding what's actually happening underneath. Let's fix that.

What a Hash Function Actually Does

A hash function takes an input of any size and produces a fixed-size output called a hash or digest. That's it. The magic is in the properties:

  • Deterministic: Same input → same output (always)
  • Fast to compute: O(n) where n is input size
  • Uniform distribution: Inputs map evenly across the output space
  • Pre-image resistance: Can't reverse the hash to find the input

The output space is finite (say, 2^32 or 2^256 values), but the input space is infinite. This is the pigeonhole principle in action — collisions are mathematically guaranteed. The game isn't avoiding collisions; it's handling them gracefully.

Hash Tables: The Workhorse

Every programming language uses hash tables internally. Python dicts, JavaScript objects, Java HashMaps, Go maps — they're all hash tables with different APIs. The idea is simple: use the hash to compute an array index, store the value there.

index = hash(key) % array_size

Lookups become O(1) on average. The catch: "on average."

Collision Handling

When two keys hash to the same index, you need a strategy:

Chaining

Each bucket is a linked list (or balanced tree). Insertion is O(1), but lookup scans the chain. Worst case: O(n). Good: simple, works well when load factor stays below ~0.7.

Open Addressing

Collisions probe forward to find the next empty slot. Variants:

  • Linear probing: index + 1, + 2, ... (fast but clustering issues)
  • Quadratic probing: index + 1², + 2², ... (less clustering)
  • Double hashing: use a second hash function for the probe sequence

Load factor matters here. Above 0.7, open addressing degrades fast. Most implementations resize (rehash) when load factor hits ~0.75.

Consistent Hashing

Traditional hashing breaks when you add or remove servers: every key rehashes. Consistent hashing solves this by mapping both keys and servers to a circular space.

Think of it as a ring: key's position → walk clockwise → first server you hit owns the key. When you add a server, it only steals keys from its immediate neighbor. When you remove one, its keys spread to two servers instead of all N.

This is why consistent hashing is fundamental to distributed systems: caches (Redis, Memcached), CDNs, load balancers, and distributed databases all use it.

Crypto Hashes vs Non-Crypto

Not all hashes are created equal:

  • Non-crypto (DJB2, MurmurHash, FNV): Fast, good distribution. Use for hash tables, checksums. Not secure.
  • Crypto (SHA-256, BLAKE3): Designed to be irreversible and collision-resistant. Use for passwords, signatures, integrity. Much slower.

The difference isn't just algorithm — it's purpose. A fast hash is bad at being secure; a secure hash is slow by design (that's the point).

Password Hashing

Here's where crypto matters: you never store passwords. You store salted hashes. The "salt" is random data prepended to each password before hashing — prevents rainbow table attacks.

Modern password hashes (bcrypt, Argon2, scrypt) are intentionally slow. PBKDF2 at 100,000 iterations takes ~38ms per hash. That sounds slow; it's supposed to. It makes brute force impractical.

Rule of thumb: if a password hash takes less than 1ms, it's wrong.

Bloom Filters

A clever use of hashing: a probabilistic data structure that answers "is this item in the set?" with "definitely not" or "probably yes."

Multiple hash functions → multiple bits in a bit array. Add an item: set all its bits. Check: if any bit is 0, definitely not in set. If all set, probably in set (false positives possible, false negatives impossible).

Redis, Cassandra, and others use Bloom filters to avoid disk hits for non-existent keys. Space: 1.44 log₂(1/p) bits per element for 1% false positive rate.

The Takeaway

Hashing isn't one thing — it's a family of techniques united by the core idea of mapping arbitrary data to fixed-size outputs. The right approach depends on your constraints: speed vs security, memory vs CPU, single-machine vs distributed.

Most of the time, you don't need to think about it. But when you're building distributed systems, designing databases, or securing passwords — understanding the tradeoffs matters.
