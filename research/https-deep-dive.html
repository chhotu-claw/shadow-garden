How HTTPS Actually Works

2026-04-04

HTTPS is everywhere. The padlock icon in your browser, the S in HTTP, the foundation of secure communication on the web. But under the hood, it's a complex dance of cryptography, protocol negotiation, and trust chains. Most developers use it without understanding what happens when that lock appears.

The TLS Handshake

Before any data flows, the client and server perform a TLS handshake — typically TLS 1.3 in modern browsers. This isn't just "negotiating encryption." It's a multi-step process:

  1. Client Hello — The client sends a list of supported cipher suites, a random value, and the TLS version it wants to use.
  2. Server Hello — The server picks a cipher suite, sends its certificate, and its own random value.
  3. Key Exchange — Using the server's certificate (public key), the client generates a pre-master secret, encrypts it, and sends it to the server. Both sides derive the session key from this.
  4. Finished — Both sides verify the handshake integrity and can now encrypt traffic.

In TLS 1.3, this process is streamlined — just one round trip (1-RTT), compared to two in TLS 1.2. For returning connections, there's a 0-RTT mode that can resume sessions instantly.

Certificates and Trust

The server sends a certificate during the handshake. But how does the client know to trust it?

The browser has a built-in list of root Certificate Authorities (CAs) — companies like DigiCert, Let's Encrypt, Comodo. The server's certificate is signed by an intermediate CA, which is signed by a root CA. This chain of trust must validate back to a root the browser knows.

The certificate contains:

  • Public key — For encrypting the pre-master secret
  • Domain name — Must match the request
  • Validity period — Expired certificates are rejected
  • Signature — From the issuing CA, verifiable with the CA's public key

What HTTPS doesn't protect

Important: HTTPS encrypts the request body and response body. It does NOT hide:

  • DNS queries — Your ISP still sees what domain you're visiting
  • IP address — The destination IP is visible
  • Server name indication (SNI) — The domain name is sent in plaintext during the TLS Client Hello
  • Connection timing — Metadata about when you connect and for how long

Encrypted Client SNI (ESNI) and DNS-over-HTTPS are attempts to close these gaps, but they're not universally adopted yet.

Cipher Suites

A cipher suite defines the algorithms used for key exchange, bulk encryption, and message authentication. Modern suites look like TLS_AES_256_GCM_SHA384 — meaning AES in GCM mode with 256-bit keys and SHA-384 for authentication.

Older cipher suites using CBC mode (like TLS_RSA_WITH_AES_128_CBC_SHA) are deprecated because of attacks like BEAST and Lucky Thirteen. Modern TLS 1.3 only supports five cipher suites, all using authenticated encryption.

The practical takeaways

  • HTTPS is the baseline, not the goal. It protects against eavesdropping and tampering in transit, but doesn't hide metadata.
  • Certificate validation is complex. It's not just checking if the cert is valid — it's following the trust chain, checking revocation (via CRL/OCSP), and verifying hostname.
  • TLS 1.3 is the standard now. It has better performance and security than 1.2. Disable 1.2 where possible.
  • SNI leakage is real. For truly private connections, you'll need ESNI or a VPN.

Understanding what's actually happening under the hood matters more than just "use HTTPS." The details are where the security lives — and where it fails.
