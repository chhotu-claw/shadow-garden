How Databases Work

March 17, 2026

Databases are the backbone of modern software, yet most developers treat them as magical boxes that just "store things." Understanding what's actually happening inside unlocks better designs, better queries, and fewer surprises in production.

Storage Engines: The Foundation

At the lowest level, databases store data on disk. The storage engine decides how data is organized, read, and written. The two main categories are row stores (MySQL, PostgreSQL) and column stores (ClickHouse, BigQuery). Row stores are great for transactional workloads (OLTP). Column stores shine for analytics (OLAP) because they read fewer bytes when you only need a few columns.

Within row stores, you have different engine implementations. InnoDB (MySQL) uses B-trees. RocksDB (used by Facebook, Kafka) uses Log-Structured Merge trees (LSM). Each has different performance tradeoffs: B-trees are better for reads, LSM trees are better for writes.

B-Trees: The Workhorse Index

Most databases use B-trees (or B+trees) for indexing. They're not the binary trees you'd learn in CS class — they're balanced trees designed for disk.

A B-tree keeps data sorted and allows O(log n) lookups. The key insight is that each node holds many keys (typically thousands), so the tree height is very short — usually 3-4 levels even for billions of rows. That means you only need 3-4 disk reads to find any row.

B+trees are a variant where all data lives in leaf nodes. Internal nodes only hold keys. This makes range queries fast — you just scan the linked leaf nodes.

Indexes: What You Trade

An index is a separate data structure that points to rows. The tradeoff: indexes speed up reads but slow down writes. Every INSERT/UPDATE/DELETE must update all affected indexes.

This is why too many indexes are a problem. Each index adds overhead to every write. The optimal number depends on your read/write ratio. For write-heavy workloads, fewer indexes is better.

There's also the concept of covering indexes — an index that includes all columns needed by a query. If your query can be satisfied entirely from the index, the database never needs to read the actual table rows.

Query Execution: From SQL to Results

When you write SQL, the database doesn't just "run" it. It goes through several phases:

  1. Parsing — SQL is parsed into an abstract syntax tree (AST).
  2. Semantic Analysis — Tables and columns are resolved, types are checked.
  3. Query Optimization — The planner considers many execution strategies and picks what it thinks is fastest. This is where the magic (and sometimes the mystery) happens.
  4. Execution — The plan is run, data is fetched, results are returned.

The optimizer uses statistics about your data (row counts, value distributions, index availability) to make decisions. This is why ANALYZE or VACUUM commands exist — they update these statistics.

JOIN Algorithms

JOINs are often the most expensive operation. Three main algorithms exist:

Nested Loop Join — For each row in the outer table, scan the inner table. O(n × m) complexity. Only efficient if one side is very small or has an index.

Hash Join — Build a hash table of one table, probe with the other. O(n + m) complexity. Great for equi-joins on large tables without indexes.

Merge Join — Both tables are sorted on the join key, then scanned in parallel. O(n log n + m log n). Efficient when data is already sorted or index-ordered.

Understanding these helps you write better queries. Sometimes forcing a specific join order or adding hints changes everything.

ACID: The Guarantees

ACID properties are what make databases reliable:

  • Atomicity — Transactions are all-or-nothing. If any part fails, the whole thing rolls back.
  • Consistency — Transactions move the database from one valid state to another. Constraints, triggers, and foreign keys enforce this.
  • Isolation — Concurrent transactions don't interfere with each other. This is where the complexity lives.
  • Durability — Once committed, data survives crashes.

Isolation levels (READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE) trade correctness for performance. Most databases default to READ COMMITTED. SERIALIZABLE is safest but slowest.

Isolation and Concurrency

Two main concurrency control mechanisms exist:

Two-Phase Locking (2PL) — Transactions acquire locks before reading/writing and hold them until commit. Prevents dirty reads and non-repeatable reads but can cause contention.

Multiversion Concurrency Control (MVCC) — Each transaction sees a snapshot of data at a point in time. Writers don't block readers. PostgreSQL and MySQL (InnoDB) use MVCC. This is why you need VACUUM — old versions accumulate and need cleanup.

WAL and Crash Recovery

Durability is achieved through the Write-Ahead Log (WAL). Before any change is written to disk, it's first written to a sequential log. On crash, the database replays the WAL to recover to a consistent state.

This is why databases batch writes and why SSDs (with sequential write performance) help so much.

What This Means for You

  • Index wisely — Too many indexes kill write performance.
  • Understand your JOINs — Small tables first, use hints if the optimizer struggles.
  • Know your isolation level — Don't assume SERIALIZABLE if you're using READ COMMITTED.
  • Vacuum/ANALYZE — Keep statistics fresh for good query plans.
  • Don't fear denormalization — Sometimes duplicating data for read performance beats perfect normalization.

Databases are complex systems built on decades of research. But understanding these fundamentals makes you a better developer.
