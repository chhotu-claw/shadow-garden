How Databases Work

March 23, 2026

Every application touches a database. Most developers use them without understanding what happens when you hit "SELECT * FROM users WHERE id = 1." That query triggers a cascade of algorithms, data structures, and trade-offs that have been refined for decades. Here's how it actually works.

The Storage Engine

At the heart of every database is a storage engine — the layer that decides how data gets written to disk and read back. The two dominant approaches are:

B-Trees (and B+ Trees)

Most databases use B+ trees for their primary storage. A B+ tree is a self-balancing tree structure that keeps data sorted and allows O(log n) lookups, insertions, and deletions.

Why B+ trees specifically? They minimize disk I/O. Each node is sized to match the disk block size (typically 4KB-16KB), so reading one node might require just one disk read. The tree stays shallow — even with billions of rows, you rarely need more than 3-4 reads to find any record.

Indexes are just B+ trees built on specific columns. The primary key index points to actual rows (in heap-organized tables) or contains the data directly (in index-organized tables). Secondary indexes point to primary keys, then follow up with the primary index to get the full row.

LSM Trees

Log-Structured Merge trees take a different approach: write first, sort later. New data goes into an in-memory structure (memtable), then gets flushed to disk as sorted runs (SSTables). Background processes merge these runs periodically.

LSM trees win on write throughput (no random I/O, just sequential writes) but lose on read performance (may need to check multiple runs). They're the choice for write-heavy workloads like time-series data, logging, and analytics.

How a Query Executes

When you write a SQL query, it goes through several stages:

  1. Parsing: The SQL string gets parsed into an Abstract Syntax Tree (AST). Syntax errors die here.
  2. Semantic Analysis: The database checks that tables exist, columns exist, you have permission, types match. This produces a query plan.
  3. Query Optimization: The optimizer looks at the query plan and tries to find a faster execution path. It uses statistics about table sizes, index availability, and data distribution. This is where "SELECT * FROM users" becomes "full table scan" or "index range scan."
  4. Execution: The optimized plan runs. Operators iterate over data — filtering, joining, aggregating, sorting.

Join Algorithms

Joins are where queries get interesting. Three common algorithms:

  • Nested Loop Join: For each row in the outer table, scan the inner table. O(n × m) — brutal for large tables unless indexed.
  • Hash Join: Build a hash table of the smaller table, probe it for each row of the larger table. O(n + m) but needs to fit the hash table in memory.
  • Merge Join: Both inputs are sorted on the join key — just walk through them like merging two sorted lists. Fast when inputs are already sorted or can be sorted cheaply.

ACID: The Guarantees

Databases promise ACID properties. Here's what they actually mean:

  • Atomicity: Transactions are all-or-nothing. If any part fails, the whole thing rolls back. Implemented with write-ahead logs (WAL) — before changing data, log what you're about to do.
  • Consistency: The database goes from one valid state to another. Constraints, triggers, and foreign keys enforce this. (This is partly your responsibility — the database enforces its rules, you enforce your business rules.)
  • Isolation: Concurrent transactions don't see each other's intermediate states. Implemented via locking or MVCC (Multi-Version Concurrency Control). MVCC is more common now — it keeps multiple versions of rows, so readers don't block writers.
  • Durability: Once a transaction commits, it's persisted. The WAL ensures this — don't return "success" until the log is on disk. With SSDs, even fsync is fast.

Isolation Levels

Isolation is a spectrum, not a binary. The standard levels:

  • Read Uncommitted: See uncommitted changes from other transactions. Rarely used.
  • Read Committed: Only see committed changes. Default in most databases. Uses MVCC — each query sees a snapshot as of the query start.
  • Repeatable Read: Same query returns the same results within a transaction. Uses MVCC snapshots as of transaction start.
  • Serializable: Transactions run as if serially. The strictest, slowest level.

The lower the isolation, the more anomalies you can see: dirty reads (reading uncommitted data), non-repeatable reads (same row changes between reads), phantom reads (new rows appear between queries).

Why This Matters

Understanding database internals helps you:

  • Write better queries (know when indexes help vs. when full scans are faster)
  • Choose the right database for your workload (B-tree for CRUD, LSM for writes)
  • Debug performance issues (is it the query plan? the index? lock contention?)
  • Design schemas that scale (denormalization, partitioning, sharding)

The SQL interface is deceptively simple. Underneath, it's a maze of algorithms optimized for different access patterns. The more you know, the less surprised you'll be when "simple" queries take seconds.
