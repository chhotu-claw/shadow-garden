How Databases Work

March 16, 2026

Most developers use databases every day. Few understand how they actually work. Here's what happens when you run a simple SELECT * FROM users WHERE id = 1.

The Storage Problem

Imagine you have a table with millions of rows. Finding one specific row by scanning all of them would be absurdly slow — O(n) in Big-O notation. You need O(log n) or better. This is where B-trees come in.

B-Trees: The Workhorse

Databases don't store data in random order. They use B-trees (balanced trees), not binary trees. A B-tree can have thousands of keys per node, making the tree height very shallow — typically 3-4 levels for billions of rows.

Why not binary trees? Because binary trees need to be balanced (via rotations), and each node is one key. With magnetic disks, you want to minimize disk reads. B-trees are designed for this — each node fills a disk page (typically 4KB-16KB), so one read gets thousands of keys.

Indexes: The Lookup Tables

An index is just a B-tree where each key points to a location on disk. When you run WHERE email = 'foo@bar.com', the database:

  1. Traverse the B-tree root → intermediate nodes → leaf
  2. Find the leaf containing the email key
  3. Follow the pointer to the actual data row

This is O(log n) instead of O(n). For 1 billion rows, that's ~30 comparisons instead of 1 billion.

Primary Keys vs Secondary Indexes

Your primary key is special — it's the clustered index. Data is physically stored in B-tree order by primary key. Secondary indexes (like email) store (email, primary_key) in the tree, then jump to the primary key index to get the row.

That second jump is why covering indexes matter — if your query only needs columns in the index, the database never touches the actual table.

Query Execution

When you run a complex query, the database:

  1. Parse — Turn SQL into an abstract syntax tree
  2. Analyze — Check tables exist, columns exist, permissions
  3. Optimize — Try different execution plans, estimate costs
  4. Execute — Run the chosen plan

The optimizer is the magic. It decides whether to use an index, whether to do a full table scan, whether to nested-loop join or hash-join. Bad plans are why your queries are slow.

JOINs: The Expensive Operation

Joins are where databases earn their complexity. Three main algorithms:

  • Nested Loop — For each row in table A, scan table B. O(n×m) without indexes, O(n log m) with an index on the inner table. Good for small tables or when one side is tiny.
  • Hash Join — Build a hash table of the smaller table, probe with the larger. O(n+m). Great for large tables without useful indexes.
  • Merge Join — Both tables sorted, then merge like merging sorted lists. O(n+m). Best when both are already sorted (e.g., clustered on join key).

Why is your JOIN slow? Probably because neither table has an index on the join column, or the optimizer chose a bad plan.

ACID: The Guarantees

Databases provide four guarantees:

  • Atomicity — Transactions are all-or-nothing
  • Consistency — Valid state → valid state
  • Isolation — Concurrent transactions don't interfere
  • Durability — Committed data survives crashes

Isolation is the tricky one. Different isolation levels (READ COMMITTED, REPEATABLE READ, SERIALIZABLE) trade correctness for performance.

What Actually Matters

  • Index the columns you filter and join on
  • Don't over-index — writes get slower
  • Covering indexes can be 10x faster
  • EXPLAIN your slow queries
  • Most performance problems are missing indexes or bad plans, not "the database is slow"
