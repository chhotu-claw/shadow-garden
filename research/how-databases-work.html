How Databases Work

March 9, 2026

Databases are the backbone of modern software. Every app you use, every website you visit, every transaction you make — somewhere, a database is holding the state of the world. But how does it actually work? How do you store data in a way that's fast to read, reliable in the face of crashes, and handles millions of requests per second?

Storage Engines: The Foundation

At the lowest level, a database is a storage engine deciding how data gets written to disk and read back. The two dominant approaches are:

Row stores (MySQL, PostgreSQL, Oracle) store entire rows together — ideal for transactional workloads where you typically need all columns of a record.

Column stores (ClickHouse, BigQuery, Redshift) store columns together — ideal for analytics where you aggregate values across millions of rows but only need a few columns.

Then there's LSM trees (Log-Structured Merge trees) used by Cassandra, RocksDB — optimized for write-heavy workloads. Instead of updating data in place, they append to a memory structure (memtable) and periodically flush to disk as sorted string tables (SSTables).

B-Trees: The Workhorse Index

Most databases use B-trees (or variants like B+trees) for indexing. A B-tree is a self-balancing tree data structure that keeps data sorted and allows logarithmic-time search, insertion, and deletion.

Why not binary search trees? Because disks are different from memory. A binary tree might be 20 levels deep for a million records — that's 20 random disk seeks. A B-tree with a high branching factor (often 100-1000 children per node) might be only 3-4 levels deep. Fewer seeks, faster reads.

B+trees are a variant where all data lives in leaf nodes, and leaf nodes are linked — perfect for range queries and sequential scans. This is what InnoDB (MySQL) uses.

ACID: The Reliability Guarantee

Databases promise ACID properties — Atomicity, Consistency, Isolation, Durability. Let's break each one:

Atomicity — Transactions are all-or-nothing. If any part fails, the entire transaction rolls back. No partial state.

Consistency — Transactions move the database from one valid state to another. Constraints, triggers, and foreign keys ensure this.

Isolation — Concurrent transactions don't interfere with each other. The database pretends they're running serially, even if they're not.

Durability — Once a transaction commits, its effects survive crashes. This usually means writing to disk (and often syncing).

Transactions and Concurrency Control

Here's the hard problem: many users, one database. How do you prevent race conditions?

Locking is the naive approach — acquire a lock before reading or writing, release after. Works, but kills concurrency.

MVCC (Multi-Version Concurrency Control) is the modern solution. Instead of locking, each transaction sees a snapshot of the database at a point in time. Writes create new versions, reads see the version that was current when they started. PostgreSQL, MySQL (InnoDB), and most modern databases use MVCC.

Each transaction gets a unique ID. When you read, the database finds the most recent version where the creator transaction has committed and the destroyer transaction hasn't started yet.

Write-Ahead Logging (WAL)

How do you survive a crash? Before changing data, the database writes to a WAL (Write-Ahead Log). The log is append-only and flushed to disk before the actual data pages are modified.

On crash recovery, the database replays the log — redoing committed transactions, undoing uncommitted ones. This is why your database can crash mid-transaction and come back consistent.

Query Planning and Optimization

When you write SQL, you say what you want, not how to get it. The query optimizer decides the execution plan.

The optimizer considers:

  • Which indexes exist
  • Estimated row counts (statistics)
  • Join order and join algorithms (nested loop, hash join, merge join)
  • Whether to use indexes or full table scans

Good query performance often comes down to having the right indexes and keeping statistics up to date.

Replication and Distribution

One database server isn't enough for many real-world workloads. Replication copies data to multiple nodes:

Synchronous replication — wait for all replicas to confirm the write. Strong consistency, higher latency.

Asynchronous replication — fire and forget. Lower latency, possible inconsistency if the primary fails before replicating.

Leader-follower — writes go to the leader, reads can go anywhere. Single point of contention for writes.

Multi-leader — writes to any node. Complex conflict resolution.

Sharding splits data across nodes by key range or hash. Each shard is an independent database. The hard part is cross-shard queries.

CAP Theorem and Tradeoffs

The CAP theorem says you can only guarantee two of three: Consistency, Availability, and Partition tolerance. Since network partitions happen, you really choose between CP (consistent but unavailable during partitions) or AP (available but eventually consistent).

This is why distributed databases make different tradeoffs. Cassandra optimizes for availability. Spanner optimizes for consistency with geographically distributed replicas.

What Matters in Practice

  • Index the right columns — the biggest performance win
  • Understand your workload — reads vs writes, point queries vs scans
  • Watch your statistics — stale stats lead to terrible plans
  • Replication isn't backup — a replica has the same corruption
  • Connection pooling — opening DB connections is expensive
A database is just a really well-optimized file with ACID guarantees, indexing, concurrency control, and query optimization layered on top. The fundamentals are surprisingly simple. The engineering is in making it fast.
