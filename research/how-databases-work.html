How Databases Work

March 12, 2026

Every application is a interface for a database. The web page, the mobile app, the API — they're all just ways to get data in and out of something that stores it. Understanding databases isn't optional for a developer. It's the foundation.

The Two Worlds: SQL vs NoSQL

The split isn't about better or worse. It's about tradeoffs.

Relational (SQL) databases organize data into tables with rows and columns. PostgreSQL, MySQL, SQLite. You define a schema upfront, tables have relationships, and you query with SQL. The structure is your scaffold.

NoSQL databases throw out the rigid schema. Document stores (MongoDB), key-value (Redis), wide-column (Cassandra), graph (Neo4j). You store whatever shape of data you need, scale horizontally more easily, but you give up the safety rails of enforced relationships.

The pendulum swings. Twenty years ago, it was all SQL. Then NoSQL was the future. Now Postgres is cool again, and "NewSQL" (Spanner, CockroachDB) tries to give you both. The answer is always: it depends.

ACID: The Guarantee

Databases promise four things. Not all keep the promise.

Atomicity — A transaction is all-or-nothing. If any part fails, the whole thing rolls back. You never get half an order.

Consistency — After a transaction, the database is in a valid state. Rules, constraints, cascades — they all fire. No orphaned records, no violated foreign keys.

Isolation — Concurrent transactions don't see each other's intermediate states. My read doesn't see your write mid-flight. This is where things get complicated.

Durability — Once a transaction commits, it's permanent. Even if the power cuts. This is why databases fsync to disk.

NoSQL systems often trade consistency for availability (the CAP theorem). They'll tell you "eventually consistent" means "give it a second and it'll be right everywhere."

Indexes: The Speed Multiplier

A table scan is the enemy. Indexes are how you avoid it.

B-Tree is the workhorse. Sorted, balanced, O(log n) lookups. Primary keys, foreign keys, ranges — B-trees handle it. Every SQL database uses them by default.

Hash indexes are instant for equality lookups (key = value) but useless for ranges. Redis uses them.

Full-text indexes (Elasticsearch, PostgreSQL tsvector) invert the problem — instead of "which rows match this value," they track "which values appear in which rows." Great for search.

Composite indexes use multiple columns. The order matters. Index (a, b, c) helps queries filtering on a, or a+b, or a+b+c — but not b alone.

The cost: every write slows down, and the index takes space. Over-indexing is as bad as under-indexing.

Query Planning: What Actually Runs

You write SQL. The database has to figure out how to execute it. This is the query planner's job, and it's doing a lot more than you think.

Cost-based optimization — The planner estimates the cost of different access paths (sequential scan vs index scan vs index-only scan) and picks the cheapest. It uses statistics (table size, index selectivity, value distribution) to guess.

Join algorithms — Nested loop, hash join, merge join. Each wins in different scenarios. The planner picks based on table sizes, indexes, and join conditions.

Predicate pushdown — Filtering happens as early as possible. Don't load rows you won't use. Push the WHERE clause down to the storage layer.

EXPLAIN ANALYZE — The most important command in your toolkit. It shows you what the planner actually decided, how long each step took, whether indexes were used, and where the bottleneck is.

Transactions: The Concurrency Control

When two users touch the same data at once, who wins?

Pessimistic locking — Grab a lock before you touch. Nobody else gets in until you're done. Simple, but blocks throughput.

Optimistic locking — Read freely, check if it changed on write, retry if it did. Works great when conflicts are rare. MVCC (Multi-Version Concurrency Control) is the implementation — every transaction sees a snapshot, writes create new versions.

Isolation levels — How much weirdness do you allow?

  • READ UNCOMMITTED — See uncommitted changes from other transactions. Rarely used.
  • READ COMMITTED — See only committed changes. The default in PostgreSQL and Oracle.
  • REPEATABLE READ — Your snapshot is stable for the whole transaction. No phantom reads.
  • SERIALIZABLE — Full isolation. Transactions run as if sequentially. The most expensive.

Storage: What's Under the Hood

The database has to write to disk. How it does that shapes everything.

Write-ahead logging (WAL) — Before the data file changes, write what you're about to do to a log. If you crash, replay the log. This is durability.

Buffer pool — Don't read from disk every time. Keep frequently-used pages in memory. The buffer pool is your cache. Sizing it correctly is one of the biggest performance wins.

MVCC versions — Each row can have multiple versions. Readers don't block writers, writers don't block readers. The vacuum cleaner (in PostgreSQL) cleans up old versions eventually.

Columnar storage — Instead of row-oriented (Row 1 has columns A, B, C), columnar packs Column A together. Queries that aggregate (SUM of all of column A) are dramatically faster. Great for analytics. ClickHouse, Redshift, BigQuery.

Sharding and Replication

One database server isn't enough. How do you split the load?

Replication — Copies of the same data on multiple machines. Async (eventually consistent, writes are fast but reads can be stale) or sync (strong consistency, but slower). Leader-follower (write to leader, read from followers) is the common pattern.

Sharding — Split the data itself. User IDs 1-1M go to server A, 1M-2M go to server B. Every query needs to know which shard to hit. The hard part is resharding — when you need to split a shard, everything has to move.

Consistent hashing — A better sharding strategy. Adding or removing nodes moves only a small fraction of keys. Used by DynamoDB, Cassandra, et al.

The Real Answer

Choose based on your access patterns, not the marketing.

Need strict relationships, complex queries, and transactions? SQL. Need to scale horizontally, store semi-structured data, or optimize for write throughput? NoSQL. Need both? Polyglot persistence — different data, different stores.

The database is the heart of your system. Treat it that way.
