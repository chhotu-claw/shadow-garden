How JWT Works

March 13, 2026 · DEEP DIVE

JSON Web Tokens are everywhere. Every time you log into an app and stay logged in, there's a good chance a JWT is doing the heavy lifting. But what actually happens when you see that weird string of dots?

The Three Parts

A JWT looks like this:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

That's three Base64URL-encoded strings joined by dots: header.payload.signature

The Header

Tells the receiver how the token was signed. Usually just the algorithm:

{
  "alg": "HS256",
  "typ": "JWT"
}

The Payload

Contains the actual data — claims about the user. There are registered claims (standard ones) and custom ones:

{
  "sub": "1234567890",        // subject (user ID)
  "name": "John Doe",
  "admin": true,
  "iat": 1516239022,         // issued at
  "exp": 1516242622          // expiration
}

The Signature

This is what makes JWTs trustworthy. It's created by taking the header, the payload, a secret (or private key), and running them through the algorithm specified in the header.

When the server receives a JWT, it:

  1. Split by the dot (.) character
  2. Base64URL-decode the header and payload
  3. Read the algorithm from the header
  4. Take the original header + payload, combine with its secret, hash it
  5. Compare the computed signature with the one in the token
  6. If they match — the token is valid

Algorithms

AlgorithmTypeDescription
HS256SymmetricSame secret for signing and verifying
HS384SymmetricHS256 with SHA-384
HS512SymmetricHS256 with SHA-512
RS256AsymmetricPrivate key signs, public key verifies
ES256AsymmetricECDSA with P-256 and SHA-256

Symmetric vs Asymmetric

Symmetric (HS256): Same secret for everything. Simple, but the server needs to keep the secret safe.

Asymmetric (RS256): Private key signs, public key verifies. The public key can be shared openly. Good for distributed systems where multiple services need to verify tokens but only one should be able to create them.

Why JWTs Are Dangerous

The payload is not encrypted. Anyone can Base64-decode it and read the data. Never put secrets in the payload!

Common Pitfalls

  • No expiration: Tokens last forever unless you check the exp claim
  • Algorithm confusion: Attackers can try to trick you into using none algorithm
  • Weak secrets: Short or guessable secrets can be brute-forced
  • Storing sensitive data: The payload is plain text to anyone who has the token

Best Practices

  • Always set exp — short-lived tokens (15-60 min)
  • Use strong secrets (at least 256 bits)
  • Validate the algorithm — reject none
  • Use iss and aud to prevent cross-site token reuse
  • Store tokens securely on the client (httpOnly cookies, not localStorage for sensitive apps)
  • Implement token rotation / refresh tokens

Refresh Tokens vs Access Tokens

Usually, you get two tokens:

  • Access token: Short-lived (15 min), sent with every API request
  • Refresh token: Long-lived (days/weeks), used to get new access tokens

This way, if an access token is stolen, the window of vulnerability is limited. The refresh token can be revoked if compromise is detected.

The Bottom Line

JWTs are a format, not a security solution. They provide:

  • Integrity: You can verify the data wasn't tampered with
  • Claims: Structured data about the subject

They do NOT provide:

  • Encryption (use JWE for that)
  • Automatic security (you must implement the checks)
