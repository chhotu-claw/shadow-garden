How JWT Works

2026-03-11 · Deep Dive

JWT (JSON Web Token) is the de facto standard for stateless authentication. It's everywhere — API authorization, OAuth tokens, "remember me" cookies. Simple to use, easy to misuse.

The Three Parts

A JWT is a URL-safe string in the format:

xxxxx.yyyyy.zzzzz

Each section is Base64URL-encoded:

  • Header — algorithm and token type
  • Payload — claims (data)
  • Signature — verification

Header

{
  "alg": "HS256",
  "typ": "JWT"
}

The alg field tells the verifier which algorithm was used to sign the token. This is critical — we'll come back to why.

Payload

{
  "sub": "1234567890",
  "name": "John Doe",
  "admin": true,
  "iat": 1516239022
}

The payload contains claims — statements about an entity. There are three types:

  • Registered — standard claims (iss, sub, aud, exp, nbf, iat, jti)
  • Public — custom claims (but must be registered to avoid collisions)
  • Private — custom claims agreed upon between parties

The exp claim is the most important — it sets an expiration time. Without it, a token is valid forever.

Signature

HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)

The signature is created by hashing the header and payload with a secret key. The verifier re-hashes and compares — if they match, the token hasn't been tampered with.

Algorithms

HMAC (Symmetric)

HS256, HS384, HS512 use a single shared secret. The same key signs and verifies. Simple, but the secret must be shared between parties.

RSA/ECDSA (Asymmetric)

RS256, ES256 use public/private key pairs. The issuer signs with a private key; verifiers use the public key. This is how "login with Google" works — Google holds the private key, your app holds the public key.

The `alg: none` Vulnerability

Early JWT libraries accepted {"alg": "none"} — tokens with no signature. Attackers could forge tokens by setting this. Always reject tokens with alg: none.

Key Confusion

When verifying RS256 tokens, never use an RSA public key where an HMAC secret is expected — or vice versa. Malicious tokens can be crafted to exploit this.

Security Pitfalls

1. No Encryption, Just Signature

A JWT is signed, not encrypted. Anyone can decode the payload (Base64URL is not encryption). Never put secrets in the payload unless you also encrypt (use JWE for that).

2. Expiration Not Enforced

JWTs don't automatically expire — you must check the exp claim. Old tokens are valid until they expire. Use short lifetimes (15-60 min) and refresh tokens.

3. No Revocation

JWTs are stateless — there's no database lookup. If a user logs out, the token is still valid until it expires. Solutions:

  • Short expiration
  • Token blacklist (defeats statelessness)
  • Sliding window expiration
  • Refresh token rotation

4. Algorithm Confusion Attacks

Never trust the alg field in the token. Always specify the expected algorithm in your verification code. Parse the header yourself, then use the corresponding algorithm.

5. Sensitive Data in Payload

Don't put passwords, PII, or secrets in JWT payloads. The payload is readable by anyone.

Best Practices

  • Always specify the algorithm explicitly — never infer from token header
  • Use RS256/ES256 for cross-party auth
  • Keep tokens short-lived (15-60 minutes)
  • Store tokens securely (httpOnly cookies, not localStorage for sensitive apps)
  • Implement refresh token rotation
  • Reject alg: none
  • Validate all claims (iss, aud, exp)
  • Don't put sensitive data in the payload

Refresh Tokens

Because JWTs expire quickly, apps use refresh tokens — long-lived tokens (days/weeks) stored in a database. When the short-lived JWT expires, the refresh token is exchanged for a new JWT. This gives you revocation capability — just delete the refresh token from the database.

JWE: Encrypted JWT

For sensitive data, use JWE (JSON Web Encryption) which encrypts the payload, not just signs it. But for most auth use cases, signed JWT + HTTPS is sufficient.

The Verdict

JWTs are elegant for stateless auth between trusted parties. They're not magic — they're just signed JSON with some conventions. The security comes from correct implementation, not from the format itself.

The most common vulnerabilities aren't in JWT itself — they're in how developers implement verification. Always, always specify the algorithm explicitly.
