How HTTPS Works

2026-03-21

HTTPS isn't just "HTTP with encryption." It's a carefully designed protocol that solves multiple problems: confidentiality (can't read the data), integrity (can't modify the data), and authentication (knowing who you're talking to). Here's how it actually works.

The Problem HTTP Doesn't Solve

Plain HTTP sends everything in plaintext. Anyone between you and the server — WiFi hotspots, ISPs, government agencies, hackers — can see:

  • What pages you're visiting
  • What you're submitting in forms
  • Your session cookies
  • Everything.

HTTP has no encryption, no verification, and no protection against tampering. HTTPS fixes all three.

TLS: The Layer That Changes Everything

HTTPS runs HTTP over TLS (Transport Layer Security). TLS is the actual protocol that provides security; HTTP is just the application protocol riding on top.

TLS handles:

  • Encryption (nobody can read your data)
  • Authentication (you're talking to the real server)
  • Integrity (data wasn't modified in transit)

The TLS Handshake: How It All Starts

Before any HTTP data moves, client and server perform a TLS handshake. This is where the magic happens:

  1. Client Hello: Browser tells server "I support these cipher suites, here's a random number"
  2. Server Hello: Server picks a cipher suite, sends its certificate, and its own random number
  3. Certificate Verification: Browser verifies the server's certificate against trusted Certificate Authorities (CAs)
  4. Key Exchange: Both sides generate session keys using the random numbers and the server's public key
  5. Finished: Both sides confirm handshake is complete, switch to encrypted communication

This whole dance happens in ~2-3 round trips and adds ~150ms latency. Not free, but worth it.

Certificates: The Trust Chain

How does your browser know github.com is really github.com? Certificates.

What a Certificate Contains

Certificate:
  Subject: github.com
  Issuer: DigiCert Inc
  Public Key: 04:8f:3a:...
  Signature: a7f3b2c1...
  Validity: 2026-01-01 to 2027-01-01

The certificate is signed by a trusted Certificate Authority (CA) — like DigiCert, Let's Encrypt, or Comodo. Your browser ships with a list of ~150 trusted CAs.

The Trust Chain

When you visit github.com:

  1. Server sends its certificate
  2. Browser checks: is this certificate signed by a trusted CA?
  3. CA's certificate is signed by... another CA (intermediate)
  4. Intermediate CA signed by... root CA
  5. Root CA is hardcoded in your browser/OS
  6. If the chain is valid and not expired → trusted

What Stops Fake Certificates?

  • CAs are audited: Only legitimate organizations can get CA certificates
  • Revocation checking: Browsers check if certificates were revoked (via CRL or OCSP)
  • Certificate Transparency: Public logs of all issued certificates — auditors can spot suspicious issuing

It wasn't perfect (remember DigiCert and Symantec scandals), but it generally works.

Cipher Suites: What Algorithms Get Used

A cipher suite defines the complete set of algorithms for a secure connection:

TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384

This means:

  • ECDHE: Key exchange (Elliptic Curve Diffie-Hellman Ephemeral)
  • RSA: Authentication (proving server identity)
  • AES_256_GCM: Bulk encryption (256-bit AES in Galois/Counter Mode)
  • SHA384: Hashing for message authentication

Modern suites use forward secrecy — each session uses unique keys that are discarded after use. Even if a long-term key is compromised, past sessions stay secure.

What Actually Gets Encrypted

Everything after the handshake:

  • Request URL (except the hostname)
  • Request headers
  • Response body
  • Cookies

What's visible:

  • Server's IP address
  • Server's hostname (from TLS SNI — Server Name Indication)
  • Approximate size of responses
  • Connection timing

HTTP/3: QUIC Changes Things

HTTP/3 uses QUIC (built on UDP) instead of TCP. TLS 1.3 is integrated directly into QUIC, meaning:

  • 1-RTT handshake (vs 2-3 in TCP/TLS)
  • Encryption is mandatory (no cleartext QUIC)
  • No head-of-line blocking at the transport layer

It's faster, especially on lossy networks.

The Hard Parts (Where Things Break)

HTTPS is solid at the protocol level. The problems are human:

  • Mixed content: HTTPS page loading HTTP resources (browser blocks this now)
  • Bad certificates: Expired, wrong domain, self-signed
  • Certificate Authority compromise: If a CA is breached, attacker can issue fake certificates for any site
  • HSTS bypass: HTTP Strict Transport Security can be stripped if user hasn't visited before
  • TLS stripping: Attacker downgrades connection to HTTP before encryption

What You Actually Need to Remember

  1. HTTPS = HTTP + TLS — TLS handles the security
  2. Certificates prove identity — chain of trust from server to root CA
  3. Forward secrecy matters — ensures past connections stay secure even if keys are compromised
  4. The protocol is solid — attacks target implementation, not the math
  5. HTTP/3 integrates TLS differently — faster, mandatory encryption

The web runs on trust. HTTPS is how that trust gets mechanized.
