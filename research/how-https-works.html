How HTTPS Works

March 7, 2026

You type https:// and a lock icon appears in your browser. Your password, your credit card, your secrets — all safe. But what's actually happening? How does your browser talk to a server in a way that nobody else can understand?

The answer involves encryption, certificates, and a carefully choreographed handshake that happens in milliseconds.

HTTP vs HTTPS

HTTP sends data in plain text. If someone intercepts your request, they can read everything — your login credentials, your messages, your data. It's like sending a postcard: anyone who handles it can read it.

HTTPS adds a security layer on top of HTTP. The "S" stands for "Secure." Instead of plain text, your data is encrypted — transformed into gibberish that only the intended recipient can decode. It's like putting your letter in a locked box.

The Two Problems

Encryption alone isn't enough. Imagine you and a stranger try to exchange locked boxes:

  • Problem 1: Encryption — How do you encrypt data so only the right person can read it?
  • Problem 2: Authentication — How do you know the server is who it claims to be?

Solving both is what HTTPS does.

Problem 1: Encryption

There are two types of encryption:

  • Symmetric encryption — Same key locks and unlocks. Fast, but you need to share the key securely.
  • Asymmetric encryption — Two keys: a public key to lock, a private key to unlock. You can share the public key openly; only the private key holder can unlock. Slower, but solves the key-sharing problem.

HTTPS uses both. Asymmetric encryption to safely exchange a symmetric key, then symmetric encryption for the actual data transfer (because it's faster).

Problem 2: Authentication

Anyone can claim to be google.com. How does your browser know it's real?

The answer: certificates. A certificate is a digital document signed by a trusted authority (a Certificate Authority, or CA) that says "this public key belongs to this domain."

Your browser comes pre-installed with a list of trusted CAs. When the server sends its certificate, your browser verifies the signature. If it checks out, great. If not, you get a warning.

The TLS Handshake

Every HTTPS connection starts with a conversation called the TLS (Transport Layer Security) handshake. It happens in milliseconds, before any actual data transfers.

Step 1: Client Hello

Your browser sends a message to the server saying: "Hi, here's what I support (cipher suites, TLS version), here's a random number."

Step 2: Server Hello

The server responds: "Great, here's my choice (TLS version, cipher suite), here's my certificate, here's another random number."

Step 3: Certificate Verification

Your browser checks the certificate. Is it valid? Is it signed by a trusted CA? Has it expired? If something's wrong, you see a warning.

Step 4: Key Exchange

Your browser creates a pre-master secret, encrypts it with the server's public key (from the certificate), and sends it. Only the server can decrypt this with its private key.

Both sides now have the same pre-master secret. They combine it with the random numbers from earlier to generate the session key — a symmetric key they'll use for the rest of the connection.

Step 5: Finished

Both sides send a finished message, encrypted with the session key. If both can decrypt and verify each other's messages, the handshake is complete.

Step 6: Data Transfer

Now all data flows through a symmetrically encrypted tunnel using the session key. Fast, secure, done.

Cipher Suites

A cipher suite is a combination of algorithms that determines how the encryption works:

  • Key exchange — How to securely exchange keys (e.g., ECDHE)
  • Bulk encryption — How to encrypt the data (e.g., AES-256-GCM)
  • Message authentication — How to verify data integrity (e.g., SHA-384)

Modern TLS 1.3 only supports a handful of secure cipher suites. Older TLS versions allowed many, which is why disabling old TLS versions is a common security recommendation.

HTTPS Everywhere

For years, HTTPS was optional — only used for sensitive data like banking or login. Then, in 2018, Chrome started marking HTTP sites as "Not Secure." Today, HTTPS is the default.

Getting a certificate used to cost money. Now it's free, thanks to projects like Let's Encrypt. The web went from mostly HTTP to mostly HTTPS in just a few years.

What HTTPS Doesn't Do

HTTPS protects the transmission of data, not the endpoints:

  • It doesn't hide that you're visiting a site — just the content of what you send/receive
  • It doesn't protect against malware on your device
  • It doesn't stop a compromised server from stealing your data
  • It doesn't verify the quality of a service, only its identity

HTTPS is necessary but not sufficient. It's the locked box — but you still need to trust the person on the other end.

The Lock Icon

When you see that lock icon, here's what actually happened:

  1. Your browser asked the server for its certificate
  2. The server sent a certificate signed by a trusted CA
  3. Your browser verified the signature against its trusted root store
  4. Your browser and server performed a TLS handshake, exchanging keys
  5. Now all data between you and that server is encrypted

All of that, in about 100 milliseconds. Every time you visit an HTTPS site.
