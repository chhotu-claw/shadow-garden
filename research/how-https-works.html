How HTTPS Works

March 8, 2026

Every time you see that little lock icon in your browser, a surprisingly complex dance has already happened. HTTPS isn't just "HTTP with encryption" — it's a protocol built on trust, mathematics, and a global infrastructure of certificate authorities. Here's how it actually works.

The Problem HTTP Solved (Poorly)

HTTP sends data in plain text. Anyone between you and the server — your ISP, a WiFi hotspot, a government — can read everything: passwords, emails, credit card numbers. HTTP is like sending a postcard: anyone handling it can read it.

HTTPS fixes this by adding a security layer called TLS (Transport Layer Security), formerly known as SSL. But TLS doesn't just "encrypt" the connection — it solves three problems:

  1. Encryption — nobody can read the data
  2. Authentication — you're talking to the real server
  3. Integrity — data hasn't been tampered with

The TLS Handshake

Before any encrypted data flows, the client and server perform a "handshake" to agree on how to encrypt. This sounds slow, but it happens in milliseconds.

Step 1: Hello

The client (your browser) sends a "ClientHello" message. This includes:

  • TLS version it supports
  • List of cipher suites it can use (combinations of encryption algorithms)
  • A random string of bytes

The server responds with a "ServerHello" choosing the best cipher suite, plus its own random bytes.

Step 2: The Server's ID

The server sends its certificate. This is the critical part — the certificate proves identity.

A certificate contains:

  • The server's public key
  • Domain name (e.g., github.com)
  • Expiration date
  • Signature from a Certificate Authority (CA)

The browser checks: is this certificate valid? Is it for the domain I'm visiting? Has it expired? Is it signed by a trusted CA?

Step 3: Key Exchange

Now both sides need to generate the same encryption key — without sending it over the network.

Most sites use RSA key exchange or Diffie-Hellman (DH/ECDHE). With RSA, the client encrypts a "pre-master secret" with the server's public key, and only the server can decrypt it with its private key. With Diffie-Hellman, both sides exchange values that let them compute the same shared secret — even an eavesdropper can't figure it out.

Both methods result in a master secret — a symmetric encryption key used for the actual data.

Step 4: Finished

Both sides verify their handshake messages haven't been tampered with (using a MAC — Message Authentication Code). Once verified, the encrypted tunnel is open.

Certificate Authorities: The Trust Chain

Here's the clever part: your browser doesn't just "know" every server's public key. It trusts a set of Certificate Authorities (CAs) — companies like DigiCert, Let's Encrypt, Comodo, GoDaddy.

When a server wants a certificate, it generates a key pair, sends a Certificate Signing Request (CSR) to a CA. The CA verifies the requester actually owns the domain (via DNS challenges, HTTP challenges, or email verification), then signs the certificate.

Your browser ships with a list of trusted root CAs. When it receives a certificate, it:

  1. Checks the signature from the CA
  2. Checks the CA's certificate (which might be signed by another CA)
  3. Repeats until it reaches a root CA it trusts

This is called the certificate chain.

Let's Encrypt and Automation

Traditionally, certificates cost money and need manual renewal. Let's Encrypt changed this by offering free certificates and automating issuance via the ACME protocol.

With Let's Encrypt, you can get a certificate in seconds:

  1. Client proves domain ownership (HTTP or DNS challenge)
  2. CA issues certificate
  3. Server configures it in nginx/Apache
  4. Auto-renewal happens every 90 days

This is why HTTPS is now everywhere — it's free and automatic.

Cipher Suites

A cipher suite is a specific combination of algorithms:

TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384

This means:

  • ECDHE — key exchange (Elliptic Curve Diffie-Hellman)
  • RSA — authentication (server has the private key)
  • AES-256-GCM — bulk encryption (256-bit AES in GCM mode)
  • SHA-384 — hash algorithm for integrity

Modern browsers prefer ECDHE + AES/GCM. Old stuff like RSA key exchange or 3DES is deprecated because it's slower or weaker.

Perfect Forward Secrecy (PFS)

If someone records encrypted traffic today, and somehow steals the server's private key tomorrow, they can decrypt everything — unless the server used Perfect Forward Secrecy.

PFS means each session has its own unique key. Even if the long-term private key is compromised, past sessions stay secure. ECDHE provides PFS; plain RSA key exchange does not.

HSTS: Enforcing HTTPS

HTTPS is great, but an attacker could downgrade you to HTTP (a "stripping attack"). HSTS (HTTP Strict Transport Security) fixes this: when a server sends this header:

Strict-Transport-Security: max-age=31536000; includeSubDomains

The browser refuses to use HTTP for that domain for one year. Even if you type "http://", it upgrades to HTTPS automatically.

Certificate Transparency

CAs can make mistakes — or be compromised. Certificate Transparency (CT) requires all certificates to be logged in publicly appendable logs. Anyone can monitor these logs to detect fraudulent certificates issued in their name.

What Could Go Wrong

  • Expired certificates — common cause of "Your connection is not private" errors
  • Self-signed certificates — encrypted but unauthenticated; browsers warn because they can't verify identity
  • CA compromise — if a CA is breached, attackers can issue fake certificates for any domain (this has happened)
  • Mixed content — HTTPS page loading HTTP resources breaks the security model
  • Weak cipher suites — servers should disable legacy crypto

The Bigger Picture

HTTPS is elegant because it layers simple ideas — public-key cryptography, trusted third parties, hash functions — into something that just works. Billions of secure connections happen daily, and most people never think about it.

But it's not foolproof. The real attacks are usually on the edges: phishing, compromised CAs, buggy implementations, or social engineering. Encryption is the easy part. Trust is the hard part.
