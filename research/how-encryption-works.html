How Encryption Works

March 14, 2026 · Deep Dive · ~15 min read

Every time you visit a website, send a message, or enter a password, encryption is working behind the scenes to keep your data safe. But how does it actually work? And why should you care?

This is a deep dive into the mathematics and mechanisms that protect your digital life — from ancient cipher wheels to modern quantum-resistant algorithms.

What Is Encryption?

Encryption is the process of converting plaintext (readable data) into ciphertext (unreadable data) using an algorithm and a key. Only someone with the correct key can decrypt the ciphertext back to plaintext.

The core guarantee: Even if an attacker intercepts your encrypted data, they cannot read it without the key.

Symmetric Encryption

Symmetric encryption uses the same key for both encryption and decryption. It's like a physical key that locks and unlocks a safe.

How It Works

  1. Sender and receiver share a secret key (they must exchange it securely first)
  2. Sender uses the key to encrypt the plaintext → ciphertext
  3. Sender sends ciphertext to receiver
  4. Receiver uses the same key to decrypt ciphertext → plaintext

Popular Algorithms

  • AES (Advanced Encryption Standard) — The gold standard. Used by governments, banks, and HTTPS. Fast and secure.
  • ChaCha20 — Modern, fast, designed for mobile devices. Used by WireGuard VPN.
  • DES/3DES — Old and broken. Don't use.
Symmetric Encryption Flow
Sender                    Receiver
   |                          |
   |---[Key]----------------->| (secure key exchange)
   |                          |
   |---[Encrypted Data]------>|
   |    plaintext → cipher    |
   |                          |
   |<--[Decrypted Data]-------|
      cipher → plaintext

The Key Distribution Problem

Here's the fundamental problem with symmetric encryption: how do you share the key securely?

If you can securely share the key, you don't need encryption at all — just send the message! This is called the key distribution problem.

The classic chicken-and-egg: To encrypt a message, you need a shared key. But to share the key, you need encryption. Circular.

Asymmetric (Public Key) Encryption

Asymmetric encryption solves the key distribution problem using two related keys:

  • Public key — Can be shared openly. Used to encrypt data.
  • Private key — Kept secret. Used to decrypt data.

Think of it like a mailbox: anyone can drop letters through the front slot (encrypt with public key), but only the owner with the key can open it (decrypt with private key).

How It Works

  1. Receiver generates a key pair (public + private)
  2. Receiver shares the public key with everyone
  3. Sender encrypts data with the public key
  4. Only the receiver can decrypt it with their private key

Popular Algorithms

  • RSA — The classic. Based on factoring large numbers. Slow but reliable.
  • ECC (Elliptic Curve Cryptography) — Modern, faster, smaller keys. Used in Bitcoin, modern TLS.
  • Diffie-Hellman — Key exchange protocol, not encryption per se.
Asymmetric Encryption Flow
Sender                                        Receiver
   |                                              |
   |<---[Public Key]-------------| (shared openly) |
   |                                              |
   |---[Data encrypted]------->| (using public key)|
   |                                              |
   |                      (only private key can decrypt)
   |<--[Decrypted Data]--------|

Hybrid Encryption

Asymmetric encryption is slow. Symmetric is fast. The solution? Use both.

  1. Generate a random symmetric key (session key)
  2. Encrypt the actual data with the symmetric key (fast)
  3. Encrypt the symmetric key with the recipient's public key (secure exchange)
  4. Send both: encrypted data + encrypted session key
  5. Recipient decrypts session key with private key, then decrypts data

This is exactly how TLS/HTTPS works.

Hashing

Hashing is one-way encryption. You can't reverse it. It's used for:

  • Password storage (never store passwords, only hashes!)
  • Data integrity verification
  • Digital signatures

Properties of Good Hash Functions

  • Deterministic — Same input always produces same output
  • Fast to compute
  • One-way — Can't derive input from output
  • Collision-resistant — Hard to find two inputs with same output
  • Avalanche effect — Small input change → completely different output

Popular Algorithms

Pro tip: Never hash passwords with SHA-256. Use bcrypt or Argon2 — they're intentionally slow to prevent brute-force attacks.

Digital Signatures

Digital signatures prove who sent a message and that it wasn't tampered with.

It's the opposite of encryption:

  • Encryption — Hide content from eavesdroppers
  • Signing — Prove you wrote it and it wasn't changed

How It Works

  1. Sender hashes the message
  2. Sender encrypts the hash with their private key
  3. Sender sends: original message + encrypted hash
  4. Receiver decrypts hash with sender's public key
  5. Receiver hashes the message and compares
  6. Match = authentic. Mismatch = tampered.

TLS/SSL: Putting It All Together

TLS (Transport Layer Security) is what makes HTTPS work. It combines:

  • Asymmetric encryption (key exchange)
  • Symmetric encryption (data transfer)
  • Hashing (integrity)
  • Digital signatures (authentication)

The TLS Handshake (Simplified)

  1. Client Hello — Client sends supported cipher suites
  2. Server Hello — Server picks cipher, sends certificate
  3. Certificate Verification — Client verifies server's certificate
  4. Key Exchange — Client generates session key, encrypts with server's public key
  5. Encrypted Session — Both use session key for symmetric encryption
TLS Handshake Overview
Client                                  Server
  |                                       |
  |-------- Client Hello ---------------->|
  |<------- Server Hello + Cert ----------|
  |                                       |
  |---[Verify Cert]---------------------->| (check CA)
  |                                       |
  |---[Generate Session Key]-------------->|
  |    (encrypted with server's pub key)  |
  |                                       |
  |========= Encrypted Data ==============>|
  (symmetric encryption with session key)

Common Vulnerabilities

Encryption is only as strong as its weakest link:

1. Weak Keys

Using password123 as your encryption key is like locking your door and leaving the key under the mat.

2. Implementation Bugs

Good algorithms can be broken by bad implementations. The OpenSSL Heartbleed bug exposed millions of servers.

3. Side-Channel Attacks

Attackers analyze timing, power consumption, or electromagnetic emissions — not the math.

4. Social Engineering

The best encryption can't help if you give your private key to a scammer.

5. Quantum Computing

未来的量子计算机可能会破解 RSA 和 ECC。Post-quantum algorithms 正在开发中。

Best Practices

  1. Use TLS 1.3 — Latest version. Faster and more secure.
  2. Use strong ciphers — AES-256, ChaCha20
  3. Use proper key management — Never hardcode keys
  4. Rotate keys — Change them periodically
  5. Use certificate pinning — Prevent MITM attacks

Summary

  • Symmetric = same key for lock/unlock. Fast, but key distribution is hard.
  • Asymmetric = public key locks, private key opens. Solves key distribution.
  • Hybrid = use both. This is how TLS works.
  • Hashing = one-way. For passwords and integrity.
  • Digital signatures = prove authenticity and integrity.

Encryption is the foundation of digital security. Understanding it isn't just for cryptographers — it's for anyone who cares about their privacy.
