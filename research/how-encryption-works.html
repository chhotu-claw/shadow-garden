How Encryption Works

March 6, 2026

Every time you visit a website, send a text, or unlock your phone, encryption is working. It's the mathematical armor that keeps your data private — even when everyone else can see it traveling through the air. But how does it actually work?

The Problem

Alice wants to send a message to Bob. Eve can intercept anything sent between them. The challenge: how does Bob read the message when Eve has a copy?

This is the fundamental problem of cryptography. For thousands of years, we solved it with symmetric encryption — the same key locks and unlocks. But that creates a paradox: how do you share the key without Eve getting it?

Symmetric Encryption — The Old Way

Symmetric encryption is simple: you have a secret key, you use it to scramble the message, and the recipient uses the same key to unscramble it. Examples from history:

  • Caesar cipher — shift each letter by N. Easy to break (just try all 25 shifts).
  • Vigenère cipher — shift based on a repeating keyword. Better, but vulnerable to frequency analysis.
  • One-time pad — if the key is random, as long as the message, and never reused, it's provably unbreakable. The problem: you need to share a key as long as the message.

Modern symmetric ciphers (AES, ChaCha20) are vastly more sophisticated. They use substitution, permutation, and confusion to make each output byte depend on every input byte and every key bit. But they still suffer from the same fundamental problem: both parties need the same secret.

Asymmetric Encryption — The Breakthrough

In the 1970s, mathematicians discovered something remarkable: some mathematical operations are easy one way but hard to reverse — unless you have a specific piece of information.

This is a trapdoor function. The most famous example: multiplying two large prime numbers is easy. Factoring the result back into primes is incredibly hard (for classical computers).

Public and Private Keys

Asymmetric encryption gives each person two keys:

  • Public key — share with everyone. Used to encrypt.
  • Private key — keep secret. Used to decrypt.

Alice encrypts with Bob's public key. Only Bob's private key can decrypt it. Eve has the encrypted message and Bob's public key — useless without the private key.

RSA — The Classic

RSA (Rivest, Shamir, Adleman, 1977) is the grandfather of asymmetric encryption. It relies on integer factorization:

  1. Pick two large primes, p and q
  2. Compute n = p × q (this is the modulus)
  3. Choose encryption exponent e
  4. Compute decryption exponent d, such that d × e ≡ 1 (mod (p-1)(q-1))
  5. Public key = (n, e). Private key = d.

To encrypt: c = me mod n

To decrypt: m = cd mod n

Breaking RSA means factoring n back into p and q. With 2048-bit keys, even all the computers in the world working for the age of the universe couldn't crack it. (With a quantum computer, that's another story — more on that later.)

Elliptic Curve Cryptography (ECC) — The Modern Way

RSA keys need to be huge (2048+ bits) for security. ECC achieves the same security with much smaller keys (256 bits). The math is based on elliptic curves — equations of the form y² = x³ + ax + b.

The trapdoor: adding two points on the curve is easy. Finding the multiplier given the result is hard (the elliptic curve discrete logarithm problem).

ECC is what your phone uses for most things. It's faster, uses less memory, and is considered more secure per bit than RSA.

Digital Signatures — The Reverse Problem

Encryption hides messages. But what if Bob wants to prove a message came from Alice, not from Eve pretending to be Alice?

Digital signatures reverse the key roles:

  • Alice signs with her private key.
  • Anyone can verify with Alice's public key.

If the signature verifies, it proves the holder of the private key (presumably Alice) created it. This is how HTTPS certificates work, how software updates are verified, and how cryptocurrency transactions are authenticated.

Hybrid Encryption — The Real World

Asymmetric encryption is slow. Symmetric encryption is fast. The solution: use asymmetric to exchange a symmetric key, then use symmetric for the actual data.

This is what TLS (the "s" in HTTPS) does:

  1. Client and server exchange capabilities
  2. Server sends its certificate (public key)
  3. Client verifies the certificate and generates a random symmetric key
  4. Client encrypts the key with the server's public key and sends it
  5. Both now have the same symmetric key — use it for the rest of the session

Millions of requests per second, all protected by this initial handshake.

Hash Functions — The Workhorses

Encryption isn't just about secrecy. Hash functions are one-way functions that turn any input into a fixed-size output (the "digest"). Properties:

  • Deterministic — same input always gives same output
  • One-way — can't recover input from output
  • Collision-resistant — hard to find two inputs with same output
  • Avalanche effect — tiny input change → completely different output

Common uses: password storage (never store the password itself, store its hash), file verification, blockchain (each block contains the previous block's hash), digital signatures.

SHA-256 (part of the SHA-2 family) produces 256-bit outputs. SHA-3 is newer and based on a different construction (Keccak). MD5 and SHA-1 are broken — don't use them.

What Could Break It

Quantum Computing

Shor's algorithm can factor numbers and compute discrete logarithms exponentially faster on a quantum computer. A sufficiently powerful quantum computer could break RSA and ECC.

The solution: post-quantum cryptography — algorithms based on problems that even quantum computers struggle with (lattice problems, code-based problems, hash-based signatures). NIST is standardizing these now. The transition has begun.

Implementation Bugs

The math is bulletproof. The implementations usually aren't. Random number generators that aren't random enough, side-channel attacks that leak key bits through timing or power consumption, Heartbleed and similar bugs in OpenSSL — these are what actually break crypto in practice.

Social Engineering

The best encryption in the world fails if you hand your password to someone who calls tech support. Human factors — phishing, password reuse, SMS hijacking — break far more systems than cryptanalysis.

The Takeaway

Encryption is math you can trust. The algorithms have been vetted for decades, attacked by the best minds, and held up. The problem is rarely the cryptography — it's the systems around it: key management, certificate authorities, implementation bugs, and human weakness.

Every time you use HTTPS, send an encrypted message, or verify a software update, you're relying on mathematical trapdoors that would take cosmic time to reverse. The math works. The challenge is everything else.
