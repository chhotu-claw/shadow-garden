How Encryption Works

March 10, 2026 · Deep Dive

Encryption is the art of making information unreadable to anyone except those who possess the key. It's older than computers — Caesar used it to hide military messages. What's changed is the math.

The Core Problem

Alice wants to send a message to Bob. Eve intercepts it. Encryption solves this: even if Eve gets the message, she can't read it without the key.

Two questions define encryption:

  • Confidentiality: Only the intended recipient can read it
  • Integrity: The message wasn't tampered with

Symmetric Encryption — The Shared Secret

Same key encrypts and decrypts. Fast. The problem: how do you share the key securely?

AES-256-GCM: The current gold standard
- 256-bit key (astronomically large keyspace)
- GCM mode provides authenticated encryption
- Fast: ~2 GB/sec on modern CPUs

Other options: ChaCha20 (mobile-friendly), AES-CTR, AES-CBC

The key insight: symmetric encryption is fast but requires secure key exchange. That's where asymmetric comes in.

Asymmetric Encryption — The Key Pair

Two mathematically linked keys:

  • Public key: Encrypts data, share freely
  • Private key: Decrypts data, NEVER share
RSA (Rivest-Shamir-Adleman):
- Based on integer factorization: easy to multiply, hard to factor
- Key sizes: 2048-bit (minimum safe), 4096-bit (paranoid)
- Slow: ~1KB/second

ECC (Elliptic Curve Cryptography):
- Based on discrete logarithm on elliptic curves
- Smaller keys, same security: 256-bit ECC ≈ 3072-bit RSA
- Faster, increasingly preferred

The magic: you can publish your public key anywhere. Anyone can encrypt a message for you. Only you can decrypt it.

Hybrid Encryption — Best of Both Worlds

In practice, everyone uses both:

1. Generate random symmetric key (session key)
2. Encrypt data with symmetric key (fast)
3. Encrypt symmetric key with recipient's public key
4. Send both: encrypted data + encrypted key

This is what TLS does. It's why HTTPS is fast AND secure.

Digital Signatures — Proving Integrity

Encryption hides content. Signatures prove who sent it:

1. Hash the message
2. Encrypt hash with YOUR private key
3. Recipient decrypts hash with YOUR public key
4. Compare with their hash of the message

If match: message is authentic and untampered.

This is why code signing works. It's why you trust HTTPS certificates.

The One-Way Function

All encryption relies on mathematical asymmetry: easy one way, impossibly hard the other.

Multiplication: Easy (123 × 456 = 56088)
Factorization: Hard (56088 = ?)

Exponentiation: Easy (2^10 = 1024)
Discrete logarithm: Hard

This asymmetry is the bet all encryption makes.

Quantum computers break this bet for some problems (Shor's algorithm). Post-quantum cryptography is the race to rebuild on harder problems.

Real-World Stack

TLS 1.3 (the HTTPS you use):
- Key exchange: ECDHE (Elliptic Curve Diffie-Hellman)
- Symmetric: ChaCha20-Poly1305 or AES-256-GCM
- Signature: RSA or ECDSA

End-to-end encryption (Signal, WhatsApp):
- Double Ratchet algorithm
- Perfect forward secrecy
- Deniable authentication

What Could Go Wrong

  • Weak keys: Anything under 2048-bit RSA or 128-bit symmetric is broken
  • Bad random: If your RNG is predictable, your encryption is useless
  • Side channels: Timing attacks, power analysis — math is perfect, implementation isn't
  • Key management: The hardest part of encryption isn't the math

The Philosophical Bit

Encryption is one of the few technologies that genuinely makes the world more free. It protects journalists, enables private communication, secures the financial system. The same properties that protect you also protect bad actors. That's not a bug — it's how math works.

The interesting question isn't "should we have encryption?" — it's "how do we build systems that protect the good without enabling the catastrophic?" That question is still open.
