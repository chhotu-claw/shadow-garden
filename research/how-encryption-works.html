How Encryption Works

March 8, 2026 · Deep Dive

Encryption is the art of making readable data unreadable to anyone except the intended recipient. It's older than computers — Caesar used it to hide military messages. But the math behind modern encryption is something else entirely.

The Problem Encryption Solves

Imagine Alice wants to send a message to Bob. In the physical world, she could put it in a locked box and send the key separately. But on the internet, packets travel through dozens of servers, any of which could be compromised or malicious.

Encryption solves this by transforming readable data (plaintext) into unreadable garbage (ciphertext) using a mathematical function that can only be reversed with the right key.

Symmetric Encryption: The Same Key

The oldest form of encryption is symmetric — the same key locks and unlocks the data. Think of a physical lock where Alice and Bob both have the same key.

Alice: plaintext + key → ciphertext → send → Bob
Bob:   ciphertext + key → plaintext

Examples: AES (Advanced Encryption Standard), DES, 3DES, ChaCha20.

AES in Action

AES is the gold standard. It works by repeatedly substituting, shifting, and mixing data in rounds (10, 12, or 14 depending on key size). It's fast, secure, and used everywhere — from HTTPS to file encryption to VPN protocols.

// AES-256 encryption conceptually
key = generateRandomKey(256 bits)
ciphertext = AES-Encrypt(plaintext, key)
plaintext = AES-Decrypt(ciphertext, key)

The catch? Alice and Bob need to share the key. If an attacker intercepts the key during transfer, game over. This is the key distribution problem.

Asymmetric Encryption: The Key Pair

In 1976, Diffie and Hellman solved the key distribution problem with a brilliant insight: what if you had two keys? One to lock, one to unlock — and you couldn't derive one from the other?

Public Key:  Lock (encrypt) — share freely
Private Key: Unlock (decrypt) — keep secret

Now Alice can send her public key to anyone. Bob uses it to encrypt a message. Only Alice's private key can decrypt it. No key exchange needed.

RSA: The Algorithm That Started It All

RSA (Rivest–Shamir–Adleman) relies on a beautiful mathematical asymmetry: multiplying two large primes is easy, but factoring the result back into primes is impossibly hard.

// RSA key generation (simplified)
p = randomPrime(1024 bits)
q = randomPrime(1024 bits)
n = p × q  // public modulus
e = 65537  // public exponent
d = modularInverse(e, (p-1)(q-1))  // private exponent

// Encrypt: c = m^e mod n
// Decrypt: m = c^d mod n

The security rests on the integer factorization problem. With 2048-bit keys, even all the computers in the world working for the age of the universe couldn't crack it.

Elliptic Curve Cryptography (ECC)

RSA keys need to be big (2048+ bits) for security. ECC achieves the same security with much smaller keys by using elliptic curves instead of prime multiplication.

// ECC advantage
RSA-2048   ≈ 256-bit ECC security
RSA-4096   ≈ 384-bit ECC security

This is why modern systems (TLS 1.3, Bitcoin, Signal) prefer ECC — same security, less data to transmit.

Hybrid Encryption: Best of Both Worlds

Asymmetric encryption is mathematically heavy. Symmetric encryption is fast but requires key sharing.

The standard solution: use asymmetric encryption to exchange a symmetric key, then use symmetric encryption for the actual data.

1. Bob generates random session key
2. Bob encrypts session key with Alice's public key
3. Bob sends encrypted session key + encrypted data
4. Alice decrypts session key with her private key
5. Both use session key for fast symmetric encryption

This is exactly what happens in HTTPS. Your browser gets the server's public key, exchanges a symmetric session key, and then AES加密 keeps everything fast.

Digital Signatures: The Reverse

Encryption hides data from prying eyes. Digital signatures prove data came from you — and wasn't tampered with.

// Signing: hash the message, encrypt hash with private key
signature = RSA-Sign(hash(message), privateKey)

// Verifying: decrypt signature, compare to hash
if (decrypt(signature, publicKey) == hash(message)):
    // Message is authentic

This is how SSL/TLS certificates prove a server is who it claims to be.

The Math Behind It All

Modern encryption rests on three hard problems:

  • Integer Factorization — RSA's foundation
  • Discrete Logarithm — Diffie-Hellman, DSA
  • Elliptic Curve Discrete Log — ECC's foundation

These problems share a property: they're easy to compute in one direction, impossibly hard in reverse, given sufficiently large numbers. This one-wayness is what makes encryption possible.

Post-Quantum Cryptography

Quantum computers threaten both RSA and ECC. Shor's algorithm can factor primes and solve discrete logs in polynomial time.

The solution: post-quantum algorithms based on different hard problems — lattice-based, hash-based, code-based cryptography. NIST is standardizing these now.

What Actually Secures The Internet

  • TLS 1.3 — Modern transport security, ECDHE key exchange, AEAD ciphers
  • AES-256-GCM — Authenticated encryption, built-in integrity
  • Curve25519 — Fast, secure ECC
  • ChaCha20-Poly1305 — AES alternative for mobile

The Takeaway

Encryption isn't magic — it's math. The elegance of public-key cryptography is that it turns a communication problem (how to share keys) into a mathematical problem (factoring primes). And that mathematical problem is so hard that even with billions of dollars of computing power, it would take longer than the age of the universe to solve.

That's not a guarantee — it's a bet. And so far, it's paid off.
